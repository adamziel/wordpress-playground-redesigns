# Design #052 — Branching tree (org-chart)

## Thesis
Every WordPress Playground session is, in practice, an experiment: spin one up, mutate it, sometimes throw it away. This design makes that explicit by turning Playground's history into a **horizontal, fork-able org-chart**. The viewport is split between a Tree canvas (left/top — a real, pannable diagram of root playgrounds branching into snapshots and forks) and a Stage panel (right/bottom — the iframe + an inspector for whichever node is selected). Saves become commits, blueprints become root seeds, "Apply Settings & Reset" becomes a *new branch from the parent*, never an in-place wipe. The metaphor isn't decorative — it's how you navigate, save, fork, settings-change, and resume.

## What I remixed / merged / removed
The Site Manager's five tabs collapse into a single **Inspector dock** that swaps content (Settings / Files / Blueprint / DB / Logs) based on what's selected — the redundant "Settings popover vs. Settings tab vs. info-notice Save" trio merges into one panel because settings are now node properties, applied by branching. "Saved Playgrounds" stops being a drawer: it *is* the tree's root row, plus a **Spawn rail** on the left edge that holds every way to seed a new root (Vanilla, WordPress PR, Gutenberg PR, From GitHub, Blueprint URL, Import .zip, plus the 43-blueprint gallery in a popover). The top toolbar shrinks to the truly global bits (refresh, address bar, the selected node's name, additional-actions menu). Toasts dock to a corner. Nothing was added.

## Trade-offs
A tree-as-primary-UI is unfamiliar — first-touch users may need a moment to recognize that clicking a node mounts it. I mitigate with a permanently-visible "Stage" panel that always shows *something* live, plus an empty-state hint. The tree's spatial cost is real: on mobile it collapses to a vertical accordion of branches with breadcrumb chips, losing the visual fork-ability that's the whole point on desktop. Forking every settings change can also produce node clutter; a "merge into parent" / prune affordance exists on each node but isn't fully fleshed out here. Finally, this design leans hard into a power-user mental model — it's optimized for the person testing five Gutenberg PR branches in parallel, not for the visitor who just wants to click Twenty Twenty-Five.
