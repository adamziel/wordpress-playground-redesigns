# Design #009 — Storybook-style component library

## Thesis
WordPress Playground is, structurally, a Storybook for WordPress instances. Each saved site is a *component*, each snapshot or blueprint is a *story*, the iframe is the *canvas*, and "settings" are literally Storybook *controls*. This redesign takes that metaphor as a contract. The left rail becomes a nested story tree (Your Playgrounds → snapshots; Blueprint Library → categories → blueprints; Sources for PR/GitHub/URL/zip). The viewport is a Canvas with sibling Docs / Files / Blueprint / Database tabs the way Storybook gives you Canvas / Docs at the top of a story. And the bottom drawer becomes Storybook's add-on panel: Controls (= settings, including PHP/WP/language/network/multisite, with one big destructive Apply), Actions (= request/router/blueprint events with timing), Logs (= PHP error log with level filter — the pain point fixed), and Outline (= a11y / structure of the rendered site).

## What I remixed, merged, removed
- **Merged** the three settings entry points (popover, sidebar Settings tab, Save-notice button) into a single dual surface: a topbar **Controls** button for a quick popover, and a full **Controls** add-on panel at the bottom that *is* the Site Manager Settings tab — same data, one schema, one Apply button with a single explicit destructive warning.
- **Merged** the Saved Playgrounds drawer and the standalone Blueprint Gallery sub-screen into one Launcher modal with a Storybook-shaped left nav (Start new / Library / Your playgrounds + category list + Sources). All six "start a new" sources (Vanilla, WP PR, Gutenberg PR, From GitHub, Blueprint URL, Import .zip) live as cards under one roof.
- **Remixed** the Site Manager tabs as Canvas-level tabs (Canvas, Docs, Files, Blueprint, Database) plus the bottom add-ons (Controls, Actions, Logs, Outline) so the user never context-switches into a separate "manager" — the Storybook canvas is always the home.
- **Removed** the floating bottom-left toast position only in spirit: kept as a Storybook-style status toast so saves are unmissable.
- The "Unsaved" warning is upgraded from a small text button to a pulsing orange chip next to the URL bar so refreshing-and-losing-everything is harder to do accidentally.

## Trade-offs
The Storybook metaphor is dense — first-time WordPress users may not parse a story tree, and "controls", "actions", "outline" are jargon-ish for non-developers. I lean into it because Playground's real audience *is* developers, theme authors, and PR reviewers; the rail's groupings ("Your Playgrounds", "Featured blueprints", "Blueprint Library", "Sources") are still plain English. The bottom add-on panel costs ~280px of vertical real estate on desktop; on mobile it collapses entirely in favor of a bottom tab bar. The Files / Blueprint / Database canvas tabs intentionally duplicate the file-tree component on the left side of their split panes — same component, different roots — which keeps the "everything is a story canvas" feel but means the tree shows up in three places (one rail, three panes). I judged the consistency worth the repetition.
