# 076 — Drag-and-drop blueprint builder

## Thesis
Playground is, at heart, a *blueprint runner*. This redesign elevates the blueprint from a hidden JSON file to the **primary surface of the app**: an n8n / Node-RED style visual canvas where each step (install plugin, set option, create user, run wp-cli) is a draggable node, connected by wires into a flowchart. The WordPress preview iframe lives in a docked panel on the right that updates every time you hit **Run** — so the canvas literally *is* the playground. Authoring, debugging, and launching all become the same gesture: drag, wire, run.

## What I remixed / merged / removed and why
The old Site Manager tabs (Settings, File browser, Blueprint, Database, Logs) collapse into a single bottom **Inspector dock** whose contents change based on what you have selected on the canvas — pick a node, see its inputs; pick the database node, see the DB browser; pick the logs node, see the log stream. The Saved Playgrounds launcher and the 43-blueprint gallery merge into one left **Library rail** with two tabs: *Templates* (the 43 + starter sources like WordPress PR, Gutenberg PR, From GitHub, Blueprint URL, Import .zip) and *Yours* (saved playgrounds). Dropping any of those onto the canvas instantiates them as a starter node. The Settings popover, the Site Manager Settings tab, and the inline "save site" notice all reduce to a single first-class **Environment** node at the top of the flow, plus an always-visible "Unsaved" pill in the top bar with a real Save button. Additional-actions (Export to GitHub PR, Download .zip) become canvas-level run targets in the Run dropdown.

## Trade-offs
A flow-graph metaphor is opinionated: linear users who just want "vanilla WordPress, now" need a single click, so the canvas pre-loads a one-node "Boot WordPress" flow and the **Run** button is a giant primary control — you can ignore the canvas entirely and the app behaves like the current Playground. Power users get visual scripting; casual users get a louder Run button. The cost: the canvas itself is screen real-estate the iframe used to own, so on small viewports the iframe gets pushed to a "Preview" tab and on mobile the canvas degrades to a vertical card list (the flow becomes a step list — still draggable to reorder). Wire-drawing is not implemented as live SVG dragging here; nodes are auto-wired in execution order, which is what 95 % of real blueprints look like in practice.
