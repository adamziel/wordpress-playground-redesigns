# 026 — Wizard / Step-by-step

## Thesis
WordPress Playground is genuinely intimidating for newcomers: there are three settings entry points, 43 blueprints with no preview, a destructive "Apply & Reset" button, and a Site Manager whose tabs each speak a different dialect. This design takes the opposite stance of "power-user dense" — every action is reframed as a friendly, numbered, single-question-at-a-time **Wizard**. A persistent progress bar tells you where you are; a guide rail on the left tells you what's next; a "What is this?" panel on the right anticipates the question you were about to Google. The iframe is still the hero on the right side of the screen, but the chrome around it is now a teacher rather than a control surface.

## What I remixed, merged, and removed
The three duplicate settings entry points (popover, Site Manager Settings tab, info-notice "Save site locally") are merged into a single **Settings Wizard** that is launched from one button and adapts based on whether the playground is saved. The Saved Playgrounds drawer, the blueprint gallery, "Import .zip", "From GitHub", "Blueprint URL", "WordPress PR", and "Gutenberg PR" are merged into one branching **"Start a new playground"** wizard whose first step is just "How do you want to start?" — six big tiles. The Site Manager tabs (File, Blueprint, Database, Logs) become four parallel **"Inspect"** wizards, each with their own progress trail (e.g., File browser = pick file → preview → edit → save). The Additional-Actions menu becomes the final step ("What's next?") of every flow — Download .zip, Export to GitHub PR, Save locally — so users discover those features in context instead of buried in a kebab. Refresh, address bar, and homepage/admin shortcuts stay as the only persistent toolbar — everything else lives inside a wizard.

## Trade-offs
Power users will hate this. Switching PHP versions becomes 3 clicks instead of 2, and you can no longer "just look at" the full blueprint gallery as a wall of 43 tiles (it's paginated/filtered inside step 2 of a wizard). The wizard scaffolding is also visually expensive: progress bar, step rail, helper pane all consume chrome that the iframe used to own. The bet is that **first-time users are the bottleneck** — Playground's whole reason to exist is "try WordPress in 5 seconds" — and the wizard framing converts ambiguous icons into plain English questions ("Which WordPress version?", "Should the site survive a refresh?"). The destructive Apply-and-Reset risk drops dramatically because the wizard's final step is always an explicit confirmation screen with a diff of what will change.
