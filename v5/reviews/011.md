# Review of v5 #011 — Auto-hide HUD

**Parent direction:** "Auto-hide HUD. Video-player style. The iframe is full-bleed. Mouse to any edge reveals chrome for that edge (top = URL, bottom = sites, left = files, right = settings). Idle 2s → hidden."
**Affordance count at idle:** 1 (heartbeat pill at line 700–704). When revealed (state 2): 8 — top strip (refresh, URL, name, menu = 4) + left rail (1) + right rail (1) + bottom (dock + fullscreen pill = 2). At cap exactly when revealed.

**Scores (out of 10):**
- Feature completeness: 10 — 10 states cover idle (1), revealed (2), Launcher pane (3), Site Manager pane Settings (4), Apply&Reset typed confirm (5), Files (6), Blueprint + Database + Logs triptych (7), Booting (8), Launcher "Start new" tab variation (9), absolute fullscreen with chrome dissolved (10). All five SM tabs visibly demonstrated. README's mapping is exhaustive.
- Restraint: 10 — true idle is 1 affordance (heartbeat). Revealed state honors the 8-cap. The 2-second timer is named in copy.
- Iframe primacy: 10 — at idle the iframe is 100% bleed; only one pixel of chrome ghost-line for screenshot legibility. State 10 dissolves *even the heartbeat pill*. The HUD locks open during boot (state 8) and when panes open — intelligent context-awareness.
- One-door: 10 — README's enumeration is exhaustive and clean: Settings → SM Settings tab. Save → URL pill chip. Launcher → left rail. Site Manager → right rail. Export/Download → kebab. WP-admin/Home → SM header. Apply & Reset → inside SM Settings. Best one-door discipline in v5 so far.
- One-secondary: 10 — same `.pane` component docks from left rail (Launcher) or right rail (Site Manager). Same React node in production. Content swaps via tabs inside.
- Direction adherence: 10 — "auto-hide HUD" is the entire IA. 2s idle timer named. State 1 shows iframe with no chrome (just heartbeat). Edge annotations name the four edges. The video-player metaphor carries throughout.
- Visual fidelity: 9 — Inter font, Lucide icons, real WordPress iframe ("Roast & Bean" coffee site with Origin/Notes/Roast strip), heartbeat dot with pulse animation, edge annotations for the reviewer (clearly marked as invisible-in-production), explicit ghost-line for screenshot purposes (line 707).
- Synthesis with v1 lesson: 10 — refines #38 glass (auto-hide *plus* glass = direction earned), #15 fullscreen (state 10 absolute fullscreen), #64 DevTools slim chrome, and inherits #55 iPad pull-tab via the side-rail pane.

## What works (specific)
- **The single .pane component** docked from either left or right is a *real* one-secondary-surface implementation. Same React node, mirrored by edge. This is the v5 §2 rule 4 absolute compliance.
- **State 10 (absolute fullscreen)** is the strongest fullscreen state of any v5 entry seen so far: chrome dissolved *including* the heartbeat pill. F + Esc.
- **The heartbeat pill** (line 700–704) carries three pieces of info — "Saved" (status), "move to any edge for HUD" (instructions), `F` (shortcut hint) — in one tiny upper-right element. Solves the "how does the user know the HUD exists?" problem with minimal pixel cost.
- **The HUD locks open during boots and pane-open states** — context-aware auto-hide. Not a hard 2-second rule but a smart rule.
- **State 7 (Blueprint + Database + Logs triptych)** shows three tabs of the same pane in one composite. Effective use of one frame to cover multiple panel states — best fidelity efficiency in v5 so far.
- README is honest: 4.71/5 average self-score with -1 deductions for the 28px blur (could be 22) and the mobile pane covering iframe (unavoidable). This is critic-grade self-critique.
- The "version chip is passive text" disclaimer in the affordance count (line 22–26) avoids the failure mode of #010 (where version chip looked clickable but wasn't).
- The save chip-as-only-Save: "Save lives only in the URL pill as a colored save-chip — no separate Save button anywhere; clicking the chip is the save gesture." Cleaner than #003's slider+pill duplication and matches the brief's "save state as primary visual cue."

## What's weak or missing (specific)
- **The first-touch problem is the design's biggest risk.** The README acknowledges this but the mitigations (heartbeat pill + one-time onboarding flash) put a lot of weight on a tiny corner element. The instruction text "move to any edge for HUD" appears in the heartbeat pill *only at idle* — by definition, that text fades away after 2 seconds. A returning user months later may not remember the gesture.
- **Slim 36px rails reduce the "100% iframe" claim slightly**: README admits "the iframe at the 'revealed' state is 1368×792 (~83% of the screen) rather than truly 100% — only the idle state is 100%." The README rates iframe primacy 5/5 *because* of the idle state, which is fair, but the revealed-state honesty is noted.
- **Mobile pane covers iframe entirely** (README -1). Mobile is the most likely "first-touch" experience and the auto-hide gesture is hardest to discover on touch (you don't have a "mouse-move" — what's the equivalent? Tap?). The README says "the phone uses 8px edge-gradients in place of 36px rails (touch targets expand on tap)" — *tapping the edge* is the gesture, but a new mobile user wouldn't know to swipe-from-edge unless prompted.
- **The 2s timer is mentioned but not adjustable.** Power users who pause to read text may resent the HUD reappearing; users who rarely move the mouse may resent it hiding. The README defends 2s as "video-player industry baseline" — acceptable, but a per-user preference would be ideal.
- **The bottom dock at 64px is wider than the other strips** — the README says it's "saved-sites dock + fullscreen pill" but those two items don't need 64 px of vertical real-estate. Could be tightened to 48 px for consistency with the top strip (44 px).
- **State 9 ("Start new" tab) confirmed in the file** — good — but it would be strong to also see state with the *blueprint search* showing typed text and filtered results, so the search behavior is rendered, not just implied.

## How to push harder (1–2 pointed suggestions)
1. **Add a *persistent* corner pill at the *visible* edge** — not a glass overlay but a 16 px pulsing dot at the screen's true corner (outside any HUD strip), always visible. This costs effectively 0 pixels (16×16 in a corner) and solves the "returning user forgot the gesture" problem permanently. The current heartbeat pill is glass *inside* the iframe area — it's not always there.
2. **Make the *first* mouse-move tutorial-style.** The first time chrome reveals after page load, label each edge with its name for 1.5 s, then collapse to icon-only. The state 1 annotations are explicitly described as "would be invisible in production" — but a one-shot teaching layer is exactly what an auto-hide HUD needs. Match the pattern Instagram uses for its first stories swipe.

## Lesson for the next designer
If your direction is auto-hide, you cannot rely on a single hint pill — it disappears with the rest of the chrome by definition. Build in a one-shot teaching state (label-all-edges for ~1.5s on first reveal) so returning users re-learn the gestures without needing a help screen.
