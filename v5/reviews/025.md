# Review of v5 #025 — Browser Tabs Across the Top

**Parent direction:** "Browser tabs across the top. Each saved playground = a real browser tab. New-tab = launcher. URL bar acts on the active tab. DevTools-style dock below remains."
**Affordance count at idle:** 8 (README-counted): tab strip · "+" tab · refresh · URL bar · save-state pill · fullscreen · dock toggle · more (kebab). At cap exactly.

**Scores (out of 10):**
- Feature completeness: 10 — all 5 SM tabs as dock content, "+" tab opens the launcher (New Tab page with featured + saved + start sources), full 43-blueprint browser, typed-phrase Apply&Reset, Export PR + Download .zip as icon buttons in dock chrome (contextually appropriate placement), explicit fullscreen state with Esc-to-exit hint band.
- Restraint: 9 — 8 exactly, no hidden idle affordances. Dock collapsed by default to a 32 px status rail.
- Iframe primacy: 10 — dock collapsed = 76 px chrome + 32 px status rail = iframe gets 792 of 900 px (88%). Fullscreen toggle + Esc to exit.
- One-door: 9 — README admits a half-point: "URL bar accepts navigation AND, on the New-Tab page, search — but that's literally how Chrome's omnibox works." Borrowing the omnibox's mixed-purpose is *correct* for the metaphor and not a violation. Settings dock-only. Save pill-only. Saved sites tabs-only. Launcher New Tab only. Clean.
- One-secondary: 10 — the dock is the only secondary surface; five tabs swap its content.
- Direction adherence: 10 — "Browser tabs across the top" is taken to its logical end: tabs + address bar + save-state pill + new-tab page + window controls — the *whole Chrome primitive set inherited*, not just the tab strip. Best metaphor commitment in v5.
- Visual fidelity: 9 — solid chrome surfaces (no glass), Chrome-native interactions, tab favicons + unsaved-dots, dock status rail with toggle, Export/Download as contextual icon buttons in dock chrome.
- Synthesis with v1 lesson: 10 — extends #64 (DevTools) by going *the whole browser*, not just the bottom panel. **Cites four prior v5 reviews explicitly with the lesson applied:**
  - Review 005: "Don't nest a second tab strip inside a sheet" → mobile dock-sheet uses one chip strip with remembered last-used tab.
  - Review 009: "A kebab can hide as much complexity as a visible bar" → kebab holds only meta items; Export/Download live on dock chrome contextually.
  - Review 003: "Don't ship the same verb in three forms" → Save lives in the pill only.
  - Review 006: "Don't import clever ideas from other v1 winners" → no spatial dock, no glass, no Stage-Manager peers.
  This is the **best synthesis-with-prior-reviews of any v5 design** — four reviews cited by number with the specific implementing element identified. Beats #018 (three citations) and #017 (two).

## What works (specific)
- **Tabs-as-playgrounds is conceptually pure** — the Saved Playgrounds dropdown is *gone*. The tab strip *is* the catalog. Active state, favicons, unsaved-dot — every visual primitive Chrome already taught the user, now functional in Playground.
- **Address bar with embedded save-state pill** uses Chrome's familiar "https://" indicator pattern for the saved-state visual cue. Chrome users already know to glance at that spot for site security; here it's repurposed for save state. Genuine primitive reuse.
- **New-Tab page surfaces "Your Playgrounds" tiles for the long tail** — same as Chrome's frequently-visited tiles. Solves the 20+ saved sites problem the README pre-empts.
- **Export/Download as icon buttons in dock chrome**, visible *only* when the dock is open and contextually appropriate. README's lesson-from-review-009: "kebab here holds only meta items (Sign in, Docs, About) — *not* duplicates of dock items."
- **Dock-as-DevTools** is the v1 #64 lineage taken to its logical end — collapsed by default (32 px status rail), expanded on toggle, five tabs swap content.
- **Zero glass, zero ambient tint** — the browser metaphor uses real chrome. README rates 5/5 on the glass critique and earns it.
- The README's "Lessons applied from earlier v5 reviews" section is the cleanest synthesis evidence in v5: four reviews cited by number, each with the specific implementing element.
- **Mobile uses the same primitives, scaled and rotated** — tab strip scrolls, dock becomes bottom sheet with sub-tabs as chip strip (per Review 005's lesson). No "mobile fallback."

## What's weak or missing (specific)
- **Tab strip overflow at 20+ saved sites**: README's mitigation (horizontal scroll + New-Tab page surfaces) is correct but the overflow behavior should be demonstrated in a state, not just described. Currently the design likely shows 4-5 tabs visible — a state with 15+ tabs would prove the scaling works.
- **Familiar = invisible** is the README's articulated risk: "the browser metaphor is so familiar it risks invisibility — designers may expect playgrounds to feel 'more like a tool.'" Defended on brief grounds (familiarity is a feature). Fair, but for a *redesign* there's a slight risk that the user-feedback "this looks like Chrome, not Playground" emerges.
- **Mobile dock-sheet drops the kebab** (README, -1). The kebab folds into the dock-sheet handle gesture. Same critique as #014's mobile drop — gestures replace affordances. Worth documenting the handle gesture explicitly in a state.
- The **"+" tab is the only door to the launcher**. Good for one-door. But if a user is *in* a saved playground and wants to find a blueprint to apply to *this* site, they'd have to open New Tab, find a blueprint, then somehow apply it to their existing site — that's a workflow gap. Either the blueprint application is a per-tab operation (then it lives in the dock's Blueprint tab) or it's a "create a new playground from this blueprint" operation (then "+" tab is correct).
- **The save-state pill is in the URL bar**, mimicking Chrome's https indicator. New users may not realize it's *both* status and the save button (clicking it saves). Hover tooltip helps; needs to be loud about the dual function.
- **No keyboard shortcut hint** for tab navigation (`⌘1..⌘N` for tabs) is implicit in the browser metaphor, but worth showing in the dock chrome or kebab for less Chrome-fluent users.

## How to push harder (1–2 pointed suggestions)
1. **Demonstrate the 15-tab overflow state.** Add a state showing the tab strip scroll-overflow with chevrons at the edges, plus the New-Tab page's "Your Playgrounds" tiles section visible for long-tail discovery. This proves the metaphor scales beyond a screenshot's typical 4-5 visible tabs.
2. **Add a "duplicate to new tab" affordance.** When in a saved playground, allow right-click on the tab → "Duplicate playground" — opens a new tab with a clone. This solves the "I want to test a destructive change without losing my work" use case that v1 power users have. Free affordance because it's a context menu on an existing element (the tab), not new chrome.

## Lesson for the next designer
If your direction reuses an existing OS primitive (browser tabs, iPad multitasking, iOS Control Center), do the *full* reuse — not just the most visible element. Inherit the address bar, the new-tab page, the unsaved-dot pattern, the keyboard shortcuts. Half-reuse looks like a borrowed flavor; full reuse looks like a primitive.
