# Review of v5 #017 — Pull-tab from the right

**Parent direction:** "Pull-tab from the right. Iframe left, 24px tab on the right edge. Pull left → 25% pane slides in. Pull further → 50%. Snaps at 25/50/75. One pane, swappable content."
**Affordance count at idle:** 6 (banner-counted): Refresh · address-bar · name-pill (= save-state + Save) · fullscreen-iframe · overflow `⋯` · pull-tab rail. Below cap. README explicitly notes "overflow menu shown open to verify Export/Download are *inside* it, not in the count."

**Scores (out of 10):**
- Feature completeness: 10 — hard-requirement checklist in README is exhaustive: all 5 SM modes with real content, all 6 start sources, 18 of 43 blueprints with 10 filter chips, both Export PR + Download .zip in overflow, save-state transition visible across multiple states (amber pip → green pip), real blueprint JSON with 7 steps (installTheme + installPlugin + importWxr + runSql + setSiteOptions + updateUserMeta), 11 realistic PHP log lines including PHP Notice + Deprecated + Fatal + WASM-GD shim retry. The "WASM-GD shim retry" log line is a particularly authentic detail.
- Restraint: 10 — 6 idle affordances, defended. The pane is the only secondary; everything else is contained.
- Iframe primacy: 10 — iframe full-bleed at idle except 24 px rail. F key + button + overflow each open fullscreen; Esc exits. Snap detents (25/50/75 %) are explicit.
- One-door: 10 — README's enumeration is exemplary: Settings/Files/Launcher (Sites)/Save/Exports each have *one* canonical home. "Triple-entry-point obliterated."
- One-secondary: 10 — literally one pane. Six modes swap via internal icon rail. No sidebars, no drawers, no popovers (except the overflow which is shown open as proof).
- Direction adherence: 10 — pull-tab visible at every state; snap dots track the detent (visual "where am I" gauge); all three detents demonstrated across states D/B/C. The gesture is annotated in state A — explicitly applying the #001 review lesson.
- Visual fidelity: 9 — Inter + JetBrains Mono + Lucide icons. Real WordPress iframe content. Color-coded save-pip (amber unsaved, green saved). 18 blueprint cards rendered. Sub-tabs inside Sites (Start · Yours · Gallery) for sticky navigation — applying the #002 review lesson.
- Synthesis with v1 lesson: 10 — extends #55 iPad pull-tab + #15 fullscreen + #64 DevTools slim chrome. **And explicitly applies two prior v5 review lessons**:
  - "the v5 critic flagged [stacked drawers] in review #002" → Sites mode uses sticky sub-tabs (Start · Yours · Gallery).
  - "the expansion-gesture-as-first-class-element lesson the v5 critic taught us in review #001" → contextual hint chip ("Drag left or click · ⌘\\") in idle state.
  Second-best synthesis-with-prior-reviews in v5 so far (tied with #018; #018 cited three reviews, this cites two by number).

## What works (specific)
- **Snap dots on the pull-tab track double as a "where am I" gauge.** This is a brilliant economy: the same visual element communicates current state *and* serves as the resnap-back affordance ("pressing the active dot toggles back to the previous detent in one click"). Spatial memory without spending an affordance.
- **Sticky sub-tabs in Sites mode (Start · Yours · Gallery)** explicitly apply the v5 #002 review lesson. Three former drawer sections become three discrete tabs the user can reach in one click *at the 25% peek detent*. This is the *exact* fix #002's review called for, and it's labeled as such in the README.
- **Contextual hint chip "Drag left or click · ⌘\\"** in idle state applies the v5 #001 lesson — making the expansion gesture a first-class element rather than a hidden interaction.
- **`⌘\\` as the keyboard shortcut** is a smart choice: rare enough to not conflict with anything common, mnemonically related to "pane" (the backslash splits the screen).
- **The name pill is the save verb**, with amber pip = unsaved, green pip = saved. Click the pill = save. No separate Save button. Cleanest save-state-as-action in v5 alongside #014 and #011.
- **Per-mode detent memory**: "Files and Blueprint modes … default to the 50% detent on first open and remember per-mode." Context-aware default detents — better than a single snap-position.
- **The README's hard-requirement checklist** is exhaustively verifiable. Every bullet is paired with a specific state and content example. This is critic-grade self-disclosure.
- The **overflow menu is shown open in state A** to verify Export PR + Download .zip live inside it, not as additional affordances. Pre-emptively countering the "is it really only 6?" objection.

## What's weak or missing (specific)
- **At 75% detent, the iframe is reduced to ~360 px (a 25%-wide sliver).** README admits this is "cramped for any mode where you'd want to *see* the site while working." This is the design's real cost; for Files and Blueprint modes (where the user isn't watching the iframe), fine — for Settings (where Apply&Reset's pre-reset state should be visible), less ideal.
- **Mobile detents are 35/70/92** instead of 25/50/75. README defends as "a necessary scale, not a different design." Fair, but at 92% detent the mobile iframe is reduced to 8% of screen — effectively gone. This is a real iframe-primacy violation on mobile at the max detent.
- **The mobile overflow menu isn't shown open** the way desktop state A is. README rates -1 honestly. A future iteration should add the mobile equivalent.
- **Pull-tab is 24 px wide** — same width as the direction prescribes. But 24 px on a 390 px mobile screen is 6.1% of width *always-visible* — that's a chunky permanent affordance for mobile. Compare to mobile entries where the iframe is full-bleed.
- The top bar has a `backdrop-filter: blur(12px)` over a 0.96-opaque white — README admits "barely noticeable and could be flat without loss." Honest, and worth flattening in a next iteration for pure restraint.
- **No Database content shown in panel B/C** per the checklist — Database is in panel E. Worth verifying it has comparable fidelity to other tabs (the README claims 7 steps in blueprint JSON, 11 log lines, etc., but Database content details aren't enumerated).

## How to push harder (1–2 pointed suggestions)
1. **Add a 4th micro-detent at 90% explicitly for "fullscreen the pane"** that *hides* the iframe entirely and presents the pane as a maximized editor. This formalizes the 75%-detent-where-iframe-becomes-a-sliver into a deliberate "pane-only" mode, paired with a single "show iframe again" return chip. Detents become 25/50/75/full-pane — four named modes, each clear.
2. **Mobile: make 92% detent unreachable; cap at 70%.** The mobile iframe should never drop to 8% of screen. Three detents (35/55/70) keeps the iframe visible at all times and matches the desktop's iframe-primacy spirit. The mobile pane is plenty at 70%.

## Lesson for the next designer
Cite prior reviews by number *and* show the fix in the design. Pasting a one-line "Lessons applied from earlier v5 reviews" section in your README and labeling the implementing element (e.g., "sticky sub-tabs in Sites mode → review #002") makes the synthesis visible to the critic. Don't make us hunt for the lesson.
