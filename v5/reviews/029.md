# Review of v5 #029 — Breadcrumbs nav

**Parent direction:** "Breadcrumbs nav. Site › Settings › PHP. Every navigation is breadcrumb-driven."
**Affordance count at idle:** 8 (annotation strip in panel A): refresh · root crumb · site crumb · URL pill · save pill · fullscreen · more · +new. At cap exactly.

**Scores (out of 10):**
- Feature completeness: 10 — panels A–M render every state including: idle, root-crumb dropdown, site-crumb dropdown, leaf-crumb as sibling-file-switcher (E), settings + Apply&Reset typed phrase, files, blueprint, database, logs, 43-blueprint gallery (panel J), export menu (panel K), fullscreen (panel L), mobile (panel M).
- Restraint: 9 — exactly 8 at idle, defended. Mobile drops to 6.
- Iframe primacy: 10 — iframe gets 100% of stage at depth-2 idle; panel L is true fullscreen + exit pill + Esc shortcut. Fullscreen is a visible icon (citing #013 lesson) plus keyboard shortcut.
- One-door: 10 — settings only via "Settings" leaf; exports only via More; launcher only via root crumb. The trail is the only navigation primitive. Best one-door discipline in v5 (alongside #018, #023, #027).
- One-secondary: 10 — literally one pane that mounts under the trail; segmented controls and dropdowns swap content.
- Direction adherence: 10 — path-as-IA is taken literally — even file switching in panel E is a crumb-dropdown (leaf-crumb is the file picker, not a tree-explorer-in-disguise). The metaphor is committed and consistent.
- Visual fidelity: 9 — Inter font (presumably), real Lucide icons, no glass anywhere (solid chrome per v5 §2 rule 6), exit-pill is the only translucent element. The leaf-crumb-as-sibling-switcher in panel E is a novel state that proves the metaphor.
- Synthesis with v1 lesson: 10 — extends #14 (Ventura clean sidebar = clean breadcrumb instead) + #15 (fullscreen toggle). **Cites *six* prior v5 reviews by number with specific implementing element** — the most synthesis evidence in v5 so far, beating #026 (5):

  - Review 006 → "don't import a clever idea from a different direction" → no Control Center, no Cmd+K, no spatial dock, no auto-hide HUD. "The trail is the entire IA."
  - Review 013 → "fullscreen toggle must be a visible icon" → maximize icon is affordance #6.

  - Review 018 → "cite prior v5 reviews by number" → done explicitly.

## What works (specific)
- **Hover-to-list-siblings on each crumb** is the design's IA accelerator: you can swap leaves without first popping. Settings → Logs in one move, not two. This compresses the depth-first nature of breadcrumbs into a tab-like lateral movement.
- **Leaf-crumb as file picker** (panel E) is the cleanest implementation of "path-as-everything" in v5. The Files leaf doesn't open a tree view; it opens a *crumb dropdown listing files*. The metaphor is consistent — there's no "tree-explorer-in-disguise" hiding inside the pane.
- **Six review citations** with specific implementing element each — this is the gold standard for v5 README synthesis. Every prior critic lesson has a checkable implementation.
- **No glass anywhere** (zero glass, 5/5 honest) — top strip and pane are both solid. Only the fullscreen exit-pill is translucent, and it's *over* the iframe (depth-justified).
- **Apply & Reset typed phrase** + "don't import other directions" restraint — the design stays in its own lane.
- **Mobile path wraps to its own scrollable row** (panel M) — solves the narrow-screen problem cleanly. URL pill drops because "the active crumb already shows location" — clever reduction.
- The "+new" affordance (eighth idle slot) is the launcher entry — single-tap to the root-crumb dropdown with the "New" segment focused. Direct deep-link.

## What's weak or missing (specific)
- **Hover-to-list is the lateral-switching mechanism, but on touch there is no hover.** README acknowledges that on mobile the trail wraps — but how do mobile users swap siblings? Tap-and-hold? Tap once to navigate, tap again for siblings? Not addressed explicitly.
- **Lateral switching via hover is slower than tabs**, README admits. For users who flip Files ↔ Logs many times per session, this is a real cost. The defense ("users learn one interaction instead of two") is reasonable but adversely affects power-user flow.
- The **pane is fixed at 56% of stage height** with no resize. Users with long log lines or wide DB rows lose horizontal real estate. README defers to future iteration ("drag-to-resize would be a future addition, not part of this direction's promise").
- **Crumb dropdowns are stable list dropdowns.** A user with a deep path (`playground › Stylish Coffee › Site Manager › Files › wp-content › themes › ... `) faces a long horizontal trail. Probably wraps as needed but worth a state demonstration.
- The **URL pill is a separate affordance from the trail** (idle affordance 4). But the trail already shows location semantically (path-as-IA). Two ways to read "where am I." Why not collapse the URL pill into the trail's leaf? Could free an affordance slot.
- **+new is a dedicated affordance separate from the root crumb's dropdown.** A user clicking "+new" arrives at the same place as clicking the root crumb → "New" segment. Two doors to the same destination — the design admits this with the deep-link defense, but a strict reading is duplicate door.

## How to push harder (1–2 pointed suggestions)
1. **Collapse the URL pill into the leaf crumb.** The trail already shows location; the URL is just an editable form of the leaf path. Make the leaf crumb editable (click into it = focus mode that accepts text input) and remove the URL pill from the strip. Idle count drops to 7 — and the breadcrumb metaphor becomes purer ("the trail is the URL").
2. **For mobile, replace hover-to-list with a *swipe-left on the active crumb* gesture** that shows siblings as a horizontal scrollable list. This preserves the lateral-switching speed on touch (one gesture, not two taps) and visually echoes the crumb structure (siblings are *adjacent* in the trail).

## Lesson for the next designer
If your direction has lateral siblings reachable by hover, you must explicitly answer "what's the touch equivalent?" — hover is not a touch primitive, and a tap-then-hold or swipe-left is not free. Design the mobile gesture *first*, prove it on phone, then check that desktop hover is the optional accelerator.
