# Review of v5 #027 — Status-only top bar

**Parent direction:** "Status-only top bar. Slim 24px bar with just PHP / WP / saved status. No buttons. Everything else via Cmd+K."
**Affordance count at idle:** 7 interactive + 1 passive label (per README's count): name+rename chip · save-state chip · WP chip · PHP chip · DB chip · BP chip · fullscreen pill · `⌘K` hint (passive). At cap (8 ceiling) exactly.

**Scores (out of 10):**
- Feature completeness: 10 — 3 states explicitly demonstrated: working session, booting, launcher/palette with four scopes shown. The palette covers /sites + /blueprints (18 cards · 10 filters) + /settings (sentence editor) + composite of /files + /blueprint + /database + /export + /logs. Typed-phrase Apply&Reset + diff preview + auto-snapshot in /sites for 24h. WP Admin + Homepage + Export PR + Download .zip all in /export.
- Restraint: 10 — 7 interactive + 1 passive at cap exactly. The bar is *typography, not navigation* — explicit framing.
- Iframe primacy: 10 — iframe is everything except 24 top pixels + 32×32 corner pill. Three paths to fullscreen (button + ⌘· + Esc to exit).
- One-door: 10 — Save: only the status chip (no separate button). Settings: only /settings. Export: only /export. Chip→palette scope-jump is a *single gesture* opening one *single destination* — not a duplicate door. Cleanest one-door discipline alongside #018 and #023.
- One-secondary: 10 — one palette container; every panel is a *scope* inside it (CLI-style, like #015). Same architecture as #015 but pushed further: zero chrome buttons, only chips.
- Direction adherence: 10 — "Status-only top bar; no buttons; everything via ⌘K." The result is literal: 24 px bar, six monospace chips, zero buttons in the bar. The fullscreen pill lives outside the top bar (correctly, since the brief requires it).
- Visual fidelity: 9 — monospace status chips (JetBrains Mono presumably), real Lucide icons in the palette, sentence editor for settings (Notion-style), 18 blueprint cards + 10 filters, real WP iframe, real booting state with chip color flips ("Saved · 2m" → blue "Booting · 62%", "DB sqlite" → "resetting…").
- Synthesis with v1 lesson: 10 — extends #64 DevTools to its limit. **Explicitly cites three v5 reviews:**
  - "the lesson from direction #50" (Notion-style settings as sentence)
  - "(per #016's 'if it's touchable, label it; if it's engraved, don't')"

## What works (specific)
- **The status bar reports state, doesn't operate on it.** This is the conceptual breakthrough. `coffee-shop.wpsite · Saved 2m · WP 6.7.1 · PHP 8.3 · DB sqlite · BP coffee-shop@v3 · ⌘K` is a *sentence*, not a toolbar. The chips' interactivity is a learned secondary affordance ("click to jump to where this fact is editable").
- **Click `PHP 8.3` → palette opens to /settings with PHP fragment focused.** This is *deep linking into the palette*, not opening a popover. README calls it "scope-jump" — the chip is a query, the palette is the editor.
- **Save chip *is* the save door** — clicking it focuses /save; ⌘S commits anywhere. No "Save" button anywhere else. Cleanest save-state-as-action in v5 alongside #014 and #023.
- **Apply & Reset with three guardrails: diff preview + typed-phrase confirm + auto-snapshot recovery in /sites for 24h.** The 24-hour recovery window is novel — it's *recoverable destruction* with an explicit timer, not just a one-shot snapshot.
- **The fullscreen pill is the only persistent button anywhere on screen** — the v5 §2 rule 1 ("one-key/one-click toggle") is met with a real visible button outside the chip bar. Principled.
- **`/sites` co-locates saved + start-new** instead of having a separate /new — README defends this as "/new is a sub-scope that auto-scrolls to the same list." Avoids the soft duplication seen in #015, #018, #024.
- README's "cursor stays an arrow" disclaimer for the chips (per #016's lesson) — even cites the review and applies the rule.
- Cmd+R is left to the browser (cite to #015) — avoids the browser-shortcut hijacking that #015's review warned against.
- Mobile uses a bottom-center "Find anything /" pill as the touch substitute for ⌘K — same architecture, different gesture.

## What's weak or missing (specific)
- **Discoverability tax is real and intentionally accepted.** README pre-empts: "A first-time visitor sees six status chips and the words 'Press ⌘K' — they may not know that the chips are jump-buttons." Mitigation: a one-time on-load hint that fades after 2s. Same critique as #014 and #015. The chips look like typography, so users may not try clicking — losing the scope-jump affordance.
- **The chips collapse to 4 on mobile** (name, save, WP, PHP), dropping DB and BP. README admits this as "information loss." For mobile users wanting to verify DB/Blueprint state, they have to open the palette. Acceptable trade but worth a "show all status" tap target.
- **No content-aware tint** (README, -1 honest). The frosted bar is static, not iframe-color-aware. Compare to #010's ambient rails which sample iframe color. Could be added with JS sampling.
- The **`⌘K` hint pill** is "labelled passive text; tappable but visually distinct from a chip via lower contrast and the monospace `⌘K` glyph." This is a 4-byte phrase. Tappable but visually low-contrast: a returning user may not realize it's tappable. README counts it as +1 of the 8-ceiling — okay, but the disposition (low contrast) makes it less of an affordance than the chips.
- **The "scope-jump" gesture has to be taught.** A click on `PHP 8.3` doesn't *immediately* change PHP; it opens the palette focused on /settings with PHP highlighted. A user expecting a dropdown gets a palette instead. The mental model is "this chip is a deep-link into the editor" — learnable but not obvious.
- Booting state (Frame 6) shows save-state chip flipping to "Booting · 62%" and DB chip to "resetting…" — but these are *display changes*, not affordances. The fullscreen pill becomes a Cancel-boot button (good — context-aware reuse). README is clear about this but mobile users may not realize the pill changes function.

## How to push harder (1–2 pointed suggestions)
1. **Add a subtle "→ palette" arrow icon on chip hover** (desktop) or always-visible at 30% opacity (mobile). The chips look like text; without a visual cue that they *jump somewhere*, users will gesture them like dropdowns. A tiny `↗` glyph next to each chip on hover makes the scope-jump pattern visible without spending a permanent pixel.
2. **Move the BP and DB chips into the `/sites` site cards on mobile** — README says they "collapse to 4 chips" but doesn't say *where* the data goes. Make the active site card in /sites carry per-site WP/PHP/DB/BP metadata, so mobile users can still verify state — they just have to open the palette to see it. Information loss becomes information *relocation*.

## Lesson for the next designer
Status chips that *are also* buttons need a *visual differentiator* from pure typography — a hover arrow, a faint underline, anything. The "engraving vs button" lesson from #016 cuts both ways: if your text is interactive, make it look slightly more so than pure data; otherwise users will read past it.
