# Review of v5 #004 — iPad Stage Manager lite

**Parent direction:** "iPad Stage Manager lite. One focused playground at full screen, two recessed thumbnails on the left, frosted dock of saved sites at the bottom."
**Affordance count at idle:** 8 (enumerated onscreen at line 938–941): name pill · URL bar · Save · Refresh · Control Center · Recessed peers · Site Manager pull-tab · Dock. The dock visibly contains ~8 site avatars + the "+ New" tool, treated as one widget. Honest count given the cluster definition.

**Scores (out of 10):**
- Feature completeness: 9 — all six new-source types, 43-blueprint gallery, all 5 Site Manager tabs, both Export-to-PR and Download .zip in CC, WP Admin & Homepage jumps, typed-phrase Apply&Reset (state 7), booting state with mid-boot log (state 2), explicit fullscreen state (state 5), 7 states total.
- Restraint: 8 — 8 idle affordances counted carefully; everything else hidden behind pull-tab / Control Center / + New launcher. The recessed-peers column is itself two pre-loaded thumbnails — technically 2 more visual elements, defended as "one cluster."
- Iframe primacy: 8 — iframe is the centered stage with mac window chrome; the pull-tab at 440 pt + recessed peers at left compress the iframe horizontally when SM is open. README admits this honestly: "the iframe-≥-65% rule … is not honored on iPad when the pull-tab is open at 440 pt." State 5 shows full-bleed fullscreen with F + Esc.
- One-door: 7 — Settings has *two* doors by design (CC for non-destructive tweaks, SM Settings tab for Apply&Reset). README defends as "different doors, different purposes." Defensible *if* the visual distinction is sharp; it isn't fully — both surfaces show version selectors. Save lives in both the top-bar Save button and (presumably) the dock-by-implication. Minor multi-door risk.
- One-secondary: 9 — the pull-tab is the single swappable surface for all 5 SM tabs; Control Center is a popover (different role); Launcher overlay is summoned only by + New tile. Three distinct surface *types* but only one *secondary panel*. Acceptable.
- Direction adherence: 10 — Stage + recessed peers + dock are all present and visually unmistakable. The metaphor is not borrowed-and-decorated; it's the IA's load-bearing structure.
- Visual fidelity: 9 — iPad shell with camera notch, iPadOS status bar, real Lucide icons, mac-style window chrome on the active iframe, color-coded dock avatars (coffee/gaming/resume/art/photo/welcome/unsaved), realistic WP admin landing with cards. The recessed peers carry tiny tb-chrome bars and badge labels.
- Synthesis with v1 lesson: 9 — extends #55 (iPad multitasking) explicitly with all three primitives; the README invokes the v1 metaphor by name. Also incorporates #15 fullscreen toggle and #38 glass; mobile concession (the recessed peers shrink to a sliver) is honestly disclosed and contained.

## What works (specific)
- The recessed peers (line 853–871) visually carry *more information than a tab strip* — each has its own mini window chrome, PHP version badge, and label. You can see "PHP 8.2 / My Coffee Shop" and "PHP 8.3 / Gaming News" without opening anything. That's genuine spatial memory in a small footprint.
- State 2 (Booting) shows a *peer* mid-boot, not the stage — meaning the user can boot a site in the background while staying focused on the current playground. That's a real iPadOS-inspired affordance, not a v1 idea borrowed.
- The dock (line 918–932) has a visible separator ("dock-sep") that quarantines the "Unsaved" playground from saved ones, plus a separator before the "+ New" tool — three semantic zones in one widget.
- Bypass: the README itself enumerates a feature mapping at line 9 — "Every Site-Manager tab … lives inside the *same* 440-pt pull-tab sheet" — and the brand-promise "Control Center for the eight quick-toggle settings; SM Settings tab for the destructive Apply-and-Reset flow only" makes the two-door split intentional rather than accidental.
- "+ New" as a dedicated dock tool (line 928–931) is a strong visual primary verb — the launcher is one *tile*, not a hidden menu.
- The "Control Center" popover (state 6) is the right place for non-destructive toggles; pull-tab Settings is the right place for the destructive flow. The split is principled — but read on for the visual-distinction critique.

## What's weak or missing (specific)
- **The Settings two-door distinction isn't visually crisp.** From the user's POV, both surfaces will show "WordPress 6.7" and "PHP 8.3" — one is read-only (CC) and one is editable + destructive (pull-tab). Make the CC version chip explicit as a *status pill* (uneditable, looks different from a select) so the user knows where to *go* when they want to change versions.
- The iframe is compressed to ~500 px wide when the pull-tab is open per the README. At 1440 px viewport with two 200 px recessed peers + 440 px pull-tab, the active iframe is genuinely thin. That's a real iframe-primacy violation that v5 §2 rule 1 ("iframe primacy. The WordPress site is the hero") would flag.
- Mobile (state in the same file, lines 514+ in CSS): the recessed peers compress to an 8 pt sliver per the README. That's a hard concession — and the v5 principle #7 (mobile = wider mobile for desktop) is explicitly weakened. The README rates this 3/5 honestly, which I'll respect, but the design's strongest IA primitive (recessed peers as spatial memory) effectively disappears at small widths.
- The dock holds 8 site avatars at idle on desktop. With the +New tool and two separators, that's 11 visible items in one cluster — which is fine if you treat the dock as one widget, but a strict critic would count individual avatars at 8+. The 8-affordance count depends on the cluster convention.
- "fs-hint" at line 907 is a small `<kbd>F</kbd> fullscreen iframe` chip — but it lives inside the stage iframe itself. Easy to miss; should be in the iframe-window's right-side chrome (where the maximize icon is at line 879) as the *button* that performs the action, not just a keyboard hint.

## How to push harder (1–2 pointed suggestions)
1. **Make the CC vs pull-tab settings distinction structural, not naming-only.** In CC, render WP/PHP as static *chips* with a small "Change to apply & reset →" link that *navigates* to the pull-tab Settings (deep-linking the secondary surface). Now there is genuinely one editor; CC is read-only-status + non-destructive toggles only. This eliminates the soft-seam README admits.
2. **When the pull-tab opens, animate the recessed peers and the stage into a temporary single-column "library" arrangement** — peers + stage become a vertical strip at the left, iframe stays ≥ 65% of remaining width. This converts the iframe-primacy compression into a deliberate trade where the user explicitly entered a "manage" mode.

## Lesson for the next designer
A two-door split for Settings is permissible *only* if the two doors are visually unmistakable as separate-purpose surfaces (status vs editor, quick vs destructive); render the read-only version as static chips, not select-boxes, or the brief's one-door rule kicks in.
