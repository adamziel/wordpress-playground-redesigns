# Review of v5 #030 — Two-row top bar

**Parent direction:** "Two-row top bar. Row 1: site identity + URL. Row 2: contextual actions for the active panel. Row 2 changes content."
**Affordance count at idle:** 8 (enumerated in README): URL+refresh · More menu · Fullscreen · Site mode · Manage mode · Library mode · Settings verb · Save. Disclaimers list brand mark, keyboard hints (1·2·3), saved-state pill, Homepage/WP Admin verbs, editable name as decoration/status/sub-elements. At cap.

**Scores (out of 10):**
- Feature completeness: 10 — 11 panels rendered including State A (Site mode idle), B (Manage mode), C (Library mode), E (logs dock), G (Files), H (Blueprint), I (Database), J (fullscreen), K1/K2/K3 (mobile). All requirements met.
- Restraint: 8 — 8 at idle with disclaimers. The two rows cost 76 px (40 + 36) — 28 px more than the slimmest entries (#009 at 32 px). README admits this and defends.
- Iframe primacy: 9 — iframe gets ~90% vertical at idle. State J shows true fullscreen + Esc-to-exit pill + F shortcut.
- One-door: 10 — Settings only in Manage; Save only as primary verb (#003 lesson); exports only in More; launcher 3 sections via Library mode (#005 lesson); editable name only in URL pill. Clean.
- One-secondary: 10 — literally one dock that opens between row 2 and iframe. Same geometry every time. Only content swaps.
- Direction adherence: 10 — two-row split *is* the IA; row 2 swaps content across 6 rendered states (A/B/C/E/G/H/I); mode-context coupling is annotated.
- Visual fidelity: 9 — solid surfaces (no glass except fullscreen exit pill), clear row hierarchy, keyboard hints (1·2·3) on the mode tabs.
- Synthesis with v1 lesson: 10 — extends #64 DevTools (slim top + dock) by *splitting* the top into identity + context. **Cites *five* prior v5 reviews with the specific implementing element:**
  - Review 001 → "show the expansion gesture" → State B captures row 2 morph with active mode pill rendered differently.
  - Review 005 → "don't nest a second tab strip inside a sheet" → Library mode dock has no inner sub-tabs; 3 sections side-by-side.
  - Review 009 → "decide what the kebab is for; don't make it a junk drawer" → More menu sorted into 2 purposes (Export, App-level help).
  - Review 013 → "expose fullscreen visibly" → icon in row 1 + F shortcut.
  - Review 018 → "cite prior reviews" → done.

## What works (specific)
- **Row 2 swaps content per mode** is the design's core gesture: 4 site verbs in Site mode (A), 5 SM tabs in Manage mode (B), search + 10 chips in Library mode (C). Different chips at three scales but always *the same row geometry*. The morph is shown as states.
- **Mode tabs with keyboard 1·2·3** in row 2 left — three-mode IA. Same primitive idea as #028's segmented controls but split across two rows so identity stays stable. Identity-vs-context separation is novel and principled.
- **Dock geometry is invariant**: opens between row 2 and iframe, same width every time. Six dock states (E logs / G Files / H Blueprint / I Database / B Manage / C Library) all render in the same place.
- **One-door discipline is exemplary**: editable name lives only in URL pill (not duplicated as Site Manager header title); WP Admin + Homepage as row-2 verbs in Site mode only (not in More); exports only in More.
- **Mobile keeps the exact same two-row metaphor — no hamburger collapse.** Three mobile states (K1/K2/K3) demonstrate. Hard 40% iframe floor on mobile dock.
- README cites five prior reviews with the specific implementation. Strong synthesis evidence.

## What's weak or missing (specific)
- **76 px of vertical chrome** is 28 px more than the slimmest (#009 at 32 px). README admits this. The iframe gets 90% vertical at idle, which is fine, but on a small-height laptop screen (768 px), 76 px of chrome is 10% off the top — noticeable. The chrome budget is the design's main cost.
- **8 affordances counted with several disclaimers** ("the editable name … is a sub-element of the URL pill") — the disclaim-list is 5 items long. A strict reading: the saved-state pill is interactive (clicking it focuses save), the Homepage and WP Admin verbs in row 2 Site mode are interactive. The honest count could be 10–11.
- **Three mode tabs with keyboard 1·2·3** — but on mobile (touch), there are no keyboard shortcuts. README doesn't specify mobile gesture for mode switching beyond tapping the tabs. Tapping is fine but the keyboard fluency is desktop-only.
- The **More menu sorted into 2 categories** ("Export this playground" + "App-level help") is cleaner than #009's flat junk drawer. But README doesn't show the menu opened explicitly — could use a panel state.
- **Apply & Reset typed-phrase confirm** is mentioned but no panel renders it explicitly (vs many other v5 entries that have a dedicated confirm state).
- The **anatomy panel below State A** (the labeled affordance diagram) is excellent self-disclosure but it's an *annotation* layer, not part of the design. Worth noting it's not chrome.

## How to push harder (1–2 pointed suggestions)
1. **Collapse row 1 to 32 px** by moving the brand mark into the URL pill (as a favicon-style anchor) and removing the editable name from row 1 entirely — let it live in Manage → Settings only. This recovers 8 px of vertical and reduces idle disclaimers (no "name is a sub-element of URL pill" caveat). Total chrome becomes 68 px instead of 76 px.
2. **Demonstrate the typed-phrase Apply & Reset confirm in a dedicated panel.** README mentions it but no state explicitly shows the confirm UI. Add a state with the destructive flow rendered: diff preview + typed phrase + the active "Confirm" button only after the phrase matches.

## Lesson for the next designer
A two-row split is the most legible "stable identity + contextual actions" IA — but you pay in vertical pixels. Don't try to win the slim-chrome contest if your direction commits to two rows; instead, prove the *trade* by showing row 2 swapping content across 5+ states. The states are the proof that the second row earns its pixels.
