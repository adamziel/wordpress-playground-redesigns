## What works (specific)
- **Traffic lights as state carriers, not window controls.** Red pulses red-deep on dirty settings; green hollow-pulses when unsaved; yellow halos when refreshing/booting. Each 14 px circle is *also* a status LED. Compresses three idle affordances + three status indicators into three pixels each. Cleanest signal-density-per-pixel in v5.
- **"Familiar shape, unfamiliar use"** is genuinely subversive design. macOS users *expect* red = close; this design rewires it to red = Settings. Mitigation is the hover-glyph (sliders icon fades into the red dot, rotate into yellow, check into green) plus a one-time pulse on first visit. New users learn in 5 s.
- **Persistent corner-lights at 40% opacity in fullscreen** is the design's most innovative state. The metaphor stays visible *even when chrome is hidden*. Reviews #011 (auto-hide teaching layer) and #014 (corner-pill always-on) merged into one element.
- **5 review citations with concrete element labels** — the README is explicit about *what* CSS class implements *which* prior review's lesson. Best implementation evidence in v5.
- **Spotlight is the universal launcher** (Sites, Start new, Blueprints, Commands) and the same surface is reachable from both the Sites button *and* ⌘K. README correctly notes this is the same surface (not a duplicate door).
- **Export/Download on the dock-right** is contextual to where the action makes sense (the dock is open when you'd export). Best contextual-action placement in v5.
- **macOS confirm-sheet conventions** for typed-phrase Apply&Reset — the design even uses the platform-native modal aesthetics.

## What's weak or missing (specific)
- **Discoverability of light meaning is the design's biggest risk.** README admits this and lists mitigations (hover glyphs, one-time pulse, tooltip names). But a returning macOS user *muscle-memory clicks the red to close* — and gets Settings. That's a real cost the README acknowledges but can't fully solve.
- **Dock is fully hidden at idle** (no peek strip). README acknowledges this as a trade — discoverability of Files/DB/Logs depends on the panel-bottom icon being recognizable. The Site Manager dock toggle (idle affordance 8) is the only hint. New users may not know there's a dock.
- **Mobile drops dock-tab labels** (-1 honest). Icon-only tabs are universally recognizable but lose meaning for less-fluent users.
- **The Sites menu and Spotlight ⌘K both summon the same Spotlight overlay** — README notes "the same surface is summoned by the Sites button *and* by ⌘K." Defensible (same destination, two entry points) but could be argued as a duplicate door under strict one-door reading. README counts as one door, which is reasonable.
- **The fullscreen hint pill fades after 6 s** — but what about the user who alt-tabs away and returns 30 minutes later? They've forgotten Esc/F. The persistent corner-lights at 40% are the teaching layer, but they don't *say* "Esc/F" — only the lights themselves.
- **The toast is glass** but README admits "could be solid without losing meaning" (-1). Honest small calibration.

## How to push harder (1–2 pointed suggestions)
1. **Show all three light meanings *inline* at first session via a 1.2 s reveal animation.** On first page load, each light briefly displays its label below itself (Settings / Refresh / Save) — then collapses to just the colored dots. This is the v5 #011 "one-shot teaching state" lesson taken further — *the metaphor teaches itself* without depending on hover (which doesn't exist on touch).
2. **Add a one-line passive hint in the title bar** ("Coffee Shop · WP 6.7 · PHP 8.3 — ⌘K to search, F to fullscreen") that reads as macOS-native title text. The title bar already carries passive label text; adding two shortcut hints turns the macOS chrome into a passive teaching layer for both lights *and* keyboard.

## Lesson for the next designer
When you rewire a familiar primitive (macOS traffic lights repurposed), the *teaching layer* must be visual and persistent — hover glyphs are good, but persistent 40%-opacity overlays in fullscreen mode are better. If the metaphor is so familiar that users will misuse it on muscle memory, your design needs a *constant* counter-cue, not a one-time tutorial.
