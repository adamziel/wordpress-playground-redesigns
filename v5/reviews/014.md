# Review of v5 #014 — Single word in a corner

**Parent direction:** "Single word in a corner. Only the playground name appears at idle (top-right corner, monospace). All chrome appears on hover. Period."
**Affordance count at idle:** 1 — the corner pill (`roast-and-bean ⌘K` at line 1453–1457). The save-state LED is declared status, not interactive. Strongest restraint score in v5 so far.

**Scores (out of 10):**
- Feature completeness: 9 — 10 states, real WordPress sites (Roast & Bean coffee in state 1, Twenty Twenty-Five magazine in state 2), all 5 SM tabs each get their own state (6 Blueprint, 7 Files, 8 Database, 9 Logs, 10 Settings), Apply&Reset typed-phrase confirm in state 10, full blueprint gallery in state 4, booting in state 5. Database is a *dedicated state* with early-access notice — better than #007 which only icon'd it.
- Restraint: 10 — only 1 idle affordance. The most disciplined v5 entry.
- Iframe primacy: 10 — chrome never reflows the iframe; everything is summoned as overlays.
- One-door: 8 — README admits two mild seams: (1) Save is reachable from cmd-menu *and* ⌘S (acceptable as a shortcut/canonical pair); (2) "Open Site Manager" appears in the cmd-menu *and* as a dedicated icon in the hover toolbar. Two roads to the same destination — that's the one-door rule's exact failure mode the brief warns about.
- One-secondary: 10 — one right drawer with 5 tabs that swap content. No second sidebar.
- Direction adherence: 10 — "single word in a corner. All chrome appears on hover. Period." The README quotes the direction and executes it literally. No imported metaphors.
- Visual fidelity: 9 — monospace JetBrains Mono for the corner pill, real WordPress iframe content for two distinct themes (Coffee Shop + Magazine), Lucide icons, animated grow-from-corner reveal (220ms), real PHP log lines with search + level filter (state 9), real SQLite table (state 8).
- Synthesis with v1 lesson: 10 — directly extends #64 DevTools (slim chrome) and #15 Win11 (fullscreen toggle, F + Esc). The single-word-pill is a *refinement* of v1 #14 Ventura's "one home per feature" taken to its logical extreme. Mobile is the same gesture (tap not hover) — extends #06.

## What works (specific)
- **One affordance.** Reads as "1" in the README and is rendered exactly so. No other v5 entry achieves this.
- **The corner pill embeds the ⌘K hint** (line 1456) so the discovery problem is bounded: a user sees `roast-and-bean ⌘K` and has a clue. This is the single biggest mitigator of the discoverability gamble.
- **Hover → grow into toolbar** (state 2, line 1503–1518) is a *grow from the pill*, not a separate surface appearing. The README emphasizes this: "materialising as a *grow* from the corner so it's clearly the pill's expansion, not a separate surface."
- **Click → command menu drops below the pill** (state 3): different gesture, different surface, *same anchor*. The pill is a small affordance with two interactions producing two outputs — that's elegant.
- The save-state LED (line 1454) is a 6 px dot tucked inside the pill. Tiny, but always visible and never mistakable for action. Clean status-vs-action separation, the right way to do #003's failed split.
- Glass usage is *named and justified* in README: pill = ambient transparency; chrome-bar = brief materialise; drawer/launcher = secondary depth. Compliance with v5 §2 rule 6 is exemplary.
- All 5 SM tabs each get a dedicated state (6, 7, 8, 9, 10). Even Database has its own state with the early-access notice — fixing the gap in #006 and the icon-only treatment in #007.

## What's weak or missing (specific)
- **The "Open Site Manager" in hover-toolbar + cmd-menu is a real one-door violation.** README says "mild overlap I'm keeping is the cmd-menu's 'Open Site Manager' row plus the dedicated icon in the hovered toolbar — same destination, two roads." This is *the exact pattern v5 forbids*. Either drop the Site Manager icon from the hover toolbar (let ⌘. and the cmd-menu carry it) or drop the cmd-menu row.
- **Hover is desktop-only.** README scores mobile 4/5 and admits hover-on-pill toolbar preview is desktop-only. On mobile the gestures collapse to tap-only, losing the gradual-discovery affordance that hover provides. That's an honest concession but it weakens "mobile = wider mobile for desktop."
- **First-touch is steep**: a brand-new visitor sees only `roast-and-bean ⌘K`. They might not know to hover, click, or type. The ⌘K hint helps if they know what `⌘K` means. A "Click me" pulse animation on first load, or a 600 ms auto-grow demo (same idea I suggested for #007), would close this gap.
- The corner pill carries the *playground name* — but the name is editable (per the v1 brief). How does the user rename? Long-press? Click into the cmd-menu's "Rename"? The README doesn't say.
- State 4 launcher dims the iframe — README defends this as "softens the never-reflow rule visually, but the iframe still doesn't move — only its luminance drops." Fair, but luminance dimming may make the iframe content unreadable, and a user trying to *see* the iframe while choosing a blueprint loses information.
- The hover toolbar contains "Back" and "Forward" buttons (line 1506–1507) — but v1 §2 inventory doesn't mention back/forward navigation in the chrome (the iframe handles back/forward internally, and the address bar handles direct navigation). These look like *new features* sneaking in. Cut them or confirm they were always there.

## How to push harder (1–2 pointed suggestions)
1. **Eliminate the duplicate Site Manager door.** The hover toolbar should *only* hold the address-bar essentials (refresh, URL, fullscreen) — not a "panel-right" icon. Site Manager belongs *only* in the cmd-menu (and via ⌘.). The hover state then becomes the URL-editing affordance and the cmd-menu becomes the everything-else affordance — two surfaces, perfectly disjoint.
2. **Pulse the corner pill on first load** for 600 ms (single subtle outline animation), then settle into idle. This is a one-line CSS animation, not a new feature, and it solves "the new visitor sees one word and doesn't know to try it." Combine with a one-time `?onboarding=1` URL flag that auto-opens the cmd-menu on first ever session.

## Lesson for the next designer
A single-affordance design's only failure mode is *unclear which gesture produces which surface*; if your idle pill has hover-yields-X *and* click-yields-Y, those two outputs must be *non-overlapping* in destination — never let a chrome icon and a menu row point at the same room.
