# Review of v5 #007 — Quarter-pie FAB

**Parent direction:** "Quarter-pie FAB. A single floating action ball, bottom-right. Press-and-hold expands into a quarter-circle of 6 spokes (Refresh, Save, Sites, Files, Blueprint, Logs)."
**Affordance count at idle:** 4 (table in README): URL field · Name pill (saved-state + Save trigger) · Fullscreen toggle · FAB. The lowest idle count in v5 so far — half the cap.

**Scores (out of 10):**
- Feature completeness: 9 — 10 states rendered, 18 blueprint cards + 5 featured visible (23 distinct titles), all 6 start sources, 4 saved + Unsaved, Apply&Reset typed phrase ("reset crema"), Export GitHub PR + Download .zip surfaced in Settings drawer, 3-frame save-state transition (Unsaved/Saving/Saved). Mobile 4 frames. Database appears as an icon-btn rather than a dedicated drawer state — slight gap.
- Restraint: 10 — only 4 idle affordances. Strongest restraint score in v5 so far.
- Iframe primacy: 10 — FAB is anchored bottom-right and the README explicitly notes "the iframe — the hero — is never obscured above the diagonal." Drawer occupies left edge only when summoned; iframe stays visible. Fullscreen with Esc.
- One-door: 9 — Save is one spoke; Settings is *one* home (long-press the Sites spoke); the triple-entry sin is named and killed. WP Admin and Additional actions live inside Settings. The only soft seam: Save lives in two visual forms (the name pill color-codes save state + the Save spoke) — but the README defends the pill as *status* and the spoke as *action*. Cleaner than #003's slider+pill split.
- One-secondary: 10 — single drawer on the left, six contents, swapped by spoke. The README claim is exact: "One secondary surface. Six contents."
- Direction adherence: 10 — quarter-pie is mathematically exact (90° arc, six 15° spokes); fixed angles + fixed colors enable muscle memory; bottom-right anchor for right-thumb ergonomics. The direction's promise is literally executed.
- Visual fidelity: 9 — Inter font, real Lucide icons, color-coded spokes, syntax-highlighted PHP code (line 2080+) with line numbers, real blueprint JSON in state 5, real PHP error log (boot + WooCommerce warning + Fatal stack trace + slow-query) in state 6, four-frame mobile demo. The pie has an SVG arc guide (line 2100–2102) showing the 90° boundary visually.
- Synthesis with v1 lesson: 10 — extends #06 mobile-first (right-thumb ergonomics, bottom-right anchor, mobile-native gesture), addresses #15 fullscreen explicitly, the muscle-memory-via-fixed-angles+colors is novel and works at both widths. The "one verb, one home" mantra at the end of the README is the v5 brief's exact language.

## What works (specific)
- **Fixed angle + fixed color per spoke** (line 2107–2136) is the cognitive payoff: spoke 1 (red, top) = Refresh, spoke 2 = Save, spoke 3 = Sites, spoke 4 = Files, spoke 5 = Blueprint, spoke 6 = Logs. README's claim that "muscle memory replaces label-scanning within ~3 uses" is plausible because each spoke has its position invariant.
- **Number-key shortcuts (1–6) per spoke** are real, visible kbd chips (line 2110, 2115, 2120…). Power-user fluency is built in: no separate keyboard-shortcut help screen needed because the hint is on the spoke itself.
- **The pie also clicks as well as drags**: README notes "a single tap also opens the pie persistently." This handles the web-latency caveat (80–120 ms drag sweet spot is not guaranteed in the browser) without breaking the direction.
- The drawer's *head label and breadcrumb chip* tell you which spoke you came from (line 2149) — solving the "I forgot which spoke this is" problem that radial menus often have.
- Settings as a *long-press* on the Sites spoke (line README) is a clever IA: 6 spokes + 1 long-press = 7 destinations, with Settings hidden behind a deliberate gesture (matches the brief's "destructive Apply requires a typed confirm phrase" since Settings is where Reset lives).
- The README's "Affordance accounting" table at line 25–32 enumerates 4 idle affordances explicitly — far below the cap. The "toast region is reactive (not an affordance)" disclaimer is honest.

## What's weak or missing (specific)
- **First-touch discoverability** is the design's biggest gamble. The FAB is one ball. A new user doesn't know it's a pie until they press it. README acknowledges this. Mitigation: spokes always show labels under icons (the README claims this) — but at idle there are *no labels visible*. The FAB has only the `plus` icon (line 2104). One subtle "long-press for menu" tooltip on first hover or a quick auto-expand-and-collapse on first page load would close this gap without spending an affordance slot.
- **Database tab** is only reachable as an "icon-btn in Settings drawer head + the icon in Sites long-press path" per README — not a dedicated spoke. v5 §3 says "The five Site-Manager tabs (Settings, Files, Blueprint, Database, Logs) all reachable; each has visible content in your demo." Database may be reachable but if it doesn't have visible content in any of the 10 states, that's a compliance gap. (Need to confirm in the HTML.)
- The pie's 6 spokes are *fixed*: Refresh, Save, Sites, Files, Blueprint, Logs. Database is omitted from the pie because Settings doubles up via long-press. That's tight but it means *seven* features map to 6 spokes plus a hidden gesture — Database is the orphan that gets stuffed into Settings' nested drawer.
- The drawer (left edge) competes with the iframe's left content area. When the drawer is at 480 px wide, the iframe is compressed to ~960 px — still ≥ 65%, but the *coffee shop site nav* is now squashed against the drawer's edge. Not as graceful as the rails of #010.
- "Glass appears in exactly three places (address pill, drawer, spokes)" per README — but each spoke is a glass element, so 6 spokes + 1 drawer + 1 pill = 8 glass surfaces, not 3. Either the spokes are not glass (clarify the README) or the count is higher.
- The Settings-via-long-press-on-Sites gesture is *clever but unsignaled*: a new user pressing Sites gets the launcher, never knowing Settings exists until they long-press. The pie's labels never show "long-press for Settings." Add a small ⋮ on the Sites spoke as a hint.

## How to push harder (1–2 pointed suggestions)
1. **Promote Database to a 7th spoke or fold it into Files.** The quarter-pie's six-spoke fixed-angle muscle-memory premise breaks if you have a hidden 7th destination behind a long-press. Either: (a) make it a half-pie with 8 spokes (Refresh / Save / Sites / Files / Blueprint / Database / Logs / Settings), or (b) merge Database into Files as a sub-tab inside that drawer (rationale: both are content-inspection tools). Right now the long-press hides the destructive home, which is *backwards* — destructive actions should be more discoverable, not less.
2. **Add a one-time auto-expand on first session.** The first time a user visits, the pie opens for 600 ms and collapses, demonstrating the gesture. Like a self-onboarding animation. This is a UX pattern the brief allows (it's not a new *feature*; it's a state transition) and it solves the entire discoverability gap.

## Lesson for the next designer
A radial menu's promise is muscle memory, but a hidden 7th destination behind a long-press *breaks* the promise — once a user learns "spokes 1–6", a hidden gesture for Settings feels like a betrayal. If you have 7+ destinations, expand the arc; don't smuggle.
