# Review of v5 #036 — Corner-widget Control Center

**Parent direction:** "Corner-widget Control Center. Frosted square widgets in each of the four corners (Sites · Settings · Logs · Sites-Manager). Always-visible at 64×64. Click → expands toward center."
**Affordance count at idle:** 8 (enumerated in frame A): 4 widgets + refresh + URL + name-pill + fullscreen toggle. At cap exactly.

**Scores (out of 10):**
- Feature completeness: 10 — 12 frames covering all expansion states, 5 SM tabs (Files/Blueprint/Database + WP Admin + Export), 3-tab Sites launcher (Start new / Your sites / Blueprints), typed-phrase confirm in Settings Danger Zone, frame L dedicated fullscreen with Esc pill, Export tab with GitHub-PR + .zip + WP Admin + Homepage side-by-side.
- Restraint: 9 — 8 exactly, enumerated. The four corners are exhaustive (you cannot have more), which is itself the metaphor's restraint mechanism.
- Iframe primacy: 9 — iframe full-bleed; widgets float (don't reflow); F + click + Esc. README admits Sites and Manager panels cover ~54% × 80-88% when expanded, leaving the iframe partially visible (right half + bottom-right corner) — honest trade.
- One-door: 8 — settings/files/logs/exports each have *one* corner. Save = name pill only (review #003 lesson). README admits "-1 for WP Admin appearing as a Manager tab AND a quick-nav button on the Export tab — defensible but a strict reading counts the second as a duplicate door."
- One-secondary: 8 — README admits 4/5: "the quadrant panel is the only swappable surface, but four corners means *which* corner is active changes the panel's geometry. Conceptually one surface, four anchors."
- Direction adherence: 9 — every detail reads as the direction (4 corners, glass widgets, live micro-status, expand-toward-center). README admits "-1 because widgets are 128px not the literal 64px — disclosed in trade-offs." The brief's "64×64" was strict; the designer chose 128×128 for legibility.
- Visual fidelity: 9 — Frosted glass widgets, live micro-status inside each, ⌘N keyboard chips per widget, distinct icon per corner, hover background-fill, iframe color bleed-through.
- Synthesis with v1 lesson: 10 — extends #38 glass + iOS Control Center (parent direction #3) but quadranted to 4 corners. **Cites *five* prior v5 reviews with specific implementing element**:
  - Review #003 → Save is *only* the name pill (+ ⌘S). Widgets carry status, not action.
  - Review #012 → 4 corners + ⌘1–⌘4 keyboard map + distinct icon per corner (compass points).
  - Review #017 → frame H shows Export tab open with both GitHub-PR + Download.zip visible.
  - Review #027 → micro-row of icons visually distinct from the bold status line so users learn "header = label, primary = status, micro = quick-actions."
  - Review #029 → mobile-first gestures; tapping the widget on mobile = same action as clicking on desktop. Hover is bonus.

## What works (specific)
- **Four corners = exhaustion of geometry as restraint mechanism.** You *cannot* have a 5th corner. The metaphor is self-limiting in a way other direction-driven IAs aren't. Elegant.
- **Compass-point muscle memory** — top-left, top-right, bottom-left, bottom-right. Four positions are *much* easier to learn than #007's 6 spokes or #012's 8 slices. README cites review #012's "8 slices strain muscle memory" and applies the lesson by going to 4.
- **Live micro-status inside each widget** — each widget is *both* status display *and* canonical door. The 128 px scale is defended as needed for legible micro-data ("name, pip, two-line secondary"). Honest trade.
- **Expand toward center, never crossing into the opposite corner** — the geometric invariant means the *other widgets stay visible and live* during panel use. Spatial memory is preserved.
- **Save = name pill only** explicitly cites review #003 lesson. Widgets carry status, not action — avoiding the "Save in three places" violation that #003 was dinged for.
- **Five review citations** with concrete element each. Cumulative critic feedback synthesized cleanly.
- **The 64 px → 128 px deviation is explicitly disclosed** in trade-offs with the rationale (legibility of micro-status). Critic-grade transparency.
- **Glass earns its keep** — iframe color bleeds through every widget so corners "feel attached to the document instead of pasted on." Per v5 §2 rule 6 with named purpose.

## What's weak or missing (specific)
- **128 px ≠ 64 px** — README admits this is a strict-direction-adherence deduction (-1 honest). A purist would dock the score harder.
- **WP Admin as a Manager tab AND a quick-nav button on Export** is the documented duplicate door (-1 self-noted). Soft seam.
- **Sites and Manager panels cover ~54% × 80-88%** — leaving roughly half the iframe visible. README defends this as "iframe content stays partially visible." But the brief's iframe-primacy spirit (v5 §2 rule 1) is "iframe always visible AND there must be an explicit *fullscreen iframe* mode." Half-visible may not satisfy a strict reading. The dedicated fullscreen state (frame L) helps.
- **On mobile, expansion fills the screen** — corners don't grant enough space at 390 px. README admits this honestly (-1). The signature mini-quadrant expansion is desktop-only.
- **Widgets are 128 × 128 = 16,384 px² each × 4 = 65,536 px²** total at idle. On a 1440×900 = 1,296,000 px² viewport, that's 5% of the screen *just for the widgets*. Plus the URL pill (~36×800 = 28,800 px²) = ~7% of viewport for chrome. Compare to #032's 14 px traffic lights or #018's 56 px rail — corner widgets are pixel-greedy.
- The **panel geometry varies by corner** (Sites is top-left so panel goes down-right; Settings is top-right so panel goes down-left; etc.). README admits this is "conceptually one surface, four anchors." Strict critic reads this as four secondary surfaces with one shared component.

## How to push harder (1–2 pointed suggestions)
1. **Scale widgets to 96 × 96 instead of 128 × 128** — splitting the difference between brief's 64 and the designer's 128. Compress the micro-status to one line (name + pip), drop the two-line secondary, and the widget reads as iconic+status. Recovers ~30% of corner pixel cost while keeping the dashboard value.
2. **Remove the WP Admin quick-nav from Export tab.** It's already a Manager tab; the Export tab should only hold export-related actions (GitHub-PR + .zip + Download blueprint + Copy share link). Eliminates the documented duplicate door and the Export tab becomes purely about *getting work out*.

## Lesson for the next designer
If your direction specifies an exact pixel dimension (64×64) and you scale up for legibility, name the trade-off in the README *with the rationale* — "the brief asks 64; mine is 128; here's why" — and accept the direction-adherence -1. Pretending the deviation didn't happen costs more than disclosing it.
