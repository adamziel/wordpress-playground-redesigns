# Review of v5 #021 — Stage Manager v2

**Parent direction:** "Stage Manager v2. Focused site at 80% center. Two recessed thumbnails on the left (smaller saved sites). Bottom dock has tools. Click a thumbnail → swaps focus with a smooth zoom."
**Affordance count at idle:** 8 (numbered pins on State 1): name pill · URL · Save · Refresh · Fullscreen · More-menu · peers cluster · dock+Boot cluster. Dock-tile clusters disclosed in legend.

**Scores (out of 10):**
- Feature completeness: 10 — 9 states including idle (1), swap-zoom mid-flight (2), various panel modes (Settings/Files/Blueprint/Database/Logs), launcher with 6 sources + 43 blueprints + saved playgrounds, typed-phrase Apply&Reset, background-boot on a peer (State 8), full-bleed fullscreen with Esc banner (State 9). Mobile suite.
- Restraint: 9 — 8 at idle. Pixel-counted with state pins.
- Iframe primacy: 8 — iframe at 80% center in every running state. Panel compresses iframe to ~312 px tall when open — README rates 4/5 honestly. State 9 is true full-bleed with maximize chevron + F + Esc.
- One-door: 10 — Settings = one door (panel's Settings tab). Apply&Reset = one path. WP Admin/Homepage = contextual on stage chrome (NOT chrome-global). Export PR / Download zip inside Settings → Additional Actions. **Explicit divergence from #004 (lite cousin):** "I removed the Control Center popover entirely" because the critic flagged the two-door soft-seam.
- One-secondary: 10 — THE panel — one element that rises from the dock; all 5 SM tabs + launcher live inside. Same panel-head, same breadcrumb, same meta-pill, same close-X.
- Direction adherence: 10 — three direction-required primitives (focused 80% center / two recessed peers / bottom *tool* dock) all visually unmistakable. **The dock has TOOLS, not sites — structural differentiator from #004 lite.** Signature swap-zoom has its own state (State 2). Background-boot on a peer (State 8).
- Visual fidelity: 9 — Stage Manager-faithful three-primitive layout, macOS-style window chrome on the stage (lights, URL bar, contextual icons, maximize chevron), peer thumbnails with mini-WP content + PHP version badges, glass on three named surfaces.
- Synthesis with v1 lesson: 10 — extends #55 iPad multitasking *and* refines #004's lite version by addressing the lite version's critic feedback directly. **Cites review #001 by lesson** ("make the morph a first-class visible element") with State 2 implementing it. **Sibling-aware with #004**: "I deliberately diverged from v5 #004 (the lite Stage Manager) on three points the critic flagged" — itemized as Control Center removed, dock is tools not sites, iframe compression solved by docking at bottom not right.

## What works (specific)
- **Three-point divergence from #004 explicitly enumerated** with the critic's feedback as the rationale. This is the model for v5 sibling-design relationships:
  1. Control Center popover removed (addressing #004 review's two-door soft-seam).
  2. Dock is tools, not sites (per direction 21's exact wording, vs #004's dock-of-sites).
  3. Iframe compression solved by docking panel at 60% stage *height* rather than at the right edge.
- **State 2 demonstrates the swap-zoom mid-flight** with explicit 350 ms cubic-bezier transform — direct application of review #001 lesson (show the morph as a first-class state).
- **State 8 shows background-boot on a peer** — a peer boots while the user stays focused on the stage. Same context-aware affordance #004 had, but rendered as a dedicated state here.
- **Recents (peers) vs Library (launcher) is the cleanest catalog-split in v5**: peers = top 2 recents with quick swap-zoom; launcher = all 4+ with metadata. Different jobs, different surfaces, *not* duplicate doors.
- **WP Admin / Homepage as contextual buttons in the *stage's* URL bar** (not chrome-global). These belong to the active site, not the app. Recovered two affordance slots.
- **Export PR + Download .zip in Settings → Additional Actions** — one door, the rest of site-level housekeeping is there too.
- **Honest 31/35 self-score** with explicit -1 on iframe compression and -1 on mobile peer legibility. Critic-grade self-critique.

## What's weak or missing (specific)
- **Iframe compresses to ~312 px tall when panel is open** (README -1). That's at full panel extent. 312 px is roughly 35% of 900 — the iframe drops below v5 #2 rule 1's spirit ("iframe primacy"). README defends as user-initiated and reversible. Compare to #017 (snap detents) which gives more granular control.
- **The peer recents cap at 2**. A user with 8 saved sites sees only the most recent 2 as peers; the rest live in the launcher. README acknowledges this and defends ("launcher is one tap away"). Spatial memory limits at 2 — could be 3 if the peer column accommodates.
- **Mobile peer thumbnails lose mini-WP legibility** at 50 px wide (README -1, rated 3/5 honestly). They become bookmark tiles. The signature thumbnail-with-content metaphor is the design's strongest visual element; it degrades on mobile.
- **The Boot tile is one of 8 idle slots** — README acknowledges that "any navigation away from the current site costs one persistent affordance." Defended on frequency-of-use grounds. Fair.
- **Save acts both action and status** (button + pill, "always the same element" per README). Same elegant compression as #014 and others — but the README notes "action ↔ status" — clearly disclosed.
- The **stage's macOS-style window chrome** uses traffic lights as decoration (not as affordances like #032). That's a visual flourish, not load-bearing. Could be removed without losing function.

## How to push harder (1–2 pointed suggestions)
1. **Add a "show all recents" gesture on the peer column.** Tapping or hovering the peer column reveals a vertical list of all saved sites in compact form (favicon + name, no thumbnail). This gives users a third+ recents view without opening the launcher, restoring some of the spatial-memory loss when you have >2 saved sites. The gesture is the peer column itself — no new affordance.
2. **Render the iframe-compression-tradeoff as a "split mode" state.** When the panel opens, show the iframe + panel as a clearly-segmented vertical split, with a "swap" button that lets the user choose which gets the larger share (60/40 vs 40/60). Currently the iframe takes the upper region; let the user invert it for tasks where the panel needs more height.

## Lesson for the next designer
When a "lite" or "v2" sibling design exists, your README should *itemize the divergences with the critic's feedback as the rationale*. #021 enumerates three changes from #004 — each tied to a specific critic concern — and earns the reader's trust by showing the lessons-applied chain explicitly. Don't pretend the sibling doesn't exist.
