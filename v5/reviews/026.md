# Review of v5 #026 — Omnibox

**Parent direction:** "A single combobox top-center that handles URLs, blueprint search, command palette, and quick actions. No other top-bar elements."
**Affordance count at idle:** 3 (counted onscreen): state pill · input · fullscreen toggle. Tied with #006, #014, #015 for tightest restraint.

**Scores (out of 10):**
- Feature completeness: 10 — 15 demonstrated states (idle, focused+empty with scheme legend, bp:art with 18 cards + 10 filters, >exp commands, mgr:files with real functions.php, mgr:blueprint with Coffee Shop JSON, mgr:db with 12 SQLite tables, mgr:logs with WC fatal stack, mgr:settings sentence-editor + typed-phrase confirm + diff + auto-zip, booting, three pill states stacked, fullscreen, content-aware tint demo, pr:+gh: source modes, mobile suite, affordance ledger). Most thorough state suite in v5.
- Restraint: 10 — 3 idle affordances. Every secondary surface is summoned via the same input.
- Iframe primacy: 10 — iframe is the only other element at idle; fullscreen button visible on the omnibox (#013 lesson explicitly cited); F to enter, Esc to exit.
- One-door: 9 — each SM tab has exactly one prefix; settings is mgr:settings only; save lives only in the state pill. README notes "soft overlap is that `db:` and `log:` shortcut past `mgr:` — but they target the same surface, not a duplicate, so it's a fast-path not a second door." Defensible (and named in the brief: address-bar-driven, fast-paths are part of the metaphor).
- One-secondary: 10 — one omnibox dropdown; body swaps for every mode; no sidebar, no docked panel, no second drawer.
- Direction adherence: 10 — "the direction is the design; nothing else exists in the top bar; every feature in §2 was reshaped into a scheme." Eight schemes (`/path`, `bp:`, `>`, `pr:wp/N`, `pr:gb/N`, `gh:`, `mgr:*`, `db:` + `log:` shortcuts). Each scheme is a deep prefix that prefills via the legend strip.
- Visual fidelity: 9 — JetBrains Mono presumably, real Lucide icons, content-aware tint demonstrated with dark gaming theme, real Coffee Shop blueprint JSON, real PHP functions.php, 12 SQLite tables, real per-request log grouping with WC fatal stack, sentence editor for settings, diff preview + typed phrase + auto-zip.
- Synthesis with v1 lesson: 10 — refines #64 (DevTools omnibox-style chrome) + #29 (Console hybrid done right). **Cites *five* prior v5 reviews with the specific implementing element:**
  - L009 → "Always show the scheme legend on first focus" (pop-legend strip)
  - L013 → "Fullscreen toggle is *visibly* on the omnibox itself, not just keyboard"
  - L014 → "state pill's click target and `⌘K` hint point to the same place — non-overlapping by design"
  - L005 → "I do not nest a second tab strip inside the dropdown. mgr: sub-modes are flat scheme chips"
  - L003 → "Save is the state pill, not the state pill *and* a `>save` row competing for primacy"
  **Most prior-review citations in v5 — tied with #025 (4) and exceeded by this one (5).**

## What works (specific)
- **The scheme legend strip on focus** is the discoverability fix the omnibox metaphor needed. Eight clickable chips that prefill the prefix → mouse-only users learn the schemes without typing. Direct application of L009 review feedback.
- **Eight schemes** with exact mnemonics: `bp:` (blueprints), `>` (commands), `pr:wp/N` and `pr:gb/N` (PRs), `gh:` (GitHub), `mgr:settings|files|blueprint|db|logs` (Site Manager). Plus `db:` and `log:` as shortcut prefixes. This is the *omnibus* of omniboxes — every v1 feature mapped to a typeable prefix.
- **Content-aware tint** is demonstrated in state 13 — README's claim is verified by an actual render, not just an aspirational note. #010 also demonstrated this with two color examples; #026 keeps the discipline.
- **State pill carries amber-dot + inline Save button when unsaved**, green dot when saved. Save action lives only here per L003. Cleanest save-state-as-action in v5 alongside #014/#023/#027.
- **Apply & Reset is the most thorough destructive flow in v5**: sentence editor + typed-phrase confirm + diff + auto-zip safety. Matches the brief's "irreversible-but-recoverable" model.
- **Mobile is the same omnibox primitive** — dropdown becomes a vertical sheet anchored to the omnibox; filter chips horizontal-scroll. README rates 4/5 honestly.
- **One glass surface only** (the omnibox + its dropdown) — legend strip and dropdown body are solid for legibility. README's glass discipline is well-named: "tint is content-aware, demonstrated; legend chips are not glass."
- The "Lessons applied" section is *categorized by review-id with the specific implementing element*. This is the model for v5 README organization.

## What's weak or missing (specific)
- **Bottom-of-screen affordances are now homeless** — README notes "toasts pop above the omnibox dropdown, and a saved-sites 'dock' doesn't exist; saved sites live as the top section of the omnibox's empty-focus dropdown." Forces users to focus the omnibox to switch sites. Reasonable for keyboard-first but high-friction for *return* visitors who saw 4 sites in a dock yesterday and now have to type to find them.
- **`db:` and `log:` shortcuts past `mgr:`** are defended as "fast-paths not second doors." Strict reading: `mgr:db` and `db:` are two prefixes that open the same surface — that's two doors by the brief's literal meaning. README rates -1 honestly and the defense is reasonable.
- **The dropdown can grow tall (660 px) when in `mgr:` mode** but never wider — keeps the one-surface rule. Good. But 660 px on a 900 px desktop is 73% of viewport — the iframe is mostly hidden during management. Worth noting; not unique to this design.
- **Schemes are keyboard-first and English-coded.** A user who speaks neither English nor reads scheme prefixes well will struggle. The legend chips help (visual). But "type `bp:` to find blueprints" is still an opinionated metaphor.
- The **`pr:wp/N`** vs **`pr:gb/N`** distinction is mnemonic but easy to confuse (`wp` for WordPress core, `gb` for Gutenberg). Users may type `pr:wp/12345` when they meant `pr:gb/12345`. A small autocomplete dropdown showing "did you mean WordPress core or Gutenberg?" on `pr:` would help.
- **No traditional Save button anywhere** — the state pill is the canonical save. New users might miss this entirely on a Saved playground (the pill shows "Saved" → they can't see *how* it saves). Hover state should clarify.

## How to push harder (1–2 pointed suggestions)
1. **Add a persistent bottom-corner "Sites" pill** (similar to #014's corner pill) showing the active site name + a count of saved sites. Tapping focuses the omnibox to the empty-focus dropdown's "Sites" section. This restores the "dock" affordance the README admits is missing, without violating the one-secondary-surface rule — it's still the omnibox dropdown, just *summoned with the site list pre-scrolled*.
2. **On `pr:` typing, show inline disambiguation:** `pr:wp/12345 (WordPress core)` vs `pr:gb/12345 (Gutenberg)` as autocomplete rows. Eliminates the prefix confusion and proves the scheme legend is more than a one-shot teach.

## Lesson for the next designer
If you're shipping the most-thorough scheme set in v5, your "Lessons applied from earlier reviews" section is your single best persuasion tool — cite *five* prior reviews by ID with the specific implementing element and the critic has nothing to ask for that you haven't already addressed. This design did so and the result is the highest synthesis density in v5 so far.
