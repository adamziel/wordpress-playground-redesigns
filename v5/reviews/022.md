# Review of v5 #022 — Chip windows

**Parent direction:** "Chip windows. Top bar carries 5 small 'window' chips (one per section). Click chip → that section opens as a small floating, draggable card over the iframe. Multiple cards allowed but rare."
**Affordance count at idle:** 8 (README-counted): URL pill (with refresh integrated) · Save name-pill · Sites button · 5 chips (Settings · Files · Blueprint · DB · Logs). Logo and fullscreen are disclaimed as decorative/hover-only. At cap exactly.

**Scores (out of 10):**
- Feature completeness: 10 — 11 scenes covering idle, Settings open, multi-window (Files + Logs), boot with Blueprint minimized + Logs floating, fullscreen, Sites launcher, full 43-blueprint gallery, Blueprint + Database windows, Apply&Reset confirm, mobile 3-states. Comprehensive coverage of every requirement.
- Restraint: 9 — 8 affordances counted; the chip metaphor *is* the chrome's structure, so the 5 chips are not separate doors but rather *status-indicators-that-double-as-buttons*. Inventive use of the affordance count.
- Iframe primacy: 10 — iframe fills 100% of workspace; chip-windows are deliberately *small* (420–560 px wide) and float *over* the iframe rather than reflow it. Fullscreen with F/Esc.
- One-door: 8 — README admits soft duplicate: "Save appearing on the name-pill *and* the info banner — but the banner only appears for Unsaved Playgrounds, so they're temporally disjoint." Reasonable defense. Settings is genuinely one door (Settings window holds popover + sidebar tab + info notice + WP Admin + Export).
- One-secondary: 8 — README admits 4/5: "one surface *type* — the chip-window — that swaps content per chip; not a single shared panel but a single shared component; rare multi-window is the trade-off." Multi-window allowed = potentially multiple secondary surfaces. README defends as policy not constraint.
- Direction adherence: 10 — chip ↔ window equivalence is the whole concept. Traffic-light dots on each chip, dashed tether back to chip, dashed-minimized state — the metaphor is mechanically faithful.
- Visual fidelity: 9 — Inter font, real Lucide icons, traffic-light styling (red close / yellow minimize / dots), dashed tether lines visualizing the chip-window relationship, real WordPress iframe content, mobile chip strip rotated below.
- Synthesis with v1 lesson: 10 — explicitly cites #64 (DevTools), #15 (Win11 fullscreen), #55 (iPad multitasking) and the *novel* synthesis is making "the chrome's status indicators and the windows literally the same object — a chip *is* a minimized window." That's a genuine IA insight.

## What works (specific)
- **Chip-as-minimized-window equivalence** is the design's most original idea. The chip shows status (closed/open/minimized via traffic-light dot color + outline style); clicking it opens or restores the window. Status and action collapsed into one element — the same trick #014's name-pill uses, but applied to *five* destinations.
- **Dashed tether from window back to chip** is the connective tissue that makes the metaphor legible: when a Files window floats in the middle of the workspace, the dashed line back to the "Files" chip in the top bar tells you "this came from there." Visual provenance.
- **`⌘1..⌘5` keyboard map** for the five chips (mentioned in legend) gives muscle-memory parity with #007's number-key spokes.
- **8 px dot per chip rather than full-pill background** keeps the toolbar reading as monochrome at idle. Color only fills in when *open*. This is *exactly* the restraint v5 §2 rule 2 demands.
- **Multi-window policy as social pressure**, not hard constraint: small card sizes + anchored tether + "chip becomes open state" all discourage clutter without forbidding it. Honest IA design.
- **Refresh integrated into the URL pill** (left icon) — one less toolbar slot. Refresh doesn't deserve its own affordance.
- README admits the chip-as-button leap-of-faith: "the dot + name reads as a status indicator, not a button, until you click one. I mitigate with the traffic-light styling on hover and the `⌘1..⌘5` keyboard map shown in the legend." Pre-emptive critic-grade self-disclosure.

## What's weak or missing (specific)
- **Multi-window is the design's biggest IA bet.** README admits "a power user could open all five at once and clutter the iframe." If the design defends one-secondary-surface as a *policy* not a *constraint*, then any v5 critic following the rubric literally must dock the score. README does (-1 honestly).
- **The chip's status indicator (traffic light dot) is small** — 8 px. At a glance, all 5 chips look similar; the *which-is-open* distinction relies on a subtle outline change (dashed vs solid background). Increase the dot size to 10–12 px or use *color* for open state (current chip turns blue) instead of relying on outline style.
- **Sites button is separate from the 5 chips** — but Sites is also a *destination* with its own window (scene 6). Why isn't Sites a 6th chip? Consistency would help; the README has Sites as a "Sites button" (different visual treatment from the chips). Either make them all chips or make the chips visually distinct from "Sites" as a navigation primitive (which is what they kind of are — chips for the *current* playground's surfaces, Sites for *switching* playgrounds).
- **Fullscreen via F key only**, with a corner pill that "appears when the iframe is the foreground" (README). Not visible at idle. Same critique as #013 — the brief asks for "one-key/one-click toggle." Add a small fullscreen affordance to the URL pill (next to refresh) for the click path.
- **Mobile uses a "chip strip rotated below"** — but on a 390 px phone, 5 chips × 64 px = 320 px of horizontal real estate, leaving 70 px for the URL pill. Either the chips need to shrink to icon-only or the URL pill needs to be on a different row. README rates -1 on mobile honestly.
- The **"Quick actions row" inside the Settings window** holds Export PR + Download .zip + WP Admin + Homepage. That's a sub-popover inside a window. Defensible but it nests once: chip → window → action-row. Compare to #017 where these are in the overflow `⋯` directly.

## How to push harder (1–2 pointed suggestions)
1. **Make multi-window an exclusive-open IA, not a social-pressure policy.** Opening Files closes Settings; opening Logs closes Files. This eliminates the one-secondary-surface trade and converts the "trade-off" into a strength. Power users who *really* want side-by-side can pop the window to a separate browser window (since these are floating cards, that's a 1-click "detach"). The dashed-tether returns when the popped window's chip is clicked.
2. **Promote Sites to a 6th chip** for consistency, with its own traffic-light dot and `⌘0` shortcut. The chip row becomes the *full* IA primitive: 6 chips, 6 windows, no special "Sites button" exception. Then the chip metaphor reads as the *only* navigation method, not "chips + the Sites button."

## Lesson for the next designer
If your direction allows multiple secondary surfaces at once (windows, floating cards), make exclusivity the *IA default* — not a social-pressure trade-off. Power users get an explicit "detach to new browser window" escape hatch, and the v5 one-secondary-surface rule is met without compromise.
