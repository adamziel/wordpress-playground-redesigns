# 022 — Chip windows

## Thesis
The Site Manager is dissolved into the top toolbar as five small "window-chips" — each chip is a minimized macOS-style window for one of the five sections (Settings, Files, Blueprint, Database, Logs). Click a chip and that section materializes as a small floating, draggable card over the iframe, with a dashed tether back to its chip so you always know where it came from. Close it (red traffic light) and it returns to the chip. Minimize it (yellow) and the chip switches to a dashed-outline "stowed" state. The iframe is *always* the primary surface; the cards are deliberately small (≈ 420–560 px wide) and rarely needed in pairs. This refines v1 #64 (DevTools), #15 (Win11), and #55 (iPad multitasking) by making the chrome's status indicators and the windows literally the same object — a chip *is* a minimized window.

## What I merged / removed and why
The three settings entry points (popover, sidebar tab, info notice) collapse into the **Settings chip-window** — the only door. The info-notice "Save site locally" still appears, but only inside that one window. The "Additional actions" overflow menu (Export to GitHub PR, Download .zip, WP Admin, Homepage) also lives inside the Settings window's "Quick actions" row — one door per feature. The launcher (currently a separate right-side drawer) becomes its own kind of window summoned from the **Sites button**, with three tabs: Start new (the six sources), Your playgrounds (4 saved), and Blueprints (the full 43-card gallery with search + 10 category filters). Refresh is integrated *into* the URL pill (left icon) — it doesn't deserve a separate toolbar slot. Fullscreen is invoked by the `F` key with a discoverability hint that appears as a corner pill when the iframe is the foreground; in fullscreen, the only chrome is a single "Esc to exit" pill. Destructive Apply &amp; Reset is upgraded from a red banner to a typed-phrase modal (you must type the playground name) — the brief explicitly demands this.

## Trade-offs
"Multiple cards allowed but rare" is more of a *policy* than a constraint — a power user could open all five at once and clutter the iframe. I lean on social pressure (small card sizes, anchored dashed tether, the chip going to "open" state) rather than hard-blocking concurrent windows; the alternative (exclusive-open) felt anti-DevTools. The chip-as-minimized-window metaphor requires a small visual leap on first use: the dot + name reads as a status indicator, not a button, until you click one. I mitigate with the traffic-light styling on hover and the `⌘1..⌘5` keyboard map shown in the legend. Five colored chips edge close to "too much chrome color"; I quantified by using a single 8 px dot per chip (not a full pill background), so the toolbar still reads as monochrome at a glance — the chip background only fills in when *open*. Glass is used on three surfaces only: top bar (ambient WordPress-theme tint), floating windows (so the iframe behind is hinted-through at the edges), and the mobile chip-row — every glass instance is doing depth-hierarchy work, not decoration.
