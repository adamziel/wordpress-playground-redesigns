# Review of #052 — Branching tree (org-chart)

**Direction:** "Branching tree (org-chart): same idea but horizontal — your playground history is a tree you can fork."

**Scores:** Feature completeness 9/10 · Direction 10/10 · IA 9/10 · Visual 9/10 · Originality 9/10

## What works
- Direction is delivered with a real horizontal org-chart of nodes: Root nodes for Gutenberg PR #58423, WordPress PR #6512, etc. Branches forking from each root, "Unsaved scratch" as a dashed-border unsaved node. The tree-as-IA is the central interaction.
- **The destructive Apply+Reset reframe is brilliant and explicitly delivered**:
  - "**Settings apply by branching.** Changing WP or PHP version creates a new node — your current node is preserved. Use **Apply & Reset** only for a clean wipe of this node." (line 1980)
  - Two distinct buttons: `<button class="btn primary">Apply as new branch</button>` (line 2029) and `<button class="btn danger">Apply & reset this node</button>` (line 2030)
  - Below them: "**Reset is destructive.** It wipes this node's filesystem and database in place. Prefer *Apply as new branch* to keep the current state." (line 2035)
  
  This is the **safe-default + destructive-option** pattern done right. The non-destructive path is the primary button; the destructive path is explicitly secondary. Best-in-class Apply UX.
- All 6 §2.2 sources rendered in the Spawn rail (lines 1548–1574): Vanilla WordPress / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip — each with tooltip. Also in the gallery popover (lines 2286–2321).
- Address bar `<input value="/wp-admin/edit.php?post_type=post">` (line 1528) — §2.1 item 2.
- Database has implicit early-access; Logs has search input (line 2146); Blueprint pane has Copy link / Download bundle (lines 2078–2079).
- Unsaved nodes visually distinguished with dashed border + warn color in the tree legend (line 1919: "Unsaved" swatch).
- README is honest: "this design leans hard into a power-user mental model — it's optimized for the person testing five Gutenberg PR branches in parallel, not for the visitor who just wants to click Twenty Twenty-Five." Owns the bet.
- The "permanent Stage panel that always shows *something* live, plus an empty-state hint" mitigates the unfamiliar-IA risk for first-touch users.

## What's missing or weak
- **The README admits forking every settings change can produce node clutter; a "merge into parent" / prune affordance exists on each node but isn't fully fleshed out here.** This is a deferred mitigation. Worth flagging.
- **Settings appear in the Inspector dock when selected** AND in a popover (verify whether multiple settings surfaces exist). The README claims "settings are now node properties, applied by branching" — but the artifact has both a Settings popover and the inspector's Settings tab.
- **Mobile collapses to a vertical accordion** — README admits the visual fork-ability is lost. Real cost.
- **The tree's spatial cost on desktop is real** — README admits, but the Stage panel mitigates by keeping the iframe always visible.
- **The PR / GitHub / URL launcher rail-btns** (lines 1556–1574) are tooltip-only triggers; the actual paste-input flow isn't necessarily rendered in the main view. Verify whether they open inline inputs vs. the gallery popover.
- **"Boot a Gutenberg PR" sub-line** at line 2302 — the popover gallery shows sub-lines for each source, but the actual paste field is implicit.
- **The Spawn rail's 6 buttons + the gallery popover** create two surfaces for the same source list — duplication, but justified by "rail = always-visible quick-spawn, gallery = browse 43".
- **Node clutter risk** — every settings-apply forking creates a new branch. The "merge into parent" affordance is deferred. A user testing many WP/PHP combinations would quickly have an unmanageable tree.
- **Originality is high** — overlaps with #051 (Git timeline). Both designs treat history as a navigable structure; #051 is vertical commit graph, #052 is horizontal org-chart. The two diverge in geometry but converge on "history-as-IA". Both are excellent direction interpretations.

## How to push harder
- Render the "merge into parent" / prune affordance the README acknowledges is needed. Without it, the tree becomes hostile to repeat power-users — the metaphor's central liability is undelivered.
- Make the "Apply as new branch" vs "Apply & reset" choice the explicit destructive UX everywhere — replace any other Apply&Reset button in the design (e.g., the popover) with the same two-button pattern. Consistency unlocks the safety guarantee.
