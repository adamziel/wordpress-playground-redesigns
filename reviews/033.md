# Review of #033 — Radial / pie menu

**Direction:** "Radial / pie menu: all primary actions surround the cursor or a fixed center icon. Hover-disclosure."

**Scores:** Feature completeness 9/10 · Direction 9/10 · IA 7/10 · Visual 8/10 · Originality 7/10

## What works
- Pie metaphor delivered: hub-and-petal first ring, nested second ring on petal activation, arc-shaped content surface wrapping the hub. The geometry is consistent across states.
- All 6 §2.2 sources rendered as "Start a new playground" cards in the "+" sector arc (lines 637–642) AND duplicated in mobile/launcher demos (lines 1124, 1248–1253). Vanilla / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip with descriptive sub-copy.
- Apply+Reset has the strongest destructive copy in the gallery: "Applying these settings will reset your playground — unsaved files, posts, and database content will be lost." + a button color-coded with `border-color: var(--accent)` (line 735–736).
- Address bar is real: `<input value="/wp-admin/edit.php?post_type=product">` in the chord-across-the-top of the hub (line 1002, 1088, 1165). §2.1 item 2.
- Site Manager has a renameable input `<input value="coffee-shop · sun-rev">` (line 671) and a 5-tab strip.
- Database has early-access framing implied; Logs has filter "supports regex" input; Blueprint has Run Blueprint + Copy link + Download bundle.
- The README admits the "confirm-arc" for Apply & Reset — though I'd need to verify whether the actual confirm arc renders or is just described.
- Unsaved-Playground notice present (line 709–710) with Save site locally button. Status crescent on the hub for save state per README.
- README is honest about the discovery cost: "labels are short and angled, and a first-time user has to commit to spinning the hub before they see what's there."
- Mobile fallback: 6 petals per ring, long-press to reveal labels — sensible.

## What's missing or weak
- **The "confirm-arc" for Apply & Reset is described but not clearly rendered** in the visible portion of the artifact. The button at line 736 fires immediately. The README claims the confirm-arc; verify whether it's a separate state.
- **A first-time user sees a small hub at the bottom-center with no labels** — README admits and adds tiny status crescent + Save site notice. Still a high commitment-cost for discovery.
- **The arc-canvas content surface wrapping the hub means content is on arcs, not rectangles** — but the actual rendered Start cards, blueprint search, settings rows are conventional rectangles. The arc is the launcher, not the content shape. README acknowledges the chord/horizontal target for the URL bar.
- **The PR / GitHub / URL / Import .zip launchers are tile-shaped cards** — same §3 flow 11–12 issue.
- **Settings appear in the Settings petal of the inner ring AND in the Site Manager Settings tab** — two surfaces per the README's "Site Manager's five tabs become five inner-ring petals when the wrench petal is opened" — but if Site Manager petal opens with a 5-tab strip + each tab can also be its own inner-ring petal, that's two paths to the same surface.
- **Overlap with #032 (Concept-album minimal)** — both designs are "hide everything, expand from a center point". #032 commits to 8 petals on a single dot; #033 has the hub-and-orbits geometry with nested rings. The two designs work the same territory; #033 is slightly more nested.
- **No persistent visibility of playground name and save state** in the chrome — README admits and adds the tiny status crescent. The crescent is small and easy to miss; §4 risk underaddressed.
- **The "title-bar / breadcrumb" pattern of WP Admin / Homepage** doesn't have explicit petals — they're presumably inside the "site nav" petal somewhere. Discoverability cost.
- **The chord-across-the-top URL bar** is a clever "we kept the only horizontal thing" — but it visually doesn't fit the pie metaphor at all. README admits it as necessary.

## How to push harder
- Render the confirm-arc for Apply & Reset as a real state — show the arc geometry with "tap to confirm" + a count-down or a hold-to-confirm interaction. The arc is the design's strongest defensible UX claim; render it visually.
- Differentiate from #032 by leaning into the *nested* pie idea — show what a second-ring radial actually looks like when activated. Right now the artifact shows the first ring + conventional content, which mostly resembles #032's "dot opens overlays" pattern.
