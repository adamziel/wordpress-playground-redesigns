# Review of #010 — Figma-like

**Direction:** "Figma-like: zoomable infinite canvas. Each saved playground is a frame at a fixed position. Properties panel on the right for settings of the focused playground."

**Scores:** Feature completeness 8/10 · Direction 9/10 · IA 7/10 · Visual 9/10 · Originality 9/10

## What works
- Strong commitment to Figma's three-zone shell — left layers/assets panel, center canvas with multiple frames at varied positions (Coffee Co. live, My Portfolio saved, Gutenberg PR booting with progress fill, WP 6.7 RC, Friends Plugin Sandbox), right Properties inspector with Properties / Files / Blueprint / Database / Logs subtabs.
- Address bar is a real `<input value="wp-admin/edit.php?post_type=product">` with chevron-left/right history buttons; the playground name pill is `editable` — §2.1 items 2–3 satisfied.
- "+ New" toolbar dropdown carries all six §2.2 source entries (Vanilla / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip) plus "Browse all 43 blueprints" as a separator entry. Direct mapping.
- The Properties inspector's "Apply settings" section has both an inline body ("Apply settings reboots PHP and wipes the running site. Save first to keep your work.") AND an Unsaved-Playground sub-panel with a "Save site locally" button — both warnings co-located. Good.
- Frame coordinates (X 80, Y 80, W 880, H 540) shown as Figma-style inputs — leaning into the metaphor without violating §5 (the README acknowledges this implies storing x/y per saved site, which is a behavior change but not a new feature).
- DB pane has the early-access notice, search input, and a real query bar. Logs pane has search + level filter chips + 9 real log lines. Blueprint pane has Run + Copy link + Download bundle equivalents.
- Mobile section described but the file mentions canvas → single-frame full-screen + bottom dock with Layers / Canvas / Inspector / + — though only mentioned in README, the desktop story is rich.
- "Booting" state rendered as a frame skeleton with progress fill (no spinner overlay) — exactly the README's "intentional" design for canvas-world loading.

## What's missing or weak
- **Settings appear in three places.** A quick popover in the Settings button (line 1330–1341), the Properties subtab of the inspector, AND inline below the Properties row (line 1786–1813). The README claims they collapse into the Inspector; they didn't. §4.
- **Apply & Reset has no confirm step.** The popover's `Apply & reset` and the inspector's `Apply & reset` both fire immediately. The "Save first" copy is good, but a confirmation step would close §4.
- **The "Site Manager" button in the top toolbar (`<button class="tool-btn" title="Site Manager"><i data-lucide="panel-right">`) does nothing visible** — it's a panel-right icon, but the right panel is always open. What does it toggle?
- **The Blueprints gallery as "Assets" panel is described as drag-and-drop**, but the artifact only shows the search input and a list (line 1434+). Drag affordance isn't demonstrated — would benefit from a draggable cursor state on hover.
- **The `+ New` dropdown's PR / GitHub / URL options are tile rows that don't show an input field**. Tapping "WordPress PR" should reveal a number paste field; the artifact shows only the dropdown row. §3 flows 11–12 incomplete.
- **No visible minimap or "Fit all" button** despite the README claiming them as discoverability aids for the infinite canvas. Critical when the README admits "new users may need a moment to realize they can pan/zoom".
- **The "12" badge on the Saved Playgrounds tool-btn** suggests 12 saved playgrounds, but the artifact only shows 5 frames on the canvas + a Layers list of 5. Where are the other 7?
- **The Unsaved-dot indicator (`<span class="dot-unsaved">`) is rendered in the play-name pill at the top, but is a small dot** — same critique as Spatial #008 and Mobile #006: the §4 risk demands a louder treatment, especially in a canvas world where focus drifts to spatial navigation.
- **The "frame skeleton" booting state has the iframe replaced with a progress fill** but the address bar still says `wp-admin/edit.php?post_type=product` in the toolbar — the toolbar address belongs to the focused frame, not the booting one. State inconsistency: top toolbar is locked to one frame even when another is focused.
- **The blueprint frame's "additional actions" kebab inside the frame title-bar is described as a duplicate of the inspector header kebab** — explicit duplication that the README justifies by "both surfaces are useful". Worth noting it's still duplication.

## How to push harder
- Render the minimap + "Fit all" button. They're the difference between an infinite-canvas concept that ships and one that confuses. With 12+ saved playgrounds, panning by mouse alone is hostile.
- Use the Figma metaphor for the destructive Apply step: render a "modal confirm" that mimics Figma's "are you sure you want to detach this component?" overlay — a real, Figma-styled confirm dialog with the blueprint's blast radius listed. That's the moment your direction wins.
