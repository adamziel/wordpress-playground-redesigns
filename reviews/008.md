# Review of #008 — Apple Vision / Spatial

**Direction:** "Apple Vision / spatial: floating glass panels in 3D space (using CSS transforms / perspective). The iframe is a 'window'. Settings hover as cards beside it."

**Scores:** Feature completeness 8/10 · Direction 8/10 · IA 5/10 · Visual 8/10 · Originality 7/10

## What works
- The WindowBar capsule attached to the iframe genuinely commits to the visionOS metaphor — refresh, address bar with `/wp-admin/edit.php`, saved/unsaved dot, Save pill, launcher/site-manager/settings triggers all present (§2.1 items 1–7).
- Side ornament rail riffs on visionOS window ornaments and acts as a second route to Settings/Files/Blueprint/Database/Logs (§2.3 tabs 1–5), with WP Admin and Homepage jump buttons (§2.3 header) and an "Additional actions" overflow with both "Export to GitHub PR" and "Download .zip" text spelled out.
- Constellation blueprint slab carries a real search field, all 10 category filters (All / Featured / Website / Personal / Content / Themes / Gutenberg / Experiments / WooCommerce / News — matching §2.2), and 24 cards generated from a data array (above the 12 minimum).
- Logs strip has level pills (All/Info/Warn/Error/Debug) and a search box — directly addresses the "Logs tab is unstyled text; no level filter" pain point.
- Three states are addressable via the top mode switcher (Booting / Working / Blueprint Gallery), and there is genuine mobile fallback with a 5-tab bottom bar.
- `prefers-reduced-motion` degrades the parallax and z-translated slabs — a thoughtful detail most designs will skip.

## What's missing or weak
- **No visible affordance for "Save site locally" → renaming the title.** §2.4 user flow 4 says the label changes from "Unsaved Playground" to the saved name. The WindowBar shows "Unsaved Playground" with a save dot, but there is no visible state where a saved playground shows its name in the WindowBar — would help validate the flow.
- **Settings duplication still present, just rebranded.** The README claims "the three settings entry points go away" but in fact: the standalone Settings slab on the front-right AND the Settings tab inside the Site Manager slab BOTH render five identical rows (WP / PHP / Language / Network / Multisite / Apply+Reset). That is exactly the "Settings appears in three places" pain point from §4 wearing different glass.
- **Apply & Reset destructive warning is shallow.** The big slab Apply button has only a tiny "Resetting wipes the current playground" note in `--ink-3` (low-contrast). The Site Manager copy of the same button has NO warning text at all — just "Apply Settings & Reset Playground" on a red button. There is no confirmation step, no inline diff of what will be lost. §4 explicitly flags this.
- **Blueprint gallery only has 24 cards, not 43.** The brief (§2.2) and §5 ("the full list of 43 if reasonable, or at least 12") leave this above the floor, but a designer leaning into a "constellation" should grid the whole sky. The BLUEPRINTS array is half-finished.
- **Site Manager title editing is unclear.** The `sm-title-edit` is a plain text input with no save affordance, no debounced "saved" indicator. How does the user know the rename was persisted vs. discarded?
- **Booting state is anemic.** In `boot` mode the iframe is hidden behind a spinner card, but the surrounding glass slabs (settings, launcher, ornament) remain fully interactive at full opacity. A real spatial OS would dim everything that depends on the booting site.
- **Ornament has WP Admin and Homepage**, but the WindowBar address bar is the only place the URL is shown. There's no visible breadcrumb/back/forward despite the visionOS metaphor inviting it.
- **Mobile sheet duplicates "Site Manager" as its own row with controls inside**, but the bottom tab bar has 5 tabs (Site/Blueprints/Files/Logs/More) — Database is not on the tab bar and the user must hunt for it under "More".
- **The "WordPress PR" and "Gutenberg PR" launcher entries are buttons but don't show how a user inputs a PR number.** Same for "From GitHub" and "Blueprint URL". §3 flow 11–12 require this affordance — even a stubbed input would land it.

## How to push harder
- Collapse the two settings duplicates into a single slab that the Site Manager points at — actually deliver the README's promise. Make Apply+Reset a two-step interaction: tap once → slab flips to reveal a confirmation pane listing what's about to be wiped, then a second tap commits.
- Push the spatial metaphor further on launch dialogs: "WordPress PR" should expand to a card with a PR-number field hovering nearer the camera; "Blueprint URL" should reveal a paste field. Right now those launcher tiles look like terminal buttons.
