# Review of #056 — Stage Manager

**Direction:** "Stage Manager: floating windows organized into 'spaces' on the left rail."

**Scores:** Feature completeness 9/10 · Direction 10/10 · IA 8/10 · Visual 9/10 · Originality 9/10

## What works
- macOS Stage Manager metaphor delivered with rigor: tilted thumbnail stacks of other Spaces on the left rail, floating co-equal windows on the stage (iframe + Settings + Files + Blueprint + DB + Logs), Launchpad as a special pinned Space accessed from the rail. Switching spaces slides windows on/off.
- **The §4 "lose-on-refresh" risk is brilliantly addressed via the yellow ribbon on Space tiles** (line 1076: `<div class="space-ribbon">Unsaved</div>`). README: "you can *see* across all your spaces which ones aren't persisted." Save is one click on the ribbon itself. This is one of the strongest unsaved-state visualizations in the gallery — it's persistent + visible at the rail level even when the user is focused on another space.
- All 6 §2.2 sources rendered as Launchpad rows (lines 1351–1367 visible): Vanilla WordPress / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL — each with sub-line.
- Apply+Reset has explicit destructive copy: "Applying resets the site." + `<button class="apply">Apply & Reset Playground</button>` (line 1019, 1229). Two surfaces (popover + window).
- Address bar `<input class="addr-url" value="/" placeholder="/wp-admin">` (line 979) — §2.1 item 2 with proper placeholder.
- Database has implicit early-access; Logs has filter input "Filter PHP error log…" (line 1281); Blueprint has Copy link button (line 1330).
- Unsaved-Playground notice: "This is an Unsaved Playground. Changes will be lost on refresh." + Save site locally button (lines 1224–1225).
- Booting Gutenberg PR #62318 state with progress steps (lines 1178–1183) — booting state demonstrated.
- README is brutally honest about size: "at 1440px the iframe shrinks to roughly 70% width to leave room for the rail and the floating window edges."
- Mobile abandons multi-window metaphor entirely — sensible.

## What's missing or weak
- **Apply & Reset has no confirm step.** The popover and the window both have `<button class="apply">` that fires immediately. The Stage Manager's metaphor invites a "Are you sure? This space's windows will be wiped" overlay; not delivered.
- **Settings appear in two surfaces** — README says "same window, two presentation modes" — a floating Settings window AND a quick popover from the top bar. Two visible surfaces.
- **The metaphor assumes users grasp Spaces/Stages** — README admits first-touch users see only one Space until they create more. Learning curve real.
- **Screen real estate cost is large** — README admits 70% iframe width.
- **The PR / GitHub / URL launcher rows in Launchpad** have sub-lines without inline paste inputs.
- **Tilt/drop-shadow aesthetic reads less playful than the current Playground** — README admits but defends with "your sandbox is a physical desk" framing.
- **Floating windows draggable + remember-position-per-space** is the strongest interaction promise — but the static artifact can't fully demonstrate dragging.
- **Originality is high** — Stage Manager is a niche macOS reference; the gallery doesn't have another design committed to spatially-arranged co-equal windows. The "rail of spaces" element overlaps slightly with #004 (Finder windows) but #056 leans into Stage Manager's specific window-arrangement-per-space idiom.
- **The "Launchpad Space" as a tile in the left rail** is a clever recursive metaphor — but a first-time user looking for "how do I start a new playground?" needs to find the + at the top of the rail, which is less obvious than a big "New" button.

## How to push harder
- Render the destructive Apply & Reset as a Stage Manager-style overlay — the current space's windows shake or dim, with a center modal "Reset this space? You'll lose all 4 windows' state." Use the metaphor's strongest visual language for the destructive moment.
- Add a "Save this space" master button next to the unsaved ribbon — currently the ribbon is the warning, but the recovery requires opening the Settings window or finding Save elsewhere. The ribbon's click should commit the save inline.
