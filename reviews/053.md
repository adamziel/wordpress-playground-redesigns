# Review of #053 — Tabs-only

**Direction:** "Tabs-only: every panel is a tab on the SAME tab strip. No sidebars. (Settings tab, Files tab, Blueprint tab, Logs tab, DB tab.)"

**Scores:** Feature completeness 9/10 · Direction 10/10 · IA 8/10 · Visual 8/10 · Originality 7/10

## What works
- Direction commitment is total: a single horizontal tab strip above the iframe (Site / Settings / Files / Blueprint / Database / Logs + dynamic Launcher / Gallery tabs). No sidebars, no drawers, no popovers. Browser-as-Playground recursive metaphor.
- All 6 §2.2 sources in the Launcher tab (lines 856–861): Vanilla WordPress / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip — each with sub-line.
- Apply+Reset has the cleanest dual-button pattern in the gallery (mirroring #052's safe-default approach):
  - "Apply non-destructive" primary button (line 1033) — for settings that don't require reset
  - "Apply settings & reset Playground" with `btn-danger` (lines 1081–1084) — explicit destructive option
  This is the second-best Apply+Reset UX in the gallery (after #052).
- Address bar `<input value="/">` (line 747) and a separate full URL bar `<input value="/?step=woocommerce">` (line 804) — §2.1 item 2.
- Database tab has the early-access notice (line 1302). Logs has filter input "Filter logs — try 'fatal' or a plugin slug…" + Download button (lines 1385, 1391). Blueprint has Run Blueprint / Copy link / Download bundle (lines 1252–1260).
- Unsaved-Playground pill prominent in chrome (lines 713, 1011, 1430). Save site locally button (line 1043).
- "Site Manager toggle is now a no-op because the tabs ARE the site manager" — README admits the toggle is preserved per brief but disabled, which is honest.
- Per-tab chrome: Site tab has the address bar + Refresh; other tabs have their own search/filter inputs. Browser-tab-style separation.
- README is honest about tab-strip crowding: "if you open Launcher + Gallery + Files + Blueprint + Logs you've burned half your horizontal pixels. I mitigate this with icon-only tabs at narrow widths and an overflow chevron." Real cost named + mitigated.

## What's missing or weak
- **Settings appear in the Settings tab only** — README claims "The Edit Settings popover is gone — there's already a Settings tab, three doors to the same room was the original sin." This is one of the few designs that actually delivers single-Settings-surface. Verify no popover surfaces in the artifact.
- **Tab strip crowding is the metaphor's central liability** — 6th-or-7th-tab users feel it. Icon-only collapse + overflow chevron mitigate but don't eliminate.
- **Mobile collapse to horizontal-scrolling chip rail** — admitted loss of spatial parallelism. Tab strips translate to mobile but lose their visual hierarchy.
- **Killing the right-side launcher drawer** trades glanceability (see iframe + saved sites at once) for focus (one panel at a time, fullscreen) — README's explicit trade-off.
- **Power users juggling a blueprint edit and the live preview will miss the split** — README admits. The brief assumes the iframe stays visible; this design hides it behind tab switches.
- **The PR / GitHub / URL launcher starter tiles** — verify whether they expose inline paste inputs. The README sub-lines ("Boot from a core PR number") hint at deeper input flows undelivered in the artifact.
- **Overlap with #027 (Single-page scroll)** in IA philosophy: both reject sidebars and drawers, both put everything in one navigation primitive (sections for #027, tabs for #053).
- **Originality is medium** — the "tabs-only" direction is itself a constraint, and #053 implements it cleanly without inventing additional moves.
- **"Apply non-destructive" + "Apply & reset"** is a great split — but it needs explicit copy about which settings trigger which. Switching PHP version requires reset; toggling Network might not. The artifact doesn't clarify which settings are non-destructive.

## How to push harder
- Render the "which settings are non-destructive vs. destructive" distinction inline. Each setting row should be tagged "(safe, no reset)" or "(destructive, reset required)" so users see the impact before committing.
- Demonstrate the tab overflow chevron in action — open 8 tabs, show how the strip collapses. The crowding mitigation is the design's main scalability claim; render it.
