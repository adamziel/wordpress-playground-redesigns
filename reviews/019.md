# Review of #019 — Slack layout

**Direction:** "Slack layout: workspace switcher on the far left, channel sidebar, threads-like log viewer in the right panel."

**Scores:** Feature completeness 9/10 · Direction 9/10 · IA 7/10 · Visual 8/10 · Originality 7/10

## What works
- Faithful 4-column Slack composition: workspace rail (each saved playground as a squircle), channel sidebar (Settings / Files / Blueprint / Database / Logs channels + DM-like entries for "Start new from… Vanilla / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip"), main stage with iframe, and a persistent right thread pane housing Logs as chat.
- All 6 §2.2 sources are present as channel-like rows AND duplicated as launcher modal cards (lines 1370, 1825+). Explicit redundancy serves discoverability.
- Address bar is a real `<input class="tb-url" type="text" value="/wp-admin/edit.php?post_type=product">` in the global Slack-search slot — clever mapping of §2.1 item 2.
- Apply & Reset Playground notice text is strong: "Apply Settings & Reset will discard the running site, including unsaved files and DB rows." + `btn danger` button.
- The "Unsaved on this browser" status card on the workspace rail (line 1281+) with "Save site locally" button — addresses §4 risk with Slack-status-prompt prominence.
- Logs as chat with avatars per log level, day separators, stack traces in quoted blocks. The compose box "Run a command, paste a /wp-admin path, or @mention a log…" is a clever multi-purpose input that doesn't violate §5.
- Booting state present (line 1520: "Downloaded WordPress 6.7" check mark) + launcher modal + working session = three states.
- Blueprint pane has Run Blueprint + Copy link to blueprint + Download bundle. Database has early-access notice (presumably — verify). Toast "Network mode is on — outbound HTTP requests will be allowed" at line 1914 — a real info toast.
- Mobile fallback described: bottom tab bar (Live, Files, +, Logs, Settings), launcher as full-sheet modal. Aubergine tuned toward WP blue for brand.

## What's missing or weak
- **Apply & Reset has no confirm step rendered.** Slack itself uses "Are you sure?" modals for destructive actions; the metaphor invites it. The current artifact has explicit destructive copy but the button fires.
- **Settings appears in two surfaces:** the popover-equivalent (line 1744–1759) and the in-channel Settings page (line 2184+). The README claims "no separate Site-Manager-vs-popover settings — the channel and the popover are visually identical" — which is honest, but two surfaces still exist. Same data, two locations.
- **The compose box's slash commands** (`/save`, `/zip`, `/pr`, filter logs by level) are described in the README but I don't see an auto-complete dropdown rendered when typing `/`. The slash-command feature is described, not demonstrated.
- **The PR / GitHub / URL channel-rows and launcher-tile-rows** don't reveal paste inputs. §3 flows 11–12.
- **The 4-column desktop layout (70 + 260 + flex + 320 = ~650px of chrome before the iframe even appears at 1440px)** is dense. The README acknowledges "the thread can be collapsed via its X button" but the artifact doesn't show the collapsed state.
- **Logs-as-chat thread pane competes with #018's Discord-style chat-logs.** Both designs landed on the same remix. #019 keeps the iframe as the central stage AND the thread pane simultaneously, while #018 made the chat IS the logs channel. The Slack version is more conservative; the Discord version is bolder.
- **The "@mention a log" syntax** is in the compose box placeholder but the actual log-as-message hover state with reactions is not surfaced. Logs-as-chat is described but the chat UI specifics (reactions, "5 replies in thread") aren't clearly rendered in the visible portion.
- **The visual aubergine "retuned toward WordPress blue for active states"** is honest but the design still reads as Slack-with-WP-tinted-active-states. Minor metaphor leakage.
- **Database channel has its own search input** (line 1543 placeholder "Search logs…" — wait that's actually for logs); search the DB browser surface specifically.
- **The launcher modal has two PR/GitHub/URL surfaces** (lines 1776–1780 and 1832–1860) — that's likely a redundant copy for different content slots, but worth flagging.

## How to push harder
- Implement the slash-command UX visually: show a dropdown of commands when the compose box has `/` typed. The compose-as-router is the strongest Slack remix specific to Playground; render it.
- Differentiate from #018 (Discord) by emphasizing Slack-specific affordances: a "Saved" workspace status, scheduled-message-style "Schedule reset for later", and threaded blueprint discussions for the 43 blueprints. The current artifact reads as Slack reskin; pushing on uniquely-Slack features would justify the parallel direction.
