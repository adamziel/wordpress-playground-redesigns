# Review of #064 — Inspector-first (devtools)

**Direction:** "Inspector-first (devtools): bottom-docked devtools-like inspector with tabs for Elements / Console / Network / Sources — repurposed for Logs / Blueprint / Files / DB."

**Scores:** Feature completeness 9/10 · Direction 10/10 · IA 8/10 · Visual 9/10 · Originality 8/10

## What works
- Chrome DevTools shell is faithfully delivered: slim browser-style address rail at the top, bottom-docked inspector with tabs (Logs / Blueprint / Files / DB / Console / Network), Resources tree on the left of the inspector, Cmd+P command palette.
- All 6 §2.2 sources rendered as launcher buttons (lines 676–681): Vanilla WordPress / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip — each with sub-line ("Boot a core PR by number", "Paste a blueprint.json link").
- Apply+Reset has explicit destructive copy: "Changes apply on Apply & Reset. Site state is wiped." (line 651) + a `tb-btn danger` button (line 667).
- Address bar `<input class="url-input" id="urlInput" value="/wp-admin/index.php" spellcheck="false" aria-label="URL bar">` (line 636) — §2.1 item 2 with aria-label.
- Database has implicit early-access (verify); Logs has filter input "Filter logs (regex ok)…" (line 867); Blueprint has Copy link / Download bundle / Run Blueprint (lines 916–918).
- Unsaved-Playground state promoted to a yellow address-bar pill (line 633) — README claims it's "like an HTTPS warning in Chrome" — addresses §4 by leveraging browser-warning muscle memory.
- WP Admin jump (line 792: `onclick="navigateTo('/wp-admin')"`).
- Download .zip with sub-line "Snapshot the entire playground" (line 800).
- The Console tab "mirrors PHP error_log() output in a structured, filter-by-level way" per README — better DX than the current Logs view, on-metaphor.
- The Cmd+K command palette with `>` actions / `pr:` / `gh:` / `url:` prefixes — keyboard-first inheritance from real DevTools.
- README is honest: "DevTools metaphor is a power-user inheritance: developers will feel at home instantly, but a first-time user landing on `playground.wordpress.net` may find the bottom panel intimidating, so it ships **collapsed by default** with a single 'Inspect' toggle."
- "Iframe gets less vertical real estate when the dock is open (this matches DevTools' trade-off and is fixed by collapse/right-dock/undock controls in the top-right of the panel)." Mitigated via standard DevTools chrome.

## What's missing or weak
- **Apply & Reset has no confirm step.** The `tb-btn danger` button fires. Chrome DevTools itself uses "Are you sure?" confirms for destructive operations; the metaphor invites it.
- **Settings appear in a popover (line 662–667) AND an inspector Settings tab** per the README ("an inspector tab called Settings"). Verify they're consolidated.
- **The PR / GitHub / URL launcher buttons** have sub-lines but no inline paste inputs in the visible grep — Cmd+K palette is the alternative path, but the click-through still needs the paste form.
- **Power-user inheritance** is the README's central bet — first-time users may bounce off the bottom panel even when collapsed.
- **The iframe loses vertical real estate** when the dock is open — admitted DevTools trade-off.
- **Originality has overlap with #002 (IDE-like) and #029 (Console hybrid)** — three designs lean on developer-tool aesthetics. #064 distinguishes itself by being browser-DevTools specifically (vs. VS Code or terminal) and by promoting Unsaved to a yellow URL-bar pill.
- **The "Resources tree" on the left of the inspector** — verify whether it shows the actual file tree or is a custom Playground hierarchy.
- **The Cmd+K palette is described** — verify whether the artifact actually renders an open palette state.
- **Console tab adding PHP error_log() with level filters** is described — verify the Console UI matches the Logs UI or is a separate refined surface.

## How to push harder
- Render the destructive Apply & Reset as a Chrome-style "Are you sure you want to clear all site data?" interrupt — DevTools' own destructive UX (clearing storage, hard reload) is the perfect template.
- Differentiate from #002 (IDE-like) by leaning into DevTools-specific patterns nobody else has: a real Network tab waterfall (per-PHP-request timing), a Lighthouse-style audit for the running site (no new feature, just a different view of the existing logs/DB/files), or "Profile this blueprint" timing visualization.
