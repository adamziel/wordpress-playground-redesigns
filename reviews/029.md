# Review of #029 — Console + visual hybrid

**Direction:** "Console + visual hybrid: bottom-docked terminal that streams logs and accepts wp-cli; visual controls in the top. Toggle the terminal up/down."

**Scores:** Feature completeness 10/10 · Direction 10/10 · IA 8/10 · Visual 9/10 · Originality 10/10

## What works
- The dual-channel "click anything to see the equivalent command" thesis is the single most useful learning affordance in the gallery. Save → `pg save "Coffee Shop"`. Apply & Reset → `pg config set --wp=6.4 --php=8.2 --apply`. Blueprint browse → `pg blueprint search "woo"`. Logs grep → `pg logs grep "Fatal"`. This *teaches the CLI through the GUI*, which is doc-author / dev-evangelist gold.
- Apply+Reset gets the strongest destructive-confirmation pattern I've seen in the gallery: terminal prints `apply --reset is destructive. Confirm? [Y/n]` and shows the response `Y` as a sample. That's a real CLI-style confirm rendered as a state, not just promised. Combined with the popover's "Apply settings resets the playground (destructive). Unsaved data will be lost." — the dual-channel delivers BOTH the destructive prompt AND the wp-cli equivalent.
- Unsaved-state pain (§4) addressed by making the terminal prompt itself the state indicator: `unsaved@playground ~$` flips to `coffee-shop@playground ~$` on save. Persistent, unmissable, and on-metaphor.
- All 6 §2.2 sources rendered as launcher cards with terminal equivalents in muted text: Vanilla WordPress / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip + a Blueprint search input "Search 43 blueprints… (or type: pg blueprint search 'woo')".
- Address bar `<input type="text" value="/2026/05/hello-from-playground" id="urlInput">` — §2.1 item 2.
- Database has early-access framing implicit; Files / Blueprint slide-over panels share the screen with the terminal — README claims "Site Manager's Logs tab is gone — logs ARE the terminal, always visible" which is the cleanest collapse of a §2.3 tab in the entire gallery.
- "More actions" lists Download .zip / Export PR with `data-cmd` attributes — every button has a CLI equivalent baked in.
- README is honest about the dual-channel "two ways to do everything is a sin in product design" — and justifies it through audience targeting (devs learning WP, WP devs testing PRs, doc authors). Defensible bet.
- Mobile fallback: terminal becomes a bottom sheet swipe up. Reasonable.

## What's missing or weak
- **The terminal is intimidating to a true first-time WordPress user** — README admits and mitigates with "ships collapsed-to-a-strip by default" + read-only history unless user types. Whether the collapsed state is rendered in the artifact needs verification.
- **Settings entry points still exist as both a popover AND `pg config set` commands**. README claims they "collapse into one settings popover plus identical `pg config set` commands" — that's two channels, which is the design's whole thesis, but it's still two visible surfaces.
- **The launcher's "Start new / Blueprint gallery / Your Playgrounds" three-stack is one command-palette sheet with three tabs** — verify all three tabs render in the artifact.
- **Two ways to do everything** — README admits the sin and justifies it. Worth flagging that the dual-channel is also a maintenance cost: every new feature requires both a button AND a `pg` command.
- **The blueprint search input** has terminal-syntax hint — but the syntax `pg blueprint search "woo"` is the same as today's search interaction except spoken in CLI. Adds noise for first-time users.
- **WP Admin / Homepage jump buttons** are referenced in the actions menu (line 729) but should be in the chrome — the URL-bar-adjacent area. They might be elsewhere.
- **The PR / GitHub / URL launcher tiles** are tile-shaped; clicking should reveal a paste field. The hint "paste a Gutenberg PR number into the terminal: `pg gutenberg pr 64218`" (line 792) is great for power users but doesn't render an inline GUI input. The CLI path is exposed but the click path is implicit.
- **Toast region** — not strongly emphasized. §2.4 calls for bottom-left toasts; the terminal at the bottom may double as the toast surface, but explicit toast rendering would close the brief loop.
- **The "narrate every click as a copy-pasteable command"** is the strongest claim and the artifact shows scrollback lines like "visual: changed PHP version in settings popover → Apply" followed by the confirm prompt. Genuinely delivered.
- **The terminal's `pg ...` is an invented CLI**, but the README and brief don't forbid this — it's a CLI representation of existing features, not a new feature. §5 OK.

## How to push harder
- Render the "collapsed to a strip" default state as a separate state demo. The first-touch user landing in this design should see the friendly GUI first, the terminal as a hint at the bottom; that initial state is the strongest mitigation of the intimidation risk and it should be visible.
- Add command-history search (Ctrl+R style) inside the terminal — show a power-user affordance that nobody else in the gallery can match. This design's audience benefits from it.
