# Review of #003 — Notion-like

**Direction:** "Notion-like: side panel of nested pages ('Sites', 'Blueprints', 'Settings'). Center pane is content. No floating toolbar — actions live in the page."

**Scores:** Feature completeness 8/10 · Direction 9/10 · IA 7/10 · Visual 8/10 · Originality 8/10

## What works
- Strong commitment to the metaphor: workspace switcher, search bar with ⌘K kbd, nested page tree with twirl-down "Coffee Journal > Overview / Settings / Files / Blueprint / Database / Logs" + "Reset test" branch + library pages New Site / Blueprints / Imports. Real Notion-style cover image, emoji page icon, H1 page title.
- The Overview page treats refresh / address / save as inline page-property blocks — so far this is the only design where the address bar (`<input type="text" value="/wp-admin/edit.php">`) is actually an editable input, satisfying §2.1 item 2.
- "Applying changes will reset this playground" is a real warning callout at the TOP of the Settings page with proper explanatory copy ("changing WP version, PHP, language, or multisite wipes the SQLite database and reinstalls WordPress"). One of the best warning surfaces I've seen.
- All 6 §2.2 launchers (Vanilla / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip) appear as source-cards with descriptions like "Boot from a WordPress core pull request — paste a PR number", and they're also represented as tree items under "New site".
- Blueprint gallery has search + view-toggle (Gallery / List / Table) + 10 category chips matching §2.2, with item counts (Featured · 5, Website · 10, Personal · 5, etc.).
- Logs has search field; Settings has both "Apply settings & reset" AND a "Save site locally" button right next to each other.
- "Your Playgrounds" tree section includes the "Unsaved Playground — Not saved to browser storage" entry verbatim from §2.2.

## What's missing or weak
- **Brief violation: invented feature.** The Settings page adds a "Storage" select with options "Browser (OPFS) — persistent / Temporary / Local filesystem (experimental)". The brief (§5) explicitly says "Don't invent new features." Persisting to local filesystem is not a feature in §2. The "Persist to browser storage" toggle is also non-§2 — there's only one save mode (browser OPFS).
- **Apply Settings & Reset is a primary button with no confirm step.** The warning callout is excellent, but the button "Apply settings & reset" fires without a "type the playground name to confirm" or a "save first?" intermediate. The README claims an "inline danger callout"; the artifact only has the callout above, no modal.
- **Three settings entry points are still implicitly present.** The README says "the popover is gone", but the Settings page coexists with: the tree's `Settings` sub-page under each site, plus an "edit playground" tree icon on hover, plus the inline callout's "Apply" verbs. Whether each is the same surface is unclear from the artifact.
- **Multiple "Save site locally" affordances.** Settings page has one; tree has "Unsaved Playground" with status; New Site list has an "Unsaved Playground" pill. None is in the header of the Overview page where a refreshing user would land, defeating the §4 "users lose work on refresh" mitigation.
- **No WP Admin / Homepage jump buttons in the Overview page header.** §2.3 mandates two header buttons. There's a button at the bottom of Settings ("Open /wp-admin") but that lives 5+ scrolls away from the iframe block.
- **The 43-blueprint grid is rendered by JavaScript (`#bpGrid`).** Without JS it's empty — the static HTML shows zero blueprint cards. §5 requires "at least 12" cards visible.
- **Mobile is described as "slide-in tree drawer" + "bottom + sheet"** but the artifact has no explicit `@media (max-width:...)` rendering of that demo. The brief requires 390×844 responsiveness.
- **The Notion convention of overflow `···` for "additional actions"** is present in the tree but the Export-to-GitHub-PR / Download-.zip text is only inside a page menu (`pmi` divs at line 607), not as a first-class action block on the Overview page.
- **Page tree active-state is permanently on "Settings" in one state and "Overview" in another** without showing how the user navigates between sites. The "browser-tab-like top is gone" tradeoff is acknowledged in the README but its replacement (tree click) is undemonstrated for inter-site switching.

## How to push harder
- Remove the Storage select and the "Persist to browser storage" toggle. They're invented features and they conflict with §2.2's single OPFS save model. The remaining settings (WP / PHP / Language / Network / Multisite) plus Apply&Reset is the §2.3 spec.
- Add a confirmation step that takes advantage of the Notion metaphor — e.g., a slash-menu / "type 'reset' to confirm" inline block right above the Apply button. Don't waste the metaphor on the wrong moment.
