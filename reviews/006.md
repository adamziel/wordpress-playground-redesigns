# Review of #006 — Mobile-first

**Direction:** "Mobile-first: design for 390×844 first, then desktop is just a wider mobile. Bottom tab bar with 4 tabs (Site, Files, Blueprint, More). Sheets slide up."

**Scores:** Feature completeness 8/10 · Direction 9/10 · IA 7/10 · Visual 8/10 · Originality 5/10

## What works
- Strict adherence to the direction: every screen is a 390×844 phone, the four-tab bottom bar (Site / Files / Blueprint / More) is unified across all six phone mockups, and the desktop fallback at the bottom is literally the same column with side rails appended — exactly what the prompt asked for.
- Six distinct phone states shown (booting with progress + step checklist, active session with toast and Sites sheet, blueprint gallery with search + 10 chip filters from §2.2, Files tab with editor, Blueprint tab with Run, More tab with Settings + DB + Logs).
- All six "Start a new Playground" entries from §2.2 (Vanilla / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip) appear as labelled tiles in both the Sites sheet AND the left desktop rail.
- Logs has level filter chips (All / Info / Warn / Err) AND a search field — directly addresses §4's "no level filter" pain point. Database has the early-access label (§2.3 tab 4).
- The destructive "Apply & Reset" action is rendered as a dedicated `more-tile danger` with the copy "Destructive · wipes current site" — better-than-baseline warning.
- The Unsaved notice in the More tab quotes §2.3 directly: "Unsaved Playground. Your changes will be lost on page refresh." with a "Save locally" button right there.

## What's missing or weak
- **The direction's biggest interpretive gap is not addressed.** §2.3 mandates five Site Manager tabs (Settings, Files, Blueprint, Database, Logs), but the bottom tab bar has only four (Site, Files, Blueprint, More) and Database + Logs + Settings + Additional Actions are all stuffed into "More". The README acknowledges this collapse but doesn't argue *why* a power user inspecting the DB should hunt under a kitchen-sink "More" tab. This compresses §2.3's five-tab structure into the direction's four-tab constraint without resolving the conflict.
- **No visible input affordance for "WordPress PR" / "Gutenberg PR" / "Blueprint URL" / "From GitHub".** They are tiles that, when tapped, should reveal a number/URL paste field. The current artifact only shows the tile face. §3 flow 11–12 require a paste path.
- **Apply & Reset is a clickable tile with no confirmation sheet shown.** The README claims "dedicated confirm sheet rather than a small red banner" but the actual demonstration never renders that sheet — and the danger tile sits next to the equally-styled "Import .zip" tile. A first-time user could tap by accident.
- **The "saved-dot" / "unsaved-dot" state in the title pill is the only signal of save state.** A 7px circle is not enough for §4's flagged risk ("users can lose a whole session by refreshing without noticing"). Compare to the unmissable amber notice in the More tab — why isn't that promoted to the topbar?
- **The "title pill" doubles as URL bar AND launcher trigger AND save-state indicator.** Tapping it does what? Open the Sites sheet (per phone 2) or edit the URL? Both? The chev icon flips up/down inconsistently across phones.
- **Blueprint gallery sheet only renders ~16 cards.** §5 says "at least 12" so it clears the floor, but a "43 Blueprints" header above 16 tiles will read as scrollable-but-undisclosed-state to a first-time user — show "...and 27 more" or a footer.
- **Side rails on desktop are described as "unhide what would otherwise be a sheet"** but the right rail in the demo shows Logs while the left shows Sites. There's no UI for the user to choose which sheet to pin to which rail, and the rails don't have close/swap controls.
- **Originality is weak.** This is "Linear/Notion mobile + iOS bottom-sheet" — well-executed but doesn't add anything mobile-first designers haven't already shipped. There's no novel reduction of, say, the apply-reset dance or the blueprint search interaction.
- **The "saved" toast for "Stylish Press" sits inside the canvas above the bottom-sheet Sites panel** — it would be obscured in real interaction. The composition stages two overlays at once which is fine for the gallery, but acknowledge it in the README.

## How to push harder
- Add a fifth bottom tab specifically for Database OR re-order so "Settings" is the fourth tab and the kitchen-sink "More" is killed. The current "More" violates the design's own one-thumb thesis: surfacing DB rows requires Tap More → scroll → DB section, which is three movements to inspect a table.
- The destructive action needs a real confirm sheet *rendered in the gallery*, not just promised in the README. Show the sheet animation, the "Save first" safe-default button, and the explicit list of what gets wiped.
