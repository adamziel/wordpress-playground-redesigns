# Review of #051 — Timeline / Git log

**Direction:** "Timeline / Git log: each save creates a node in a vertical commit-graph. Branch a playground from any node. 'Reset Playground' = checkout main."

**Scores:** Feature completeness 9/10 · Direction 10/10 · IA 9/10 · Visual 10/10 · Originality 10/10

## What works
- Best direction interpretation in the gallery, paired with thoughtful execution. Three-pane Git app shell: vertical commit graph (with branch lines, commit messages, hashes), working-tree center with Files/Blueprint/DB/Logs tabs, inspector right. "Initial: Vanilla WordPress 6.5" commit (line 1535), branching, "Apply & commit" buttons replacing "Apply & Reset" (line 1313).
- **The destructive Apply+Reset is reframed as a calm `checkout main` operation.** The README's central thesis ("'Reset Playground' stops being a scary red destroy-button and becomes a calm checkout main") makes the §4 destructive UX risk into a non-event — every change becomes a recoverable commit. Strongest reframe in the gallery.
- All 6 §2.2 sources rendered as "New branch from..." entries (lines 1991–2006): Vanilla WordPress / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip.
- **Every source has a real inline paste input** (lines 2296–2389):
  - WordPress PR: `<input value="https://github.com/WordPress/wordpress-develop/pull/6234">`
  - Gutenberg PR: `<input value="52341">`
  - From GitHub: `<input value="WordPress/twentytwentyfive">` + radio (theme/plugin/wp-content)
  - Blueprint URL: `<input placeholder="https://example.com/my-blueprint.json">`
  - Import .zip: file upload (presumably)
  This is one of the most complete implementations of §3 flows 11–12.
- Address bar `<input value="/wp-admin/edit.php?post_type=page">` (line 1250) — §2.1 item 2.
- WP Admin pill jump button "Jump to /wp-admin" (line 1557).
- The "Apply & commit" button (line 1313) commits the settings change as a labeled chore commit (`chore: bumped to PHP 8.3`) — turning the destructive Apply+Reset into a labelled, recoverable operation.
- Database has implicit early-access; Logs has filter input (line 1766); Blueprint has the conventional buttons.
- README is honest: "Saves aren't actually diff-based content-addressable commits under the hood (it's still OPFS), so the metaphor is a UI fiction; the trade-off is enormous narrative clarity at the cost of being technically loose."
- "Squash trivial commits" toggle promised in README to control graph length for power users — sensible.
- Inline glossary tooltips ("commit = save", "branch = fork", "checkout = switch") for unfamiliar users per README.

## What's missing or weak
- **Apply & commit doesn't actually require a confirm step** — it relies on the "every commit is recoverable" reframe to make destructive actions safe. That's a strong UX claim, but if a user is unfamiliar with the metaphor, they may not realize they can `checkout main` to undo. The README mitigates with glossary tooltips; they should be rendered.
- **Settings appear in a popover (line 1299)** AND in the Settings tab. README claims unification ("The three settings entry points collapse into one") but the popover persists.
- **The git metaphor is opinionated** — README admits users unfamiliar with version control may need the inline glossary tooltips. First-touch user discoverability tax is real.
- **Graph at all times costs horizontal real estate** — README admits, mitigates with mobile collapse to "History" sheet.
- **Mobile collapse not necessarily fully demonstrated** in the artifact.
- **Saves aren't actually content-addressable commits** — the metaphor is a UI fiction, README admits. Technical purists may bristle.
- **Treating every settings-apply as a commit will make the graph long for power users** — README admits, mitigates with "squash trivial commits" toggle. Verify the toggle is rendered.
- **Originality is high** — the "everything is a commit" reframe is novel enough that overlap with other designs is minimal. The Maps/Tube designs share the "spatial navigation" idea; the Git timeline borrows version-control instead.
- **The 43-blueprint gallery is rendered as "Clone a repository" modal with search + category chips** — sensible mapping; verify all 10 §2.2 categories are present.

## How to push harder
- Render the inline glossary tooltips the README promises. Hover-states or first-time-user inline annotations like "commit = save your current state — you can come back to this exact moment later" would close the discoverability gap.
- Demonstrate the `checkout main` UX as a real interaction state — a user clicks a past commit, the inspector shows "Checkout to this state? This will discard 2 uncommitted changes (or, save them as a branch first)" — the destructive moment that the README's reframe is designed for. Make the recoverability *visible*.
