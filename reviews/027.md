# Review of #027 — Single-page scroll

**Direction:** "Single-page scroll: no panels, no modals. Scroll down to reveal settings, blueprints, file browser. Sticky iframe-preview on the right."

**Scores:** Feature completeness 9/10 · Direction 10/10 · IA 8/10 · Visual 8/10 · Originality 7/10

## What works
- Faithful execution of the direction: sticky iframe right (~55% width), long-scroll document left, slim left rail with numbered anchor jumps (1–9 keystrokes per README). The "everything is a section" thesis is delivered consistently.
- Settings genuinely merged into a single Settings section — README claims "no longer a duplicate quick-popover" and the artifact shows the inline Settings panel as the only settings surface I grep, with the Unsaved-Playground notice block right above it (line 1236–1239).
- All 6 §2.2 sources rendered as cards in the "Start a new Playground" section (Vanilla / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip) AND a `<select>` + `<input>` combo at line 1107–1112 that picks the source type AND has a paste URL input — directly addressing §3 flows 11–12 with a concrete input.
- Address bar `<input id="urlInput" value="wp-admin/edit.php?post_type=page" spellcheck="false">` — §2.1 item 2.
- "Apply Settings & Reset Playground" section has the destructive title + the README claims a confirm step ("the inline scroll model means users could trigger it while skimming — it's gated by a confirm step in the section itself"). Whether the confirm UI is rendered or just the button needs verification but the README acknowledges the risk.
- Database has early-access notice (implied by section §2.3). Logs has filter input with example "WP_DEBUG, wp_die, Notice". Blueprint has Copy link + Download bundle chips.
- "Featured tiles and the full 43-card gallery with category filters and search live in the same section, with featured pinned to the top of the same grid" — collapses two §2.2 surfaces into one.
- WP Admin / Homepage / Export PR / Download .zip live as buttons next to the section header — additional actions un-hidden.
- README addresses the mobile fallback: "the iframe collapses to a sticky 220 px-tall card at the top of the viewport and the rail becomes a horizontal scrollchip strip." Sensible compromise.

## What's missing or weak
- **Apply & Reset confirm step claimed in README** ("it's gated by a confirm step in the section itself") — I see the button (`<button class="btn">Apply &amp; Reset</button>`) at line 1302 but no rendered confirm UI in the grep. Verify whether a "are you sure?" pane is actually shown.
- **The sticky iframe at 55% width means power users editing files have a narrower workspace** — README admits "multi-file editing in the Blueprint section is therefore a single-pane editor with a thinner file column, not a 50/50 IDE." Real cost.
- **The PR / GitHub / URL launcher cards** are listed but the `<select>` + `<input>` combo at line 1107–1112 is a separate widget. Two paths to start a new playground from a source (card click vs. select+paste). Worth showing how they share state.
- **"Anchors-jumps sized big and given keystrokes (1–9)"** — the left rail with numbered hotkeys is great for power users, but a first-time user won't know `2` jumps to Blueprints. Visible kbd labels on each anchor would help.
- **The "Save site locally" button** lives inside the Unsaved-Playground notice block — same as today's design. The §4 risk (users refreshing and losing work) is reduced because the iframe is always visible alongside the notice. Solid mitigation.
- **The "no modals" claim** means the confirm step for Apply&Reset has to be an inline panel — render it as a fourth state.
- **Scroll-as-only-navigation** can feel sluggish on long sessions — README admits "terrible for power users who already know exactly where to go" and offers keyboard jumps as the escape hatch.
- **The 43-blueprint gallery's "featured pinned to the top of the same grid"** — verify whether the artifact has 12+ cards visible statically vs. JS-rendered. (`<input placeholder="Search 43 blueprints…" id="bpSearch">` at line 1130 suggests JS filtering.)
- **The mobile fallback collapses the iframe to a 220px sticky card** — sensible, but the document column is then nearly full-width and the iframe becomes secondary. Worth showing.

## How to push harder
- Render the inline Apply & Reset confirm panel that the README promises. A "type the playground name to confirm" inline gate in the same section would close §4's destructive-action risk.
- Add visible `1` / `2` / `3` kbd badges to each left-rail anchor. The hotkeys are a power-user gift but they need to be discoverable.
