# Review of #080 — Multi-pane workspace

**Direction:** "Multi-pane workspace: 4 quadrants — preview, blueprint editor, file tree, logs. Each can be maximized."

**Scores:** Feature completeness 9/10 · Direction 10/10 · IA 9/10 · Visual 9/10 · Originality 9/10

## What works
- Direction faithfully delivered: four equal quadrants (Preview / Blueprint editor / Files / Logs) with pane-pickers per quadrant, maximize-to-100% + minimize-to-edge-tabs, swap and retarget controls. Workspace-level "Workspace" menu for additional actions.
- All §2.3 Site Manager tabs reframed as pane content types — Settings / Files / Blueprint / Database / Logs / Preview / Launcher / Saved Playgrounds are all just things a pane can be set to.
- Apply+Reset: "**Apply & Reset** wipes this Playground. Save it first if you want to keep your changes." (line 1451) + `pill-btn primary` "Apply & Reset" (line 1455).
- Address bar `<input value="/wp-admin/edit.php?post_type=post">` (line 1074) — §2.1 item 2.
- Database tab labelled "(early access)" inline (line 1038). Logs search input (line 1250); Files search (line 1206); Blueprint with Copy link / Run Blueprint (lines 1144–1146).
- Unsaved-Playground notice (line 993): "Unsaved Playground · Not saved to browser storage" with title attribute.
- "Network on" / "Single site" status pills at the bottom (lines 1290–1291) — sticky status indicators.
- WP Admin pill button (line 1077).
- **Named layouts**: "saved as a layout per playground — so a 'Debug' layout (Preview + Logs + DB + Files) and an 'Author' layout (Preview + Blueprint + Files + Launcher) live alongside the default 'Quad' layout. This is just a save of which-pane-shows-what, no new functionality." Brief-aware feature framing — README explicitly calls it "a remix, not a new feature".
- README is honest: "The four-quadrant idea is exhilarating on a 1440 desktop and useless on a 390 phone — so on mobile the layout collapses to a single active pane with a swipe-able pane switcher at the bottom."
- "Power users will love being able to see logs as they type a blueprint; novices will see four panes and wonder where to look — we mitigate with a labeled 'Try one of these layouts' callout on the empty default and by making the Preview quadrant the visually dominant one in the default layout."

## What's missing or weak
- **Apply & Reset has no confirm step.** README claims "a real confirmation dialog instead of a banner" — verify whether the actual confirm modal is rendered as a state.
- **Settings appear in the Settings pane content** AND verify whether there's still a quick-settings popover anywhere. README claims popover kept "as a quick variant on mobile."
- **The PR / GitHub / URL launcher tiles** are inside the Launcher pane — verify whether they expose inline paste inputs.
- **The maximize / minimize / restore interaction** — verify whether one of the states renders a maximized pane with the other three as edge-tabs.
- **Originality is high** — overlaps with #056 (Stage Manager) and #081 (Floating widgets) in the "multiple panes simultaneously" lane. #080's distinguishing move is the rigid four-quadrant grid + pane content types.
- **The named layouts ("Debug" / "Author" / "Quad") promise** — verify whether they're rendered as actual switchable states. The artifact may only show the default.
- **No real demo of pane-swap or pane-retarget** — the metaphor's strongest interaction needs visual demonstration.
- **Mobile collapse to single pane with swipe-switcher** — admitted necessary downgrade.

## How to push harder
- Render the "Debug" layout vs "Author" layout vs default "Quad" as three separate states. The named-layouts feature is the design's distinguishing claim and undelivered if only one layout is shown.
- Implement the destructive Apply & Reset confirm as a real modal — show it open with the explicit list of what's about to be wiped (file count, post count, DB size). Multi-pane workspaces invite per-pane diff views for the destructive moment.
