# Review of #059 — Fullscreen iframe + heads-up overlay

**Direction:** "Fullscreen iframe + heads-up overlay: chrome only appears on hover/edge-swipe (like a video player)."

**Scores:** Feature completeness 9/10 · Direction 10/10 · IA 7/10 · Visual 9/10 · Originality 8/10

## What works
- Direction fully delivered: iframe edge-to-edge, four edges with summon-gesture overlays (top = address + main controls; bottom = dock w/ refresh/save/settings/actions/scrubber; left = launcher; right = site manager). Symmetric IA.
- All 6 §2.2 sources rendered in the left-edge launcher (lines 675–695): Vanilla WordPress / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip — Gutenberg PR also has a sub-line ("Try a Gutenberg PR before it lands in core.").
- Apply+Reset surfaces in two places, both with explicit warning:
  - Settings popover: "Apply resets the site." + Reset / Apply & Reset button pair (lines 517–518)
  - Site Manager Settings: "Applying settings resets the Playground. Save your work first." + same button pair (lines 891–892)
- **The "Unsaved Playground" warning is a persistent translucent pill at bottom-center** (lines 629–633): "This is an **Unsaved Playground**. Refresh will wipe it. [Save site locally]" — addresses §4 risk despite the heads-up overlay metaphor. The README explicitly designed this exception: "so you cannot miss it without a wall of chrome."
- Address bar `<input value="/wp-admin/edit.php?post_type=product">` (line 493) — §2.1 item 2.
- Database has implicit early-access; Logs presumably has filter; Blueprint pane present.
- README's mitigation for non-mouse users: "a tiny pill at the top-left and bottom-center stays visible when no pointer is hovering, and keyboard shortcuts (cmd/ctrl + arrow keys) trigger the same edges." Touch + keyboard fallbacks explicit.
- "Permanent dim 'hint glow' on the left edge plus a brief tutorial of the four edges on first boot" — first-time discoverability mitigation described.
- Playground rename `<input value="Coffee Shop">` (line 839).
- "Reset" + "Apply & Reset" as two distinct buttons in the settings panel — Reset alone might be non-destructive (return to defaults), Apply & Reset is destructive. Verify the distinction is clear.

## What's missing or weak
- **Apply & Reset has no confirm step beyond the inline warn.** The two-button pattern (Reset / Apply & Reset) is good, but the Apply & Reset still fires immediately.
- **Settings appear in two surfaces** — both top popover and right-edge Site Manager. README admits.
- **The pattern rewards mouse users** — README admits. Touch fallback exists but the gesture-based UI is desktop-first.
- **Idle for two seconds and everything fades** — claimed in README. The artifact's static state doesn't necessarily show idle vs. summoned visually.
- **First-time visitors get a "tutorial of the four edges on first boot"** — undelivered in artifact.
- **The PR / GitHub / URL launcher rows** are tile rows without inline paste inputs.
- **No keyboard-shortcut legend rendered** — README mentions cmd/ctrl+arrow but doesn't render a kbd-hints display.
- **The "scrubber" on the bottom dock** — described, undelivered visually. The video-player metaphor invites a real timeline; the artifact might just show buttons.
- **Originality is high** — the edge-swipe pattern is genuinely distinct from other gallery designs (#032 radial dot, #033 pie menu both hide chrome but at center, not edges). #059's four-edge symmetry is its own move.
- **No explicit WP Admin / Homepage buttons** in chrome — they're presumably in the bottom dock or top overlay. Verify.

## How to push harder
- Render the destructive Apply & Reset as a video-player "Save before quitting?" interrupt — the iframe pauses, the overlay shakes, the destructive Confirm button requires a secondary press. Use the metaphor's existing destructive UX language.
- Demonstrate the idle-fade state as a separate state — show the iframe with no overlays at all, just the persistent unsaved pill. The "site is the star, tools are summoned" thesis needs that visual proof.
