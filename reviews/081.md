# Review of #081 — Floating widgets desktop

**Direction:** "Floating widgets desktop: everything is a draggable floating widget. Save layout per-playground."

**Scores:** Feature completeness 9/10 · Direction 10/10 · IA 7/10 · Visual 9/10 · Originality 8/10

## What works
- Direction faithfully delivered: desktop wallpaper canvas, draggable/resizable/snappable widgets for iframe + Settings + Files + Blueprint + Database + Logs + Launcher + Your Playgrounds, a bottom dock for closed widgets.
- All 6 §2.2 sources rendered in the Launcher widget (lines 991–1016): Vanilla WordPress / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip.
- Apply+Reset: "**Apply & Reset** Applying these settings boots a fresh Playground — current site data is wiped." (lines 920–921) + `btn danger` (line 923).
- Address bar shows the path `/wp-admin/edit.php?post_type=post` (line 830) — verify whether it's an input. The grep doesn't show a direct `<input>` for the URL, so the URL may be a label rather than typeable. **Potential §2.1 item 2 miss.**
- Database labelled "(early access)" inline at line 1425.
- Logs search input "Search logs..." (line 1078). Blueprint pane has Run Blueprint / Copy link / Download bundle (lines 1152–1154). 43-blueprint gallery search (line 1191).
- Unsaved-Playground label (lines 835, 854, 945). Plus a "Layout saved to 'Unsaved Playground'" toast (line 1256) — explicit confirmation that layout persistence is per-playground.
- WP Admin jump buttons (lines 794, 859).
- Download .zip in additional-actions (lines 870, 1428).
- **Layout-per-playground is the design's signature**: "Saving layout per-playground is great for power users but means switching sites also rearranges your workspace; the design accepts that as a feature — your 'writing on the train' playground can look different from your 'WooCommerce debugging' playground." Sensible framing.
- README is honest: "A draggable canvas is harder to learn than a fixed app shell — first-time users may not realize widgets are movable, so the default layout is pre-arranged and the dock is always visible with labeled icons."
- "Snap-to-edge and 'focus mode' double-click on the WordPress widget reclaim" iframe area.

## What's missing or weak
- **The address bar may not be editable.** Grep shows `<span class="path">/wp-admin/edit.php?post_type=post</span>` (line 830). §2.1 item 2 mandates a typeable input. **Verify whether the URL is actually editable in the artifact.**
- **Apply & Reset has no confirm step.** The danger button fires.
- **Settings appear in one widget (line 920)** — README claims "exactly one Settings widget — you either have it on screen or you launch it from the dock." Best single-Settings claim if delivered.
- **Floating windows hostile on mobile** — README admits, collapses to stacked column with dock at bottom.
- **The "layout-per-playground" promise** — verify whether the artifact shows two different playgrounds with different layouts (or just one default layout).
- **The PR / GitHub / URL launcher tiles** are tile names; no inline paste inputs visible.
- **Originality is high** — overlaps with #056 (Stage Manager) and #004 (Finder windows) in the "floating windows" lane. #081's distinguishing move is layout-persistence per playground + a dock for closed widgets.
- **Mobile sheet (line 1418+)** — `sh-row` items for navigation. Verify mobile state demonstrated.

## How to push harder
- Make the address bar a real `<input>` — `<span>` doesn't satisfy §2.1 item 2 if it's display-only.
- Demonstrate the layout-per-playground promise by rendering two playgrounds with visibly different widget arrangements. The "your writing-on-the-train playground looks different from your WooCommerce-debugging playground" framing is the design's strongest claim and undelivered as a visual state.
