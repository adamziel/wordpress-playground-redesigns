# Review of #041 — Print-document inspired

**Direction:** "Print-document inspired: A4 page layout, serif body, footnotes, marginalia for tips."

**Scores:** Feature completeness 9/10 · Direction 10/10 · IA 8/10 · Visual 10/10 · Originality 10/10

## What works
- This is the most thoroughly editorial design in the gallery. The Playground reframed as *The Playground Quarto*: running heads, folios, drop caps, marginalia, footnotes, signature lines, card-catalog index, A4 pages bound into a journal. The print metaphor is doing real IA work, not just decorative dressing.
- §2.3 Site Manager tabs all reframed as document sections: Colophon (Settings), Manuscript Sources (Files), Recipe (Blueprint), Datastore (Database), Errata (Logs) — accessed via thumb-index. The README's promise of "eliminating the dual-sidebar problem" via the thumb-index is delivered.
- All 6 §2.2 sources appear as table-of-contents bibliography entries (lines 1227–1257 and 1699–1704): Vanilla WordPress / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip — each with italic Latin-style sub-copy ("boot a core pull-request", "paste any blueprint JSON URL", "upload a previously bound playground").
- Address bar `<span class="typewriter" contenteditable="true" spellcheck="false">/wp-admin/edit.php?post_type=post</span>` (line 1028) — typewriter-styled but editable. Note: a `contenteditable` span is not a form `<input>` but is keyboard-typeable; §2.1 item 2 satisfied via the print metaphor.
- Apply+Reset has the strongest editorial framing in the gallery:
  - Settings section: "Warning — Apply & Reset will erase the plate. Sign your work first." (line 1119)
  - Footnote-style: "Reset. Choosing Apply Settings & Reset Playground tears the plate from its binding and reprints it from scratch — files, posts, plugins, the lot. There is no undo. Cf. Colophon, p. 3." (line 1145)
  - Final colophon block: "Applying these settings *resets the plate*: all posts, plugins and uploads will be unbound." + "Apply & Reset is destructive. The warning rule is red ink, intentionally." (line 1395, 1407)
  
  Multiple cross-referenced warnings with red ink, footnote anchors, and explicit metaphor cues. Excellent.
- Database labeled `§ IV — The Datastore (early access)` (line 1542). Logs (Errata) has search input. Blueprint has Run Blueprint / Copy link / Download bundle as ink-buttons.
- Unsaved-Playground notice exists ("The Unsaved Page" — line 1127) with metaphor-appropriate copy.
- Marginalia replaces tooltips per README — "friendly typeset tips" on every page.
- Mobile keeps the cover-page aesthetic + flattens marginalia into bottom footnote rail (line 1661+) — admitted compromise.
- Bind & Download as .zip / Copy link to current plate are in a menu (lines 1095, 1099).

## What's missing or weak
- **No actual confirm step for Apply & Reset.** The footnote/warning copy is the strongest in the gallery — but there's no inline confirmation gate (e.g., "Type the playground name to confirm" or "Sign here ____"). The metaphor invites a signature-pad confirmation; the design renders only warnings.
- **Settings appear in two surfaces**: the colophon settings (line 1114) AND a later colophon block (line 1388). They're both labeled as colophon but they're two separate panels. Same data, two visible pages.
- **Information density is lower than a conventional toolbar** — README admits. Serif type + generous leading = fewer affordances per pixel.
- **The PR / GitHub / URL launcher entries are bibliography lines** — clicking should reveal a paste field; the artifact only shows the typeset entry.
- **"Power users may find the playful chrome (folios, footnotes) slow on first encounter"** — README admits. Genuine speed cost.
- **The card-catalog metaphor for the 43-blueprint index** is on-metaphor but loses thumbnail recognition.
- **The print metaphor strains slightly on Database and Logs surfaces** — README admits and styles them as errata sheets / catalogue cards. Database as "Datastore" is fine; SQL queries inside a "Datastore" page may read oddly.
- **The "Apply" pick/reset pick interaction (line 1116: `<span class="pick">apply</span> · <span class="pick">reset</span>`)** is unclear — is this two separate verbs (apply changes vs. apply + reset)? The print metaphor obscures the actual button affordance.
- **Mobile flattening marginalia into footnote rail** — sensible but means the typeset narrative loses depth on small screens.
- **The "Sign your work first" save copy** is delightful but a first-time user landing on it may not parse "sign = save". README does mitigate with explicit labels in the toolbar.

## How to push harder
- Render the destructive Apply & Reset as a literal "sign here ____" line — an inline `<input>` styled as a signature-rule that, when typed, enables the reset button. The print metaphor's strongest existing convention (signing a manuscript) maps perfectly to commit-to-destructive.
- The "Apply / reset" disambiguation needs an unambiguous button or two — picks separated by `·` are too subtle. Use a clear distinction between "apply (no reset)" and "apply + reset" if both exist, or fold into one.
