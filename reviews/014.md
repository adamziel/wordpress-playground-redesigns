# Review of #014 — macOS Settings.app (Ventura style)

**Direction:** "macOS Settings.app (Ventura style): sidebar of categories. Right pane swaps. The iframe lives in its own 'Live Preview' category."

**Scores:** Feature completeness 9/10 · Direction 9/10 · IA 8/10 · Visual 9/10 · Originality 7/10

## What works
- Ventura-faithful sidebar with grouped section headers (Playground / Authoring / Data), real searchable list, row icons in colored squircles. The metaphor is consistently applied across 5+ state demonstrations.
- The "Live Preview" category is the default selection, putting the iframe at full pane width — directly addresses the README's stated trade-off and keeps the iframe-first user happy.
- All 6 §2.2 sources rendered as `new-chip` buttons in the Saved Playgrounds pane: Vanilla WordPress / WordPress PR… / Gutenberg PR… / From GitHub repo… / Blueprint URL… / Import .zip. Complete and consistently labelled with trailing `…` (signalling further input).
- The README's split of settings into "WordPress Version" and "PHP & Runtime" rows is delivered — but actually I see in the final state (line 1745) an "Apply & Reset" button with sibling rows for Networking + Multisite, AND the destructive warning copy ("Apply & Reset is destructive — your WordPress database, uploads, and theme tweaks will be wiped"). Clear and specific.
- Address bar is a real `<input value="/wp-admin/edit.php?post_type=product">` with a `<button class="go">Go</button>` next to it — explicit submit affordance.
- Database has the early-access notice with GitHub-tracker link. Logs has real timed entries (HTTP routes, etc.). Blueprint pane has Run Blueprint + Download bundle + Copy link.
- Unsaved-Playground notice quoted verbatim: "This is an Unsaved Playground. Your changes will be lost on page refresh." + Save site locally button.
- Mobile is described as "iOS Settings.app-style stack of rows; tapping pushes a detail view (CSS-only)" — though I don't see the explicit mobile demo in the artifact, the long sidebar with row-arrows would adapt naturally.

## What's missing or weak
- **The README claims "the three settings entry points collapse into a single canonical category"** but the artifact has TWO settings categories ("WordPress Version" + "Network & Multisite") that surface the same data points. The semantic split feels right (different rows for different concerns) but the README claims "one home per setting" — Network & Multisite share a row, while WP version and PHP get separate categories. The taxonomy is inconsistent.
- **Apply & Reset has no confirm step.** Despite explicit destructive copy, the button fires immediately. A native macOS confirm sheet (anchored to the window's title bar, with a "Save first" safe-default) would be on-metaphor and missing.
- **The "Saved Playgrounds" pane combines saved sites list + "New Playground" toolbar pill** per the README, but the artifact has the chip row prefixed with "Import .zip" elsewhere as a button (line 1202) AND in the `new-chip` row (line 1220). Import has two surfaces.
- **WP Admin / Homepage jump buttons** are not visible in the chrome — they exist behind the address bar (Go button) but not as the §2.3-mandated dedicated buttons.
- **The 12-row sidebar is long.** The README admits this and adds section headers + filter; that helps. But "Live Preview" being at the top is good only for returning users — first-time landing on a settings-app shell may not parse it as "this is the running site".
- **"Live Preview" as a category demotes the iframe semantically.** The brief (§2) frames the iframe as the dominant viewport that the chrome surrounds; this design inverts that. It's an interesting reframe but worth acknowledging that the inversion may confuse "I came here to use WordPress" users.
- **The trade-off the README admits ("getting to the live site is now one click on Live Preview instead of the whole screen IS the live site")** is real, and the design doesn't compensate beyond making it the default selection. Returning users opening to a different category have an extra click vs. today's single-iframe-everywhere model.
- **The "Network & Multisite" sidebar row appears in multiple state demos** (lines 1038, 1177, 1336, 1442, 1625) — fine for consistency demo purposes, but the actual interaction (toggling them does what?) is not shown across the states.
- **No visible WordPress PR / Gutenberg PR / Blueprint URL input fields rendered** — even though their entries have `…` suffix indicating "more steps", the actual paste field is never demonstrated.

## How to push harder
- Render a native-feeling macOS confirm sheet for Apply & Reset — a sheet that drops from the window's title bar with "Apply settings and reset playground? This will erase the WordPress database, uploads, and any theme tweaks." + a "Save first" default button. Use the metaphor's strongest pattern.
- Reduce the 12-row sidebar by collapsing "WordPress Version" + "PHP & Runtime" + "Network & Multisite" into a single "WordPress" category (matching the brief's §2.3 Settings tab), and let the Apply+Reset bar at the bottom be shared. The current taxonomy invents categories the brief doesn't have.
