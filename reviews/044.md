# Review of #044 — Wapuu-centric mascot UI

**Direction:** "Wapuu-centric mascot UI: Wapuu (WP mascot) lives in a sidebar, narrates state changes, recommends blueprints. Stop short of being annoying."

**Scores:** Feature completeness 9/10 · Direction 10/10 · IA 8/10 · Visual 8/10 · Originality 8/10

## What works
- The "stop short of being annoying" constraint is taken seriously and honored. Wapuu's bubbles are color-muted, max one bubble visible at a time, no idle animations, with an explicit "Mute Wapuu" button in the rail footer (line 781). Anti-Clippy guardrails are not just claimed; they're rendered.
- Wapuu narration replaces low-severity toasts but real errors still fire — preserves §2.4 toast region functionality while adding the mascot layer.
- All 6 §2.2 sources rendered (lines 1154–1175): Vanilla WordPress / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip — each with a sub-line describing the input expected. Plus "Tip from Wapuu" inline guidance for PRs.
- Apply+Reset has the Wapuu-warning model:
  - Primary surface: "Apply & Reset Playground — Applying these settings will wipe the current site contents and reboot WordPress. Wapuu will warn you again before this fires." (line 952–953) + `btn-danger` (line 954)
  - The "will warn you again" sub-line explicitly promises an interrupt before commit — which is the kind of safety pattern other designs don't articulate.
- Address bar `<input id="addrInput" value="/wp-admin">` (line 796) — §2.1 item 2.
- "Wapuu recommends" section (line 756) — surfaces 2 contextual blueprints, which fits the README's "recommends two relevant blueprints" claim.
- "Wapuu remembers: last week you booted Stylish Press three times. Want to save it as a starting template?" (line 1255) — context-aware suggestions based on usage. Doesn't invent personalization features, just exposes the saved-history that already exists.
- Database (early access via §2.3 implicit). Logs has search input (line 1098). Blueprint has Copy link / Download bundle.
- Companion Rail collapses to a 56px icon strip via toggle. Mobile becomes a bottom-sheet + FAB. Trade-offs explicitly defended.
- README is sharply honest: "The biggest risk is that power users will mute him on day one and never see his recommendations — accepted, because the UI still works fully without him; he is a *layer*, not a load-bearing element." Best framing of mascot-design risk in the gallery.

## What's missing or weak
- **Settings appear in two surfaces** — README admits "both kept, but unified visually so they read as the same form." Same data, two homes.
- **Apply & Reset still has no rendered confirm step.** The README/inline copy promises "Wapuu will warn you again before this fires" but the actual second-step warning UI isn't demonstrated. Big claim, undelivered visual.
- **The Companion Rail costs 280px on desktop** — significant. Mitigated by collapse toggle but a power user with a 1366×768 laptop loses ~20% of horizontal real estate by default.
- **Wapuu's bubbles "max one visible at a time"** but the timeline persists — a "View timeline" link surfaces history. The timeline UI itself isn't necessarily rendered as a separate state.
- **Originality has overlap with #043 (Children's book)** — both lean on Wapuu as a mascot. #043 makes the mascot the visual identity; #044 makes the mascot a narration *layer* on top of a professional blue UI. The #044 framing is the more mature take — "personality without losing professional surface". Worth recognizing as a real differentiation.
- **The PR / GitHub / URL launcher rows** are sub-line-described cards; no inline paste input. §3 flows 11–12.
- **The "Apply & Reload" button (line 846) vs "Apply & Reset" button (line 954)** — two different labels for what may be the same destructive action. Verify whether "Apply & Reload" is non-destructive (cosmetic restart) vs. "Apply & Reset" (wipe). Disambiguation needed.
- **"Mute Wapuu" button** is great — but the inverse "Bring Wapuu back" isn't clearly demonstrated. After mute, how does the user re-enable narration?
- **The "Roastery demo" / saved-playground rename input (line 889)** is present — playground rename satisfied.
- **The "Wapuu remembers" personalization-style suggestion at line 1255** — this is a smart use of session history, not an invented feature. But it implies persistence beyond OPFS (the brief is OPFS-only) — verify that the session count is just an in-memory record.

## How to push harder
- Render the "Wapuu will warn you again before this fires" interrupt as a real state. The promise is the design's strongest safety claim; making the warning bubble appear as a confirm modal with two buttons would close the loop.
- Disambiguate "Apply & Reload" vs "Apply & Reset" — either rename one or wire them through a single destructive path. Two button labels for the same action invites mistakes.
