# Review of #063 — Reader-first

**Direction:** "Reader-first: the iframe is paramount, taking 75% of width. Everything else collapses into a thin vertical rail with icon-only buttons + tooltips."

**Scores:** Feature completeness 9/10 · Direction 10/10 · IA 8/10 · Visual 8/10 · Originality 6/10

## What works
- Direction faithfully delivered: 56px vertical left rail with icon-only buttons + tooltips, 75% iframe canvas, slide-in 360px panels (never modals, never overlays).
- All 6 §2.2 sources rendered as launcher panel rows (lines 1338–1363): Vanilla WordPress / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip — each with descriptive sub-line.
- Apply+Reset has explicit destructive copy in two surfaces:
  - Main Settings panel (line 1308): "Apply Settings & Reset Playground" + "Apply is destructive — your current site contents will be wiped and a new playground will boot with these settings." (line 1312)
  - Quick-settings panel (line 1505): `btn danger` "Apply & Reset"
- Address bar `<input id="url-input" value="/">` (line 1719) — §2.1 item 2.
- Database has the early-access notice with GitHub-issue-tracker link verbatim (line 1619). Logs has search "Search PHP error log…" (line 1656); Files has search (line 1524); Blueprint has Run Blueprint button (line 1572).
- Unsaved-Playground notice: "**This is an Unsaved Playground.** Your changes will be lost on page refresh." + "Save site locally to browser →" button (lines 1248–1249).
- WP Admin row with `/wp-admin` meta (line 1788) — chrome jump button.
- "Show labels" toggle described in README — pins names beside icons (78px rail) for users who don't trust icons.
- README is honest: "every action is one extra click behind a rail icon — power users who lived in the old sidebar will feel the indirection."
- Mobile: rail flips to bottom dock with same icons + bottom sheets — different idiom, same vocabulary.
- "These mirror the full Settings panel" (line 1485) — explicit acknowledgment of the two-surface settings.

## What's missing or weak
- **Apply & Reset has no confirm step.** Both surfaces fire the destructive action immediately.
- **Settings appear in two places** — Quick-settings strip (`Apply & Reset` button) AND the full Settings panel. README admits "These mirror the full Settings panel — pick a version and Apply to re-boot."
- **Originality is medium** — overlaps with #017 (Linear/Vercel minimal) and #009 (Storybook) in the "icon rail + slide-in panel" lane. #063's distinguishing move is the explicit 75% iframe canvas requirement; other designs share the rail-and-panel pattern.
- **Power users feel the indirection** — README admits.
- **Icon-only rail relies on user noticing iconography** — README mitigates with "Show labels" toggle that pins names beside icons.
- **The PR / GitHub / URL launcher rows** are listed with sub-lines but no inline paste inputs.
- **Tooltips do heavy labeling work** — first-time discoverability cost.
- **WP Admin / Homepage live in the playground-name menu** at the rail's bottom — verify they're surfaced as direct rail icons or only as overflow.
- **The destructive "Apply & Reset" notice copy at line 1312** is strong, but the button is right above it — a user might tap before reading.
- **Mobile bottom-dock + sheets** — described, verify rendered state.

## How to push harder
- Render the destructive Apply & Reset as a confirm modal — even a minimal "Are you sure? This is destructive." overlay would close §4 risk.
- Differentiate from #017 and #009 by emphasizing the "reader-first" reading-flow optimization: e.g., a "Focus mode" toggle that hides even the rail to give 100% canvas, with a top-edge swipe to bring back the rail. The brief's "reader" framing invites this.
