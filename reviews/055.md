# Review of #055 — iPad multitasking

**Direction:** "iPad multitasking: split-view with the iframe + a secondary pane you can swap (files / blueprint / logs). Pull-tab to resize."

**Scores:** Feature completeness 9/10 · Direction 10/10 · IA 8/10 · Visual 9/10 · Originality 8/10

## What works
- Direction faithfully delivered: iframe as primary app, Split View with pull-tab pane chooser, draggable center divider with 30/50/70/100% snap zones, Slide Over card for launcher, frosted Dock at bottom for saved playgrounds, iPadOS-style status bar with Control Centre tile for quick settings.
- All 6 §2.2 sources rendered as start-tiles in the Slide Over card (lines 907–912): Vanilla WordPress / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip — each with sub-line ("Boot from core PR #", "Paste any blueprint JSON").
- **Apply+Reset has a real Cancel + Apply two-button confirm (line 887)** with warning copy: "Apply will wipe the current site. Save first if you want to keep this state." (line 886). The README claims iPadOS action-sheet pattern; this delivers it.
- Address bar `<input value="" placeholder="wp-admin">` (line 835, 983) and `<input value="wp-admin/post-new.php">` (line 1048) — §2.1 item 2.
- Unsaved-Playground state visible in multiple places: title-chip with alert-triangle (line 837), Dock icon with `unsaved` class (line 803), full notice in Slide Over (line 946) "Unsaved Playground — Not saved to browser storage. Changes will be lost on refresh."
- WP Admin / Homepage as iconbtn in chrome (line 838 — "Save site locally" iconbtn accent).
- Database / Logs / Files all surface as secondary-pane content per the multitasking pull-tab.
- README is honest: "First-time desktop visitors may not immediately understand that the grey pill at the top of a pane is the pane-swap affordance, so I labeled it on hover and on first launch."
- "Split View only shows one secondary pane at a time, so you can't see Files and Logs simultaneously — that's true on real iPadOS, and on smaller breakpoints I trade Split View for a Slide-Over secondary panel instead." Real-iPadOS-fidelity defended.
- Mobile degrades to single-primary-app + bottom sheet = iPhone behavior. Recognizable fallback.

## What's missing or weak
- **The Cancel + Apply pattern at line 887 is the destructive confirm** but it's still a single-step confirm — no "type the name" gate like #066 or #040. Could be stronger.
- **Settings appear in the Control Centre tile (lines 867–887)** AND potentially in the Settings pane (Split View secondary). README claims "three settings entry points collapsed into one Control Centre tile" — verify there's no Settings tab in the split-view.
- **The "grey pill at the top of a pane is the pane-swap affordance"** is admitted as low-discoverability — labeled on hover + first launch.
- **The Split View limit (only one secondary pane at a time)** is real iPadOS but limits power users — README admits.
- **Dock metaphor scales to 8–10 saved sites; overflow chevron handles long tail** — described.
- **The PR / GitHub / URL start-tiles** are tiles with sub-lines, no inline paste inputs.
- **Originality is medium** — overlaps with #014 (macOS Settings) in Apple-system-lookalike territory, but the iPad multitasking specifics (Split View, Slide Over, Dock) are distinct.
- **The "filter:blur(4px) brightness(0.85)" iframe canvas in the launcher state** (line 990) is a nice iPadOS effect — Slide Over dims background. On-metaphor.

## How to push harder
- Promote the destructive confirm to require typing the playground name (like #066). The "Cancel + Apply" pattern is good but the typed-confirmation is the strongest existing pattern in the gallery.
- Render the pull-tab affordance with a small "Tap to swap pane" first-time tooltip — the README acknowledges the discoverability cost; static rendering of the mitigation closes it.
