# Review of #004 — Mac Finder windows

**Direction:** "Mac Finder windows: each playground is a 'window' with its own chrome. Title bar with traffic-light buttons. Multiple windows can be open as tabs in a desktop-like shell."

**Scores:** Feature completeness 9/10 · Direction 8/10 · IA 6/10 · Visual 8/10 · Originality 7/10

## What works
- Genuinely commits to the desktop metaphor: menu bar with File/Edit/View/Site/Blueprint/Window menus, dock, desktop icons, real traffic-lights with the close/min/zoom glyphs revealed on hover. SF Sonoma palette and translucency are convincing.
- Every §2 feature is at least staged: refresh, URL bar with `playground.wordpress.net/?page_id=2`, name pill ("Coffee Shop — Playground"), Save button, /wp-admin and / quick-jumps, additional actions menu containing "Export to GitHub PR" and "Download .zip" verbatim from §2.3.
- Inspector ("Get Info") cleanly merges the three settings entry points into one panel (WP / PHP / Language / Network / Multisite / persist toggle), and the Apply+Reset button carries an explicit red note about losing files (better than most designs will manage on this).
- Blueprint Finder window genuinely lists ~42 individual blueprints in column 2 (the most complete in the gallery I've seen so far) and matches §2.2's category list All / Featured / Website / Personal / Content / Themes / Gutenberg / Experiments / WooCommerce / News.
- "Locations" sidebar surfaces "New From…" with all six required entries (Vanilla / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip), and a dedicated Save dialog with name field is shown.
- DB window has the early-access notice text, file editor has gutter+syntax tokens, Logs has timestamps + level chips (BOOT / INFO / WARN / ERR / OK). All five Site Manager tabs (§2.3) are present as window types: Site / Files / Blueprint / DB / Logs.

## What's missing or weak
- **IA is "look at the screenshot," not "use the app."** The deliberately overlapping windows demo skeuomorphism beautifully but the design has no statement about which window manages which playground. If a user has two saved playgrounds, do they get two iframe windows? The README says "multiple windows can coexist" but the artifact only shows one running site + helper windows attached to it. The "tabs in a desktop-like shell" from the direction is not actually demonstrated — there is no tab strip.
- **No Logs filter/severity affordance.** Logs window has level chips painted into each row (INFO / WARN / ERR / OK / BOOT) but there is no level filter UI, just a "filter logs…" placeholder that does nothing. §4 calls this out as a current pain point and the design didn't address it.
- **Blueprint tab "File browser column" overlap with the Files tab is unresolved.** §4 explicitly flags this duplication. Window #3 has a "Bundle" sidebar but doesn't differentiate from the standalone Files window's tree.
- **"Apply & Reset" is in the additional-actions menu marked `danger`, in the inspector with a red note, AND in the dock as the Trash icon.** That's three destructive entries with three different metaphors and no confirmation step — the inspector's "Apply & Reset" button fires directly with no diff/confirm.
- **No visible affordance for typing a path into the URL bar.** It is rendered as a static `<span>` (`/?page_id=2`) — not an `<input>`. §2.1 item 2 requires submit-to-navigate; without an input it's a label.
- **Blueprint preview pane fabricates fields** ("Boot time ~3.4 s", "Plugins: WooCommerce, Stripe (test)") that don't exist in §2.2's data model. Minor but it's added scope.
- **WP-PR and Gutenberg-PR entries in the sidebar/launcher are clickable rows with no input affordance.** The "New From…" grid in Window 7 finally shows the input copy ("Paste a PR number"), but only as descriptive text, not as a field. §3 flows 11–12 want an actionable input.
- **Menu bar items (File / Edit / View / Site / Blueprint / Window / Help) are decorative** — no dropdowns rendered. The README claims this provides "a complete keyboard-discoverable command surface" but you only see the labels.
- **Mobile shrinks to a single phone view** that drops the "Window" metaphor entirely — fair fallback per §5, but the bottom tab bar exposes Site/Blueprints/Yours/Settings/More with no Files/Blueprint/DB/Logs tabs. Those features are buried under "More" with no indication of which Site Manager tab the user would land on.

## How to push harder
- Actually deliver the "tabs in a desktop-like shell" promise: a tab strip on the active window where each tab is a different running playground. Right now multiple playgrounds live in one Finder list — the windows are roles (Site, DB, Logs), not sites. Pick a lane.
- Make Apply+Reset a real macOS confirm dialog (system-modal sheet attached to the window's titlebar, with "Save first" as the safe default button), and remove the dock Trash + danger menu duplicates. One destructive surface, not three.
