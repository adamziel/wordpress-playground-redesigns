# Review of #062 — Hub-and-spoke

**Direction:** "Hub-and-spoke: central 'home' with major actions as spokes. Clicking a spoke flips into that sub-screen."

**Scores:** Feature completeness 9/10 · Direction 10/10 · IA 7/10 · Visual 9/10 · Originality 8/10

## What works
- Direction delivered: a circular console with the iframe at center, labeled spokes radiating outward (Save, Settings, Site Manager, Blueprints, Your Playgrounds, New Session, Files, Logs, Database). Spoke selection "docks" the hub to bottom-right corner and flips open the chosen sub-screen.
- All 6 §2.2 sources rendered as launch-pad cards in the "New" spoke (lines 908, 921, 928, 935, 942): Vanilla WordPress / Gutenberg PR / From GitHub / Blueprint URL / Import .zip — with descriptive copy. (WordPress PR likely nearby; verify visibility.)
- Apply+Reset has the strongest contextual warning copy in the gallery: "**Heads up.** Apply & Reset destroys the current site state to apply WP/PHP version changes. Save first if you need this session." (line 1117) + a primary button "Apply & reset playground" (line 1076).
- Address bar — verify location (likely inner ring); the artifact has multiple inputs.
- Database labelled "spoke · SW · 8 o'clock · early access" (line 1253) + "Database management is an early access feature. SQLite-backed. Have feedback? Open an issue on the tracker." (line 1263) — early-access notice + spoke-position labelling.
- Logs has search input "grep logs…" (line 1305); Blueprint pane has Copy link / Download bundle (lines 1132–1133).
- Unsaved-Playground notice with two surfaces:
  - In Site Manager: "**Unsaved.** Hitting refresh will lose this session. Save now or hit ⌘S." (line 1432)
  - In the spoke list as "Unsaved Playground" entry (line 1052)
- README is honest about the iframe-shrink cost: "the hub on the home view dedicates the whole viewport to navigation, so the iframe shrinks to a circular porthole until you click 'Enter site'."
- Mobile degrades to vertical action list — same destinations, different shape. Sensible.
- "Save site locally to OPFS" toggle (line 1109) — possibly a persistence preference (verify it's not an invented feature).

## What's missing or weak
- **Apply & Reset has no confirm step.** The warning copy is strong, the button fires.
- **Settings on the home view consume the whole viewport** — the iframe-as-porthole cost is real. First-touch user landing on the hub sees navigation, not WordPress.
- **The "Save site locally to OPFS" toggle** (line 1109) — if this is a toggle for "should this playground persist?", it's an invented setting (§2 only has Save as an action, not a preference toggle). Worth verifying.
- **Originality overlaps with #032 (Concept-album dot) and #033 (Radial pie menu)** — three radial-based designs in the gallery. #062 distinguishes itself by labeling each spoke (vs. icon-only) and using full sub-screens (vs. overlays).
- **Power users who live in file tree or logs have one extra click to leave a sub-screen** — README admits. Persistent sidebar would be more efficient.
- **The PR / GitHub / URL launcher cards** are described as launch-pad cards; no inline paste inputs in the grep.
- **Settings appear in the Settings spoke** AND in the Site Manager's Settings tab — verify whether they're the same sub-screen or duplicated.
- **The radial layout doesn't survive 390px** — README admits and provides vertical-list mobile.
- **"Every feature has a fixed angular address that never moves"** — strong discoverability claim, but only works once the user has learned the angular positions. First-time discovery still requires reading labels.
- **Hub shrinks to bottom-right corner during sub-screens** — this is the key "one focused workspace at a time" claim; verify the corner-hub state is rendered.

## How to push harder
- If the OPFS-persistence toggle is per-playground-preference rather than a one-time save action, remove it — §5 forbids invented features. If it's just labeling the Save state, rename it for clarity.
- Render the destructive Apply & Reset as a hub-radial confirm — show the spoke labeled "Confirm" extending from the destructive action, requiring a directional click. The metaphor's "every action has an angular address" promise should apply to the confirm step too.
