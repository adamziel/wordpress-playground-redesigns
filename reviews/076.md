# Review of #076 — Drag-and-drop blueprint builder

**Direction:** "Drag-and-drop blueprint builder: visual canvas. Drag steps (install plugin, set option, create user) into a flowchart. Run = boot."

**Scores:** Feature completeness 9/10 · Direction 10/10 · IA 8/10 · Visual 9/10 · Originality 9/10

## What works
- Direction faithfully delivered: n8n/Node-RED visual canvas with draggable nodes, left Library rail (Templates / Yours tabs), bottom Inspector dock that swaps content based on selected node, iframe in docked Preview panel that updates on Run.
- All 6 §2.2 sources rendered both as Quick-Start cards (lines 724–729) AND as Library-rail items (lines 803–808): Vanilla WordPress / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip — explicit duplication for discoverability.
- Apply+Reset: `<button class="apply">Apply & Reset</button>` (line 713) + "**Heads up.** Apply re-boots the playground and wipes all unsaved data." (line 716).
- Address bar `<input value="/wp-admin/post-new.php">` (line 669) — §2.1 item 2.
- Database / Logs are inspector-dock contents when their respective nodes are selected per README. Blueprint pane has Copy link (line 1030).
- Unsaved-Playground notice with explicit copy + Save button (line 762).
- Library search input "Search blueprints, steps…" with `⌘ K` keyboard hint (line 799).
- "Environment" node at the top of the flow per README — the settings popover + Site Manager Settings + inline save notice all collapse into this first-class node.
- Additional actions in Run dropdown per README: Download .zip / Copy link to blueprint (lines 775, 783) with descriptive copy.
- Featured wins:
  - The blueprint moves from hidden JSON file to **primary surface** — the design's central thesis.
  - "Run button is a giant primary control — you can ignore the canvas entirely and the app behaves like the current Playground" — first-touch user mitigation.
  - "The canvas pre-loads a one-node 'Boot WordPress' flow" — sensible default.
- README is honest: "Wire-drawing is not implemented as live SVG dragging here; nodes are auto-wired in execution order, which is what 95% of real blueprints look like in practice." Acknowledges interaction limitation.

## What's missing or weak
- **Apply & Reset has no typed-confirm gate.** The danger copy is good; the button fires.
- **Settings appear in a popover (line 710–716) AND as an Environment node** per README — verify whether the Environment node is the canonical source and the popover is just a quick-edit alias.
- **The canvas eats screen real-estate the iframe used to own** — README admits, mitigates with Preview tab on small viewports.
- **Wire-drawing not implemented as live SVG dragging** — README admits. Auto-wired in execution order. Real cost for the drag-and-drop direction.
- **The PR / GitHub / URL Library items** are tile faces; no inline paste inputs visible.
- **Mobile degrades canvas to vertical card list (step list, still draggable to reorder)** — sensible.
- **Originality is high** — overlaps with #022 (DAW/Ableton clip-as-step) in the visual blueprint authoring lane. #076 commits to flow-graph; #022 commits to track-lanes. Both are legitimate direction interpretations.
- **The "linear users who just want vanilla WordPress" mitigation (giant Run button)** is described but the canvas's presence still adds visual noise even when ignored. Pre-loaded one-node flow helps.

## How to push harder
- Implement the live SVG wire-drawing — even if 95% of real blueprints are linear, the metaphor's strongest interaction is dragging a wire between nodes. Render at least one branching flow to prove the canvas concept.
- Add a typed-confirm gate for Apply & Reset — "Type 'reset flow' to wipe the canvas and reboot." The canvas metaphor invites the language: clearing the canvas = clearing the playground.
