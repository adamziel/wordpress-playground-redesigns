# Review of #017 — Linear / Vercel / minimal

**Direction:** "Linear / Vercel / minimal: very tight, mostly monochrome, sharp typography, accent color sparingly. Keyboard-driven."

**Scores:** Feature completeness 9/10 · Direction 10/10 · IA 8/10 · Visual 9/10 · Originality 7/10

## What works
- Direction adherence is exemplary. Inter + JetBrains Mono, hairline borders, single accent color (indigo `#5E6AD2`-style) reserved EXCLUSIVELY for the Apply & Reset destructive primary — explicitly framed by the README and respected in the artifact ("accent (used once: Apply & Reset)" labelled at line 1598).
- Command palette (`⌘K`) is the actual primary navigation: launcher entries (Vanilla / WordPress PR… / Gutenberg PR… / From GitHub… / Blueprint URL… / Import .zip…) each appear with descriptive subtext explaining the action, AND keyboard shortcuts visible (`N V`, `N P`, `N G`, `N H`, `N B`, `N I`). Every entry has a `…` suffix signalling the input step.
- Two palette modes shown: command/launcher mode AND a blueprint filter mode (`Filter 43 blueprints… /tag, /featured, /woo, /experiments`). Genuine search-with-tags.
- Apply & Reset has both:
  - Inspector inline destructive notice: "Applying settings will reset the playground. All unsaved data inside the iframe will be destroyed." + `Apply & Reset ↵` button.
  - A separate "Apply" status row with the same button — so the user sees the destructive action surfaced even when they're not in Settings.
- The §4 lose-on-refresh pain is genuinely addressed: persistent yellow dot on the playground label + the `⌘S` shortcut hint animates on first interaction (per README). Better than the typical "small text" baseline.
- Logs has filter input with example "level:error, /woocommerce" — directly addresses §4 logs pain point. DB browser has SQL field placeholder "SELECT * FROM wp_options WHERE …" + early-access notice.
- Blueprint pane has Copy link / Download / Run Blueprint as separate buttons. Files tree has its own pane. Inspector tabs: Settings / Files / Logs / DB.
- Mobile composition rendered (line 1481+) with same minimal aesthetic and bottom-sheet inspector.
- Color/typography legend at the bottom (line 1598) is a thoughtful Linear-style "design tokens" reveal — and confirms the single-accent constraint is real.

## What's missing or weak
- **The address bar input might be missing.** I grepped for `<input value="/wp-admin"` and didn't see it for design 017 specifically — the toolbar may treat the URL as a label opened by `⌘K` URL mode (which is what the README claims on mobile). Verify whether desktop has a real input or punts entirely to the palette.
- **Apply & Reset still has no confirm step.** The destructive notice is good, but Linear's actual UI prompts "Are you sure?" inline. The keyboard shortcut `↵` would fire the destructive action immediately.
- **The README claims "the Site Manager + Saved Playgrounds drawer + Edit-Settings popover all collapse into one Inspector"** — but the artifact also has a "context strip" at the top, an inspector with multiple tabs, AND the command palette opening categorically different views. The single-inspector claim partially holds; "all" is a stretch.
- **Files tab's "Inspect" sub-pane for Blueprint+Database** per the README — this is the file-tree-in-three-places consolidation, but it adds an extra click for users who today have dedicated Blueprint/DB tabs. README admits this trade-off.
- **Monochrome means no thumbnail recognition for blueprints** — README acknowledges this and compensates with monospace tags + category chips. Fair trade-off, worth noting that visual hierarchy among 43 blueprints suffers when they all look like text rows.
- **The launcher palette and the rail-item palette both list the same 6 sources with shortcuts** — that's intentional (palette + a sidebar's quick-pick rail) but the user must remember which surface they're using. Slight redundancy.
- **No visible "Save Site Locally" surfacing in the toolbar.** It exists in the inspector Settings tab ("Save site locally ⌘S") but not in the iframe-adjacent chrome. A returning user on a tab refresh has to know to invoke the inspector.
- **WP Admin / Homepage jump buttons** — I see "Open Database" and various inspector verbs in the palette but not dedicated WP Admin / Homepage buttons in the chrome. The §2.3 Site Manager header buttons might be inside the inspector header (line 1054 has Copy link / Download bundle icon-btns) — verify whether WP Admin and Homepage are actually surfaced.
- **The animated ⌘S "first interaction" hint** is a great mitigation but it's described, not rendered — the artifact shows the static state. A keyframe or implementation note would close the loop.

## How to push harder
- Render the URL bar as a real `<input>` in the toolbar — not just behind the palette. Linear's URL bar is real, not modal. Linear-clones that route everything through ⌘K often lose this and the design is one input away from satisfying §2.1 item 2 cleanly.
- Implement the Apply&Reset confirm as a 2-keystroke inline interaction: pressing `Apply & Reset` triggers a "Type 'reset' to confirm" inline input — exactly the Linear-y pattern for destructive actions. Right now the keyboard verbs are everywhere except where they matter most.
