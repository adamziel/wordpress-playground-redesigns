# Review of #035 — Brutalist web 1.0

**Direction:** "Brutalist web 1.0: monospaced, system fonts, 1px borders, brutalist newspaper grid. No rounded corners, no shadows."

**Scores:** Feature completeness 10/10 · Direction 10/10 · IA 8/10 · Visual 9/10 · Originality 9/10

## What works
- Direction commitment is total: monospace everywhere, 1px hairline rules, all-caps section headers, `<fieldset>`+`<legend>` semantic HTML, `[BUTTON]`-bracketed labels, ruled section numbers (§1, §2, §3 ...), `<pre>`-formatted log block. The page reads like a Library of Congress reference document.
- **Every PR / GitHub / URL launcher has a real working input field rendered inline:**
  - WordPress PR: `<input type="text" value="https://github.com/WordPress/wordpress-develop/pull/9182">` (line 630)
  - Gutenberg PR: `<input type="text" value="68421">` (line 638)
  - From GitHub: `<input type="text" value="github.com/WordPress/twentytwentyfive">` with radio "Theme/Plugin/wp-content" (lines 646–650)
  - Blueprint URL: `<input type="text" value="https://playground.wordpress.net/blueprints/coffee-shop.json">` (line 659)
  - Import .zip: `<input type="file">` (line 666)
  This is the most complete implementation of §3 user flows 11–12 in the gallery. Every source has its native input affordance visible.
- All 6 §2.2 sources rendered as named `<header>` blocks inside the Start section + a separate Vanilla WordPress row in the Saved table — complete inventory.
- Apply+Reset has explicit destructive copy with `<b style="color:var(--hot)">reset the running playground</b>` (line 590) + "There are no in-place upgrades" — explicit constraint statement.
- Database labelled `— SQLite, early access` (line 795) directly on the section header. SQL query input present (line 817). Logs have grep input + level filters (ERR/WRN/INF/DEBUG checkboxes) + a real `<pre>` log dump.
- Address bar `<input class="url" type="text" value="/wp-admin/edit.php?post_type=post" aria-label="Address">` — §2.1 item 2.
- Title is `contenteditable="true"` with a dashed outline — playground rename visible.
- Unsaved-Playground notice quoted: "Changes evaporate on refresh." + `SAVE LOCALLY [S]` button (the `[S]` indicates the keyboard shortcut).
- Footer "ACTIONS" block — README claims it's always visible. Export PR / Download .zip in that block.
- README is honest: "Density over discoverability: a new user has to read. ... Some users will find this hostile; that's the direction." Owns the trade-off.

## What's missing or weak
- **Brief violation: invented features.** The settings checkboxes at line 608–610 include "Persist OPFS between reloads", "Verbose PHP error log", "Enable WP_DEBUG". The §2.3 Settings inventory is WP version / PHP / Language / Network / Multisite. WP_DEBUG toggle and Verbose PHP error log and OPFS persistence are not in §2. Same overreach as #003 and #007.
- **Apply & Reset has no confirm step.** The destructive copy is in red but the button fires. Brutalist would invite a `<form>` requiring the user to type "RESET" into a field — that's on-metaphor and undelivered.
- **The 43-blueprint listing as a sortable table** (per README) is on-metaphor but loses thumbnail recognition entirely — README acknowledges as a deliberate cost.
- **Mobile collapses columns + hides FILES/BLUEPRINT split behind `[+] EXPAND`** — described, requires reading to find features. README admits.
- **The "ACTIONS footer always visible"** is great in theory but the document is long; "always visible" means sticky-footer, which the brutalist aesthetic might or might not allow. Verify.
- **Color is reserved for `--hot` destructive emphasis** — good restraint, but no other accent. Power users scanning for "where am I" rely entirely on capitalization + section numbers, which is intentional but slow.
- **The section structure (§1 toolbar / §2 figure / §3 settings / §4 start / §5 blueprints / §6 saved / §7 site manager)** is brilliant for IA but the linear top-to-bottom reading model means a power user scrolls a lot.
- **WP Admin / Homepage buttons** — exist in the toolbar or anchored in §2 figure caption. The brutalist aesthetic wants them as labeled `[BUTTONS]` not icons; verify they're surfaced clearly.
- **The trade-off "thumbnails would betray the aesthetic"** is the right call but leaves the 43-blueprint discovery harder than today's gallery of tiles.
- **The "Saved" table includes an "Unsaved Playground" row** (line 874) — fits the brief but the row is in the same table as saved sites; the distinction is in the row content (`boot src: blueprint #13`) which is subtle.

## How to push harder
- Remove the invented settings (WP_DEBUG, Verbose PHP error log, Persist OPFS) — they violate §5. Stick to the §2.3 five settings only; the brutalist constraint is "show everything plainly" not "extend the API".
- Render the Apply&Reset confirm as a brutalist `<form>` requiring the user to type the exact string "RESET" into an input before the submit button enables. That's the most brutalist destructive UX possible and it's the strongest §4 mitigation in the gallery.
