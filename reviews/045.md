# Review of #045 — WordPress Studio / Local-by-Flywheel-like

**Direction:** "WordPress Studio / Local-by-Flywheel-like: faithful desktop-app feel with environment cards, 'Open Site' / 'Stop' big primary buttons."

**Scores:** Feature completeness 9/10 · Direction 9/10 · IA 8/10 · Visual 9/10 · Originality 7/10

## What works
- Desktop-app metaphor is committed-to: macOS traffic lights, fixed top bar, sidebar source list with Running/Stopped pills, tabbed inspector (Overview / Settings / Files / Blueprint / Database / Logs), bottom status dock with PHP / WP / status / inline log toasts. Feels like a native app.
- All 6 §2.2 sources rendered in the "+ New environment" wizard surface (lines 1645–1675): Vanilla WordPress / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip — each with sub-line.
- Apply+Reset has a `.destructive-row` CSS class with explicit copy: "Apply Settings & Reset wipes all DB & file changes for this environment." (line 1262) with a red-bordered button. The destructive-row pattern appears in 3+ places (lines 1260, 1944) — consistent treatment.
- Address bar `<input value="/?page=shop" aria-label="Inner site path">` (line 1024) — §2.1 item 2 with aria-label.
- "Save site locally" huge primary button (line 1180) — most prominent save affordance in the gallery.
- Environment search `<input placeholder="Search environments… (⌘K)">` (line 1050) — keyboard shortcut visible.
- Blueprint search `<input placeholder="Search blueprints, plugins, themes…">` (line 1686) AND log search "filter by file, function, level" (line 1590) — multiple typeable filter inputs.
- The Running/Stopped pill metaphor is acknowledged in the README as "slightly fictional" (every WASM instance is technically running) but committed-to for clarity. Honest reframe.
- Database has implicit early-access; Logs has search + download-log button (line 1600); Blueprint has Run / Copy link / Download bundle (lines 1511–1513).
- The Additional Actions menu has "Export to GitHub PR" / "Download .zip" + ⌘⇧D shortcut (line 1256).
- Unsaved-Playground notice exists: "Settings differ from the saved version. Apply Settings & Reset will rebuild the site from scratch." (line 1178) — explicit about state change implications.

## What's missing or weak
- **Apply & Reset has no confirm step.** Both surfaces (lines 1263, 1947) fire immediately despite the destructive-row's strong copy. Local by Flywheel itself confirms; the metaphor invites it.
- **Settings appear in multiple surfaces** — the Overview tab's Settings section (line 1308) AND a separate Settings tab AND a popover. README claims they're "merged into a single Settings tab inside the Overview pane" — but the artifact has settings duplicated.
- **Overlap with #014 (macOS Settings) and #046 (Vercel/Netlify dashboard expected)** — the desktop-app sidebar-of-environments approach is similar to other gallery entries. Originality depends on the Running/Stopped distinction + "huge Save site" button.
- **The fixed top bar + status dock + tab strip eat vertical real estate** — README admits the iframe is smaller than edge-to-edge.
- **The iframe lives inside Overview, not the entire window** — same demotion as #014's "Live Preview" category. The brief's intent of "iframe dominates" is reinterpreted.
- **The PR / GitHub / URL launcher entries** are tile faces without inline paste inputs. §3 flows 11–12.
- **The Running/Stopped pill** is great metaphor but functionally maps to "iframe mounted vs not" — and the README admits "every Playground is technically 'running'." For a power user this distinction might cause confusion.
- **Mobile collapses sidebar to sheet and inspector to bottom-sheet stack** — described, "loses the simultaneous see-environments + see-site feel" admitted.
- **"Removed the bottom-left toast region in favor of a dock-style activity strip"** — that's a §2.4 reorganization, not a removal. Toasts now live at the bottom; PHP/WP version pills share the strip.
- **The "Open Site" big primary button promised by the direction** — verify whether the Open Site button is prominent vs. the more generic "Save site locally" button (which IS big but is for save, not for boot).
- **The destructive-row "Apply" button without "Reset" in the label** at line 1947 — labelled "Apply" only. A user might tap it expecting non-destructive apply.

## How to push harder
- Render the destructive Apply & Reset confirm as a native-feeling macOS sheet — dropped from the title bar with "Apply Settings and Reset Playground? This will wipe X posts, Y plugins, Z files." + a "Save first" safe-default button. Local by Flywheel actually does this pattern; copy it.
- Render the "Open Site" primary button the direction explicitly calls for, as the dominant action in the environment row. Right now Save dominates; in Local, Open Site is the biggest button per environment.
