# Review of #070 — Onboarding-driven

**Direction:** "Onboarding-driven: the first screen asks 'Are you a theme author / plugin author / site owner / learner?' and tailors the IA. Persists choice."

**Scores:** Feature completeness 9/10 · Direction 10/10 · IA 9/10 · Visual 9/10 · Originality 8/10

## What works
- Direction fully delivered: a first-run modal asks "Why are you here?" with four persona cards (Theme author / Plugin author / Site owner / Learner) + explicit "Skip — give me everything" escape (line 507). Persona persisted, surfaced as a coloured avatar pill in the toolbar, swappable any time.
- The persona doesn't gate features — it only re-ranks and re-defaults. README is explicit: "no feature is hidden behind it" + "Saved playgrounds are deliberately not persona-scoped." Smart constraint that protects the brief's no-removal rule.
- Per-persona color theming via CSS data attributes (lines 33–34): `--accent:#ff7ab6` for theme, `--accent:#7c5cff` for plugin — real visual change per persona.
- All 6 §2.2 sources rendered as nav items (lines 623–627) + tile cards (lines 775–801): Vanilla WordPress / WordPress PR / Gutenberg PR / From GitHub / Blueprint URL / Import .zip — with "Try a Gutenberg PR live" sub-line.
- Apply+Reset: `<button class="btnApply">Apply Settings & Reset Playground</button>` (line 1268) + "Heads-up: Apply wipes the running site. Save first if you care." (line 1270).
- Database has explicit early-access notice with GitHub-tracker link (line 1198): "**Early access.** Database management is an early access feature. Report bugs on the GitHub issue tracker."
- Address bar `<input value="wp-admin/plugins.php">` (line 582) — §2.1 item 2.
- Unsaved-Playground notice (line 587).
- "Recommended for you" with `<span class="pBadge">Plugin author</span>` (line 689) and "Sorted for Plugin authors first. Change persona to re-sort." (line 757) — persona-rationalization visible to user.
- Persona-aware defaults are surfaced inline per README: "PHP 8.3 is starred and Networking is pre-on" — users see *why* the defaults are what they are.
- README is honest: "Some power users will resent the modal and just hit 'Skip — give me everything', which I made an explicit, no-shame escape hatch (it falls through to 'Plugin author' because that surfaces the most tools)."
- "Persona-tailored IA can hide useful surface area from people who picked the 'wrong' persona — I mitigate this by never *removing* features, only re-ranking, and by surfacing the chosen persona prominently in the toolbar."

## What's missing or weak
- **Apply & Reset has no typed-confirm gate.** The destructive copy is good; the button fires.
- **Settings appear in two surfaces** — README admits "the tab and popover are explicitly marked as the *same* settings, with persona-aware defaults annotated inline." Two surfaces, but explicitly co-labeled.
- **The PR / GitHub / URL launcher tiles** have `<b>` titles + `<small>` sub-lines but no inline paste inputs visible.
- **Persona pill adds visual weight to an already busy toolbar** — README admits.
- **First-run modal is friction** — README admits and provides the Skip escape.
- **The "louder destructive warning per persona" claim** in the README isn't clearly delivered — verify whether Learner gets a louder Apply&Reset warning than Plugin Author.
- **Originality overlaps with #071 (Persona-tabbed top nav)** — both designs target the same 4-persona axis. #070's distinguishing move is the *first-run* modal + persona-aware defaults (vs. #071's persistent top-tab persona-switcher). #070 commits more to the onboarding metaphor.
- **The persona-aware defaults inline annotations** are described — verify whether they actually render (e.g., "★ PHP 8.3 (recommended for Plugin author)" inline).

## How to push harder
- Render the "louder destructive warning per persona" claim — Learner persona could see "This will erase EVERYTHING you've made. Are you sure?" with extra explanation; power-user personas could see a tighter "Wipes site." copy. Persona-aware safety is the strongest defensible UX of this direction.
- Implement the typed-confirm pattern for Apply & Reset — even better, gate it differently per persona (Learner gets a "type RESET" full gate; Plugin author gets a single-click confirm with a short warning).
