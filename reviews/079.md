# Review of #079 — Dual-iframe split

**Direction:** "Dual-iframe split: two playgrounds side-by-side for A/B comparison (e.g., WP 6.4 vs 7.0). Synchronized URL bar optional."

**Scores:** Feature completeness 9/10 · Direction 10/10 · IA 8/10 · Visual 9/10 · Originality 9/10

## What works
- Direction faithfully delivered: two iframes side-by-side with stacked mini-toolbars per pane, a center sync-lock rail, per-pane URLs ("WP 6.7" vs "WP 7.0 RC" titles at lines 1555 and 1723), per-pane Site Manager bottom sheets, unified launcher with "Load into A / Load into B / Load into both" choices.
- Apply+Reset has the strongest dual-pane treatment in the gallery:
  - Per-pane Apply & Reset: "**Apply & Reset Playground** rebuilds the SQLite DB. Your saved files persist." (line 1614) + `Apply & Reset` button
  - "**Apply & Reset both panes**" button (line 1618) — explicit dual-pane destructive action
  - Pop warn: "**Apply & Reset** will wipe Pane A's database. Save the playground first if you need to keep it." (line 1542)
- Address bar `<input value="2025/03/coffee-blend-launch/">` (line 1479, 1672) — §2.1 item 2 per pane.
- Pane B has `<div class="wp-admin-bar" style="background:#0a3622">` (line 1693) — distinct green tint to differentiate visually.
- Database has implicit early-access; Logs has search `<input placeholder="Search logs...">` (line 1743); Blueprint pane present.
- Playground rename per pane (lines 1555, 1723).
- Download .zip in additional-actions (line 1634).
- Booting state with disabled URL inputs (lines 1811, 1867).
- README is honest: "Duplicating the toolbar costs horizontal space, which on mobile means we collapse to a stacked single-pane view with a tiny 'Swap to B' pill (still functional, just not simultaneous)."
- "First-timers might not realize the panes are independent — so we lean on the lock icon, the divider color, and a brief tour toast on first open."
- "Allow pane B to be collapsed to a 32px 'ghost rail' until needed" — solo-user mitigation.

## What's missing or weak
- **Apply & Reset has no typed-confirm gate.** The dual-pane treatment is genuinely innovative, but the actual destructive click fires.
- **Per-pane Settings popover (line 1539)** + per-pane Settings tab (line 1608) — two surfaces per pane = 4 total settings surfaces with both panes open.
- **Toolbar duplication costs space** — README admits.
- **Mobile collapses to single-pane** — admitted loss of the comparison metaphor's whole point.
- **The "sync lock" in the center rail** — verify whether it's rendered as an interactive control with sync states (URL / scroll / settings sync toggles).
- **Originality is high** — overlaps with #066 (TweetDeck multi-column) in the "multiple playgrounds visible at once" lane, but #079's "explicit A/B comparison" with sync-lock is its own move.
- **The PR / GitHub / URL launcher tiles in the unified overlay** — verify whether they expose Load-into-A / Load-into-B / Load-into-both choices on the tile itself.
- **The "ghost rail" collapse of Pane B** is mentioned but verify whether the artifact shows the collapsed state.
- **No real demo of synchronized navigation** — the sync-lock claim is "URL, scroll, settings mirror across panes" — show it in a state.

## How to push harder
- Render the sync-lock interactive state — show "URL sync ON: typing /wp-admin in Pane A auto-types it in Pane B" with a small lock icon between the inputs. The comparison metaphor's strongest move is undelivered as a visible interaction.
- Implement a "diff highlight" between the two panes — when sync is on and the panes render differently, the visual difference is what users came for. A faint outline or comparison overlay would land the metaphor's whole point.
