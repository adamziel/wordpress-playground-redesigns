A note from the editor

The settings page is just one column.

The editorial idiom rejects the busy three-pane Settings tab familiar from configuration consoles. A newspaper section page is one long readable column. Each setting reads like a TOC entry. Each control sits to the right, mono-typeset and small, because the value is the news — not the chrome around it.

Five rail destinations means each pane is single-purpose. Site Manager is for runtime — anything that requires reset. Files, Blueprint, Database, and Logs are their own plates on the next pages.

"All runtime controls in one inline column — no tabs, because Sunday papers don't have tabs."

Apply Settings & Reset is the only red mark on the entire cream page. It demands a typed phrase, offers an auto-snapshot, and never fires without both an active phrase and an explicit click. The save-pill in the footer stays the single source of save-state truth.

A note from the editor

Files are a plain tree.

The Files pane is the smallest possible take on a code browser. Tree at left, editor at right, hairline divider between. Mono type in the editor, serif in the chrome — the same typographic discipline as the rest of the supplement.

The Blueprint pane (Plate IV-B, on the rail's third icon) shares this layout but pins blueprint.json as its open file and adds Run Blueprint, Copy link, and Download bundle actions in the footer. A sample appears below.

A note from the editor

The database is one file.

Everything you need to know about the inner WordPress DB sits in a single typeset block: engine, path, size, table list. The actions sit immediately below. Two outside admin tools are one click away — Adminer and phpMyAdmin — because some people prefer their queries spreadsheet-shaped.

The Logs pane is on the next plate; it shares the editorial idiom but has its own search and per-level small-caps tags.

Early-access notice — the database surface is the youngest of the five panes; please file rough edges on GitHub.

A note from the editor

Logs read like a blotter.

Every line is timestamped, given a small-caps level tag (OK · DEP · WARN · ERR), and typeset in proportional ink type. No black terminal field; the log is part of the paper.

Notice the deprecation, the missing include(), the timing on the 200, the WP-cron run — exactly the kind of mid-debug noise a developer needs to see while debugging a theme PR or testing a Blueprint URL.

Each rail destination — Sites, Site Manager, Files, Blueprint, Database, Logs — gets a full pane like this. No two tools are ever shown side by side. Reading one thing at a time is the whole thesis.

A note from the editor

The same idiom, folded.

On mobile the rail rotates ninety degrees to a 78-px bottom tab bar — same five docks, same numbered eyebrows. The pane is replaced by a bottom sheet at sixty-two percent of the viewport, capped by the same gold rule. The corner "Now Reading" stamp shrinks to a single capsule above the tab bar.

The Library overlay (Plate VII) becomes a full-screen sheet on mobile, with the category index collapsed into a horizontal scroll-strip at the top of the sheet — the editorial idiom is preserved, only the geometry changes.

  • iTab bar — five rotated docks (Sites · Manager · Blueprint · DB · Logs); active state marked by the same gold 2-px rule.
  • iiBottom sheet — the swap-content pane, dragged up to 75 % when reading the Library.
  • iiiCapsule capsule — refresh + address + save-rosette + fullscreen, stacked at the top under the status bar.
  • ivStamp — site name + state + dot, parked above the rail.
