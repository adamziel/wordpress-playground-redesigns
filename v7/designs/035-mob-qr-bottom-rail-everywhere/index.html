v7 #035Bottom-rail everywhere — MOB + QR

A mobile-first quiet rail that does not rotate when the viewport grows. The rail anchors to the bottom edge of the viewport at every width: a 5-dock tab bar at 1440×900 and a 5-dock tab bar at 390×844. The "loud" surface is a single bottom sheet rising from the rail at three snap detents (peek 40 % · mid 70 % · full 92 %). The iframe is always the wallpaper above the rail. Same gestures — swipe-up-from-rail to expand, swipe-down to dismiss — work at both widths. This is the mobile-first thesis taken literally: the desktop borrows mobile's anchor, not the other way around.

Why the rail does not rotate

v5 #018's quiet-rail-to-loud IA was originally a left-edge primitive. Sibling design #015 rotates that rail to a bottom tab bar on mobile. That works — but it means mobile inherits the rail's mobile form, not that desktop inherits mobile's. The MOB principle is "desktop = wider phone." Here I take the opposite move: the rail stays at the bottom on desktop too. The desktop literally is a wider phone — same gestures (swipe-up-from-rail to expand the sheet, swipe-down-to-dismiss), same anchor, same shortcuts. The composition with QR is therefore QR's swap-by-dock semantics (the rail picks which content the single secondary surface shows) on a bottom anchor.

Shared CSS — proof of parity

.sheet, .m-sheet         { /* same surface · different parent width */ }
.sheet.peek, .m-sheet.peek   { height: 38% / 36%; }
.sheet.mid,  .m-sheet.mid    { height: 62% / 64%; }
.sheet.full, .m-sheet.full   { height: 92% / 88%; }
.det, .d                 { /* same dot — desktop seam + mobile grab-handle */ }
.det.on, .d.on           { /* same active state token */ }
.rail, .m-rail           { /* same 5-dock layout, same anchor */ }

The desktop seam gauge (.det-gauge) and the mobile grab-handle dot row (.det-dots) are the same three-position vocabulary. Three detents only — no momentum gauge, no auto-grow, no 35/65/50/50/60/40 menagerie. This is QR snap-detent, not SR.
