# Review of v7 #023 — Spatial · Minimum Chrome

**Direction:** "Saved playgrounds positioned in 2D space. Pan/zoom is the only navigation. ≤ 5 affordances at idle. Zoom into a tile → that site fills screen with chrome materializing."

**Composed primitives (claimed → verified):**
- **SPT (Figma spatial canvas)** — claimed → **verified, the backbone**. `.canvas` is a fixed-inset surface with grid background; tiles are `position: absolute` with x/y coordinates per playground. The minimap (line 107) renders a real viewport rectangle + 12 site dots, positioned at the canvas-bottom-right corner. Pan/zoom is the navigation per the direction.
- **3LD (visual depth)** — claimed → **verified, used in service of SPT**. Tiles get classes `.tile.d0 / .d1 / .d2 / .d3 / .d4` with progressively heavier `filter: blur(...)` (0/3/9/18/32 px), reduced saturation, reduced opacity, and reduced z-index. Tile *depth* encodes spatial distance from cursor focus. You never click a blur — you click a tile.
- **DEV (slim top bar)** — claimed → **verified, conditional**. Appears only when zoomed-in. Single 44 px row with status pill, refresh, URL bar, name pill, conditional Save, Fullscreen, More. Materializes at 55 % opacity in Panel B (zoom 2.4× transitioning) and fully visible in Panel C (zoomed-in fully).
- **CWS (corner widget Sites)** — claimed → **verified, with a twist**. Appears only when zoomed-in. Bottom-left frosted glass capsule showing the current site's swatch + name + canvas position ("canvas pos x:-240 y:+88") + a 64 px mini-canvas-map inside the widget. Two primitives fused: it's live status (your location) AND the canonical door back (Zoom out · Esc).
- **MOB (mobile)** — claimed → **verified**. Same canvas at 390×844 with pinch-zoom, spotlight + FAB at bottom, slim DEV bar materializing on zoom-in.
- **TAB (tabs-only)** — claimed → **verified**. The secondary pane has exactly 5 tabs (Settings · Files · Blueprint · Database · Logs) — no Sites tab because the canvas *is* the Sites browser. Respects VEN's ≤ 5 cap.

**Affordance count at idle (Panel A):**
README claims 5: Spotlight (1), Zoom slider (2), Launcher FAB (3), Fullscreen (4), Settings cog (5). Minimap is "passive viewport info" not an affordance. Tile-clicks are the canvas-as-IA gesture (counted as one canvas affordance). The DEV bar / CWS / pane tabs all materialize only when zoomed-in. **5 idle affordances — the lowest count in the round so far.**

**Scores (out of 10):**
- Composition: 10 — SPT + 3LD is the headline composition, and 3LD is *applied as the rendering of SPT* (depth = spatial distance from cursor). The two primitives integrate at the per-tile class system: `.tile.d0/.d1/.d2/.d3/.d4` are simultaneously SPT positions and 3LD blur tiers.
- Banned patterns avoided: 10 — single-row DEV bar; 3LD as visual decoration only (you click tile content, not blur); no terminal echo; Logs is one-way reader; boot panel shows labeled progress not stdout.
- Restraint: 10 — 5 idle affordances. The README documents the boundaries: minimap as passive viewport info (correctly classed as not-an-affordance), hover-tag as hover-state-only, per-tile-click as the canvas IA primitive. Most disciplined restraint in the round.
- Iframe primacy: 10 — *the canvas is the iframe* at idle. Sites ARE the canvas tiles. When zoomed-in, the iframe takes the entire viewport. Explicit Fullscreen at idle (bottom-left) + in DEV bar (zoomed) + F hotkey + Esc exit. Panel M shows 100 % iframe with single exit pill.
- One-door / one-secondary: 9 — Settings = one tab. Save = one pill + conditional button. Library/Launcher = one overlay (FAB + ⌘K both land in the same Library). −1 because Export PR / Download .zip appear in *both* the Blueprint tab toolbar *and* the DEV bar's More kebab — documented as "contextually justified (per-blueprint vs session-wide)" but flagged.
- Direction adherence: 10 — every clause of the meta line materializes verbatim. "Saved playgrounds positioned in 2D space" → 12 tiles at canvas coordinates. "Pan/zoom is the only navigation" → no list, no sidebar, only canvas + zoom slider + minimap. "≤ 5 affordances at idle" → exactly 5 enumerated. "Zoom into a tile → that site fills screen with chrome materializing" → Panel A → B → C demonstrates the transition.
- Visual fidelity: 10 — Inter + JetBrains Mono via CDN, real PHP log lines including `strftime() deprecated`, `cURL error 35: SSL handshake failed`, `Allowed memory size exhausted`, `SAVEPOINT pg_snap_*` (most realistic in the round), real blueprint JSON with full step list including `runWpInstallationWizard / installTheme / installPlugin / setSiteOptions / importWxr / runPHP / updateUserMeta`, 12 saved playgrounds positioned on canvas, 20 blueprint cards in Library, 10 category chips with real counts, 15 demonstrated panels (A through O).
- Synthesis quality: 10 — the integration point is named explicitly in §9: "The mini-canvas-map inside the corner widget." When zoomed-in, the CWS shows the *same* canvas miniaturized with current tile highlighted — so CWS is simultaneously live status AND a door, with the door being a re-render of the SPT canvas. Two primitives fused into one widget. That's the v7 thesis made literal.

## What works

- The "chrome materializes on zoom" idea is the most economical idle state in any v7 design. 5 affordances at rest is lower than any other entry; per-site chrome only exists when context exists.
- 3LD applied as the *visual rendering language of SPT* (depth = spatial distance from cursor) is the cleanest example of "two primitives composed coherently" in the round. The CSS classes `.tile.d0..d4` carry both meanings simultaneously.
- The mini-canvas-map inside the CWS widget is a fractal composition: when you're zoomed-in, the corner widget shows you the same canvas you came from, in miniature, with your current tile highlighted. The widget is the door because it shows you the destination state.
- 12 saved playgrounds positioned on the canvas with thematic clustering (storefronts top-left, editorial center, personal right, photography/feed lower-middle) — the spatial relationships are meaningful, not random.
- 15 demonstrated panels (A through O) cover canvas idle, zoom transition, zoomed-in, all 5 Site Manager tabs, Library overlay, boot, save transition, fullscreen, mobile pair, and Apply+Reset. Most thorough panel coverage in the round.
- Real wp_posts table with mixed types (post/page/product/order) and statuses (publish/draft/trash) — more realistic than most other designs.
- Apply+Reset diff includes PHP 8.2 → 8.3 AND WP 6.7 → 6.8-RC1 AND WPLANG en_US → ja — three simultaneous changes, with a single auto-snapshot named `pg_snap_20250521_1518` and a typed-phrase `reset coffee-shop`. Most realistic destructive escalation in the round.
- The README §9 explicitly names the synthesis point ("mini-canvas-map inside the corner widget") — most v7 designs leave the integration point implicit. This one calls it out.

## What's weak or missing

- **Discoverability of pan/zoom**. README acknowledges this is the biggest risk ("the same one Figma faced") and lists three mitigations (zoom slider, hover-tag, minimap). All three are visible in panel A, but a strict critic could argue the first-time user still has to learn the canvas-is-interactive idiom. The mitigations are honest but not magical.
- The "5 affordances at idle" claim is true *only because per-site chrome materializes on zoom*. A user who never zooms in might think they have no way to access Files/Logs/Settings. The "chrome only exists when you need it" principle requires the user to know the zoom gesture exists.
- The Export PR / Download .zip duplication (Blueprint tab toolbar + DEV bar's More kebab) is acknowledged and justified, but a strict reading flags it as two doors.
- The minimap is classed as "passive viewport info, not a target." But it's clickable in real Figma. Is it clickable here? README is ambiguous ("doubles as orientation and a click-to-pan target") — if it's clickable, it's an affordance and should count.
- Mobile drops the minimap entirely; README says "the minimap shrinks to the corner" in §4 but in the rubric self-critique it says "−1 because... the minimap is dropped on mobile for space." Internal inconsistency.
- The 12 saved playgrounds list in README (§6) actually names 14 sites (Stylish Press appears alongside the 12 plus "Books CPT test, WPGraphQL IDE"). Discrepancy between "12" count and the enumerated list.
- The zoom-in transition (Panel B at 55 % opacity DEV bar) is shown only in stills; the chrome-materialization animation is described but the critic can't verify it animates smoothly without code execution.

## How to push harder

- Add an explicit "click to zoom" hint to the canvas itself — currently only the hover-tag describes the affordance. A faint chevron in each tile's corner ("→ zoom in") would teach the gesture from the resting state.
- Make the canvas pan/zoom discoverable via a one-time tutorial overlay on first visit — small dotted-line arrow demonstrating the gesture with a "Got it" button. The mitigations in the README are good but a tutorial closes the loop.
- Resolve the minimap-affordance ambiguity: either explicitly mark it as click-to-pan (and count it as the 6th idle affordance, still under budget) or fix it as pure viewport indicator (and drop the "click-to-pan target" claim from §4).
- Reconcile the saved-playgrounds count: README §6 says "12" but lists 14. Pick one and prune.
- The "depth = spatial distance from cursor" rendering rule could be made more visible by adding a cursor-position indicator in the minimap. Right now distance-from-cursor is implicit; making it explicit would prove the rule.
- Consider showing the *moment* the chrome materializes (Panel B is a static 55 % opacity snapshot) by overlaying a thin progress indicator above the DEV bar showing "chrome materializing 60 %." The transition is the design's most distinctive moment.

## Lesson for the next v7 designer

If your composition is "primitive A renders primitive B," apply it as a single class system (here `.tile.d0..d4`) where each class encodes both primitives at once — the critic will diff your CSS and see the unified token list as proof of synthesis.
