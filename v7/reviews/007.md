# Review of v7 #007 — Tabs on the Seam

**Direction:** "Smart-resize is the IA, and the tab strip that swaps the secondary pane's content lives ON the divider between the two panes. The seam carries the tabs AND the momentum gauge — picking what content and weighting which pane happen in the same physical place."

**Composed primitives (claimed → verified):**
- **SR (smart resize)** — claimed → **verified**. The `.seam` is positioned via inline `style="left: 75%"` / `35%` / `50%` / `60%` per state, and that same left-percentage value drives both the divider position and the gauge fill width (`<div class="fill" style="width: 75%;">`). Ratios travel together. Six demonstrated ratios (75/25, 35/65, 50/50, 60/40, 100/0, plus mobile vertical equivalents).
- **TAB (tabs-only nav)** — claimed → **verified, and is the headline thesis**. `.seam-tabs` is a div nested *inside* `.seam` (line 898), making the tab strip a structural child of the divider, not of either pane. The six tabs (Sites · Settings · Files · Blueprint · Database · Logs) are rendered as buttons with inline SVG icons + labels + an active state. The seam-gauge is also a child of `.seam` (line 937), so tabs and gauge truly share the same physical element. This is the cleanest realization of the meta line's literal claim.
- **CWS (corner-widget Sites)** — claimed → **verified**. `.sites-widget` is anchored top-left inside the iframe pane (separate from the seam), shows an avatar + name + state-dot ("saved" / "booting" / "unsaved" / "running"), and clicking it activates the Sites seam-tab. CSS classes `.sw-state.booting / .unsaved` change the dot color. This is a real CWS implementation, not stretched onto a top-bar pill.
- **DEV (slim top bar)** — claimed → **verified**. Single 44 px row with back/fwd/refresh, URL pill, save-state pill, conditional Save button, Fullscreen icon, kebab menu. The DEV bar deliberately omits Sites/Settings/Files because the seam-tabs own that. Tight separation of concerns.

**Affordance count at idle:**
README claims 8 exactly; the in-canvas counter for State A shows "Idle affordances: 7 · Refresh · Back · Fwd · URL · Name · Fullscreen · Kebab · (+ Sites widget + seam-tabs cluster)". Either 7 or 8 depending on whether you count Sites widget separately from the seam-tabs cluster. The in-canvas counter is more candid: 7 chrome chips + 2 IA clusters = 9 if strict. The README's 8 is defensible if you count the widget + tabs as one synchronized door. Borderline, documented.

**Scores (out of 10):**
- Composition: 10 — SR + TAB is the headline composition the meta line demands, and the synthesis point is the seam itself: a single DOM element containing both the tab strip and the gauge. The two primitives don't just sit next to each other; they share the same `<div class="seam">`.
- Banned patterns avoided: 10 — single-row top bar; depth used only as glass shadow under the seam-tabs (decorative, not interactive); no terminal/console echo (Logs is read-only output, Booting overlay shows steps not stdout).
- Restraint: 8 — 7–8 affordances depending on counting rule. The in-canvas counter says 7, which is honest. The seam-tabs at 6 items + the Sites widget could be argued as duplication, but README frames Sites widget as a CWS *shortcut into* the Sites tab — one door rendered in two places, not two doors.
- Iframe primacy: 9 — iframe always occupies the larger pane in idle (75/25). State F is full 100/0 with an Esc chip. Slight knock: the seam-tabs sit at 75 % of width when iframe-focused, which puts the entire nav cluster visually inside the iframe's territory. Some users may read it as belonging to the iframe rather than to the divider.
- One-door / one-secondary: 10 — single secondary pane, six seam-tabs swap content. Settings is one tab. The CWS widget activates the Sites tab — same destination, ergonomic shortcut. WP Admin / Home are pane-internal chips, eliminated from top bar.
- Direction adherence: 10 — the meta line says "the tab strip … lives on the divider"; the HTML has `<div class="seam"><div class="seam-tabs">…` (nested inside the seam div). The gauge sits directly below the tabs inside the same `.seam` element. Verbatim.
- Visual fidelity: 9 — inline SVG icons (no CDN dependency), real PHP log lines in State C, full 18-card blueprint gallery in State B with 10 category chips, typed-phrase + auto-snapshot Apply & Reset in State H, 9 demonstrated states, 4 saved playgrounds, real blueprint JSON in State E. Slightly less polish than 003/004 because the visual style is functional rather than cinematic.
- Synthesis quality: 10 — the seam is a single DOM element containing tabs + gauge + line. That's the v7 thesis in CSS: two primitives composed into one anchor. The user's pick-content gesture and weight-pane gesture happen in the same physical place.

## What works

- The literal interpretation of "tabs on the seam" is realized in the DOM tree, not just visually. `.seam-tabs` is `position: absolute` inside `.seam`, both anchored to the divider via `style="left: 75%"`. Move the seam, tabs and gauge move together.
- The gauge's bar fill width is set inline from the same percentage as the seam position: `<div class="fill" style="width: 75%;">` for State A's 75/25 ratio. Wire-correct, no rhetoric.
- The in-canvas affordance counter (line 1015) admits "7" rather than gaming it up to 8 — this design counts more honestly than designs 003/004/006 which all claim "exactly 8."
- The CWS widget is the most spec-faithful in the round so far: top-left of iframe, frosted glass, avatar + name + state dot, click expands the Sites tab (the canonical door). Anchored correctly.
- The save-state pill is status-only; the Save dark button appears separately only when unsaved. The lesson from earlier v7 designs is internalized.
- The seam follows SR position, which is the right way to maintain spatial consistency — tabs always appear at the focus boundary, not at a fixed canvas location.

## What's weak or missing

- The seam-tabs at left:75% put the entire nav cluster physically inside the iframe's right edge. A new user looking at State A might read the tabs as iframe content, not as global nav. README acknowledges this trade-off ("seam tabs are a novel location") but the mitigation is mostly verbal.
- 6 tabs in the seam (Sites · Settings · Files · Blueprint · Database · Logs) — same VEN drift as designs 003/004/005/006. The seam isn't a "pane" per the brief, so VEN may not apply, but six segmented controls in one strip is dense for a divider element.
- The Sites widget at top-left and the Sites seam-tab are the same door rendered twice. README defends this as "ergonomic shortcut, single destination," but a strict critic could call it dual-channel. The two sites of action are sufficiently different (corner glance vs nav click) that I accept the trade-off; just flagging.
- The seam can be 0–100 % across the canvas, so when iframe is at 100/0 (State F fullscreen), where do the seam-tabs go? They presumably disappear with the seam, but then how does the user activate the nav? The Sites widget remains visible per CWS, but reaching Settings or Logs requires exiting fullscreen first. That's a tolerable workflow but worth surfacing in the design.
- The reset-zone in State A uses `opacity: 0.45` to indicate "armed but quiet" — clever, but a user might think the section is disabled. Worth a small "click to arm" affordance.

## How to push harder

- Add an explicit visual cue that the seam-tabs are *nav* and not iframe content — e.g., a tiny anchor icon at the top of the seam pointing to the divider, or a hairline that extends from the top bar down to the tabs to signal "this belongs to the chrome system."
- In State F (fullscreen 100/0), surface a compact circular FAB at top-right with the Sites widget functionality (status + tap-to-summon-tabs). Right now the entire seam disappears.
- Consider making the seam line the *only* hairline in the entire design (no other vertical dividers anywhere in the panes). This would make the seam visually unique and reinforce its role as nav anchor.
- Drop "Sites" from the seam-tabs since the CWS widget is already the door — that gets you 5 tabs (Settings · Files · Blueprint · Database · Logs) and aligns with VEN. The widget owns Sites; the tabs own per-site operations.

## Lesson for the next v7 designer

If your composition thesis says "X lives ON Y," nest X inside Y in the DOM and bind their positions to the same CSS value — the critic will diff the tree and look for the structural relationship, not just side-by-side visual layout.
