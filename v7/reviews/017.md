# Review of v7 #017 — Four Corners · Sharp Roles

**Direction:** "All persistent chrome lives in four corner widgets, each with a sharp non-overlapping role and a live-status mirror. Top-left = Sites. Top-right = Settings + fullscreen. Bottom-left = Logs. Bottom-right = Site Manager. Each widget is its sole door and expands inward from its own corner into a single glass panel anchored to that corner. No top bar. No rail. No drawer."

**Composed primitives (claimed → verified):**
- **CWS (corner-widget Sites interaction) — used as the backbone, not as a side garnish** — claimed → **verified, the most distinctive composition in the round so far**. Four `.cw.tl / .cw.tr / .cw.bl / .cw.br` widgets, each with its own role label, live-status content, and accent color (violet TL / teal TR / amber BL / WP-blue BR). When expanded, `.panel-card.tl-open` / `.tr-open` / `.bl-open` / `.br-open` anchor to the respective corner with `top/bottom + left/right` positioning, and a `.anchor-pin.tl / .tr / .bl / .br` glyph marks the origin (line 215-218 CSS). This is the canonical CWS pattern, multiplied by four.
- **3LD (visual depth)** — claimed → **verified**. CSS defines `--z0 0px` (iframe), `--z1 10px` (idle widget), `--z2 20px` (expanded panel), `--z3 40px` (library scrim). Applied via `backdrop-filter` on widgets/panels. Depth is purely decorative; the user navigates by clicking the widget, not the blur.
- **GLS (glassmorphism)** — claimed → **verified**. All four widgets use frosted glass with per-corner accent seam (`.panel-card::before` background gradient per corner color). Iframe is the only solid surface. Glass is in service of depth/identity, not ornament.
- **DEV** — claimed → **verified, as a *single floating capsule* not a full bar**. A 36 px pill-shaped `.addrcap` centered above the iframe holds refresh, address, save-pill. The README is explicit: "Not a top bar. Centered. Floating." This is the most disciplined DEV interpretation in the round — DEV-as-pill rather than DEV-as-bar.
- **VEN (≤ 5 items per pane)** — claimed → **verified**. TL Sites has 3 segments. BR Site Manager has 5 tabs. BL Logs has 4 level chips + tail. Library overlay has 10 categories.
- **MOB** — claimed → **verified, with corner-to-dock collapse**. The four corners collapse on mobile to a single top capsule + a 4-cell bottom dock. Same four roles preserved.

**Affordance count at idle (Panel A):**
README claims exactly 8: refresh, address bar, save-pill, fullscreen, settings gear, TL widget, BL widget, BR widget. On-canvas annotation in panel A counts these 8 + acknowledges that TR widget contains *two* affordances (gear + fullscreen) sharing one container. The WP/PHP pill in TR is explicitly classed as status, not affordance. Strict count = 8 — at the budget ceiling, documented.

**Scores (out of 10):**
- Composition: 10 — CWS applied four times with sharp non-overlapping roles is the most original v7 composition in the round so far. The other CWS designs use CWS as a side-garnish; this one makes it the whole IA.
- Banned patterns avoided: 10 — no top bar (only a floating capsule); depth used as visual hierarchy only; no terminal echo. The README explicitly notes the BL Logs count and the panel render are *one channel of data through two surfaces*, not a duplicate UI mirror.
- Restraint: 8 — exactly 8 affordances at the ceiling. The README is candid: "If I had inlined fullscreen into the address capsule the count would drop to 7, but at the cost of mixing fullscreen (escape-hatch verb) into the address bar (navigation surface)." Defensible reasoning.
- Iframe primacy: 10 — the iframe gets the entire viewport between the four corner widgets (about 96 % of pixels). It's the only sharp z0 surface. Fullscreen button is explicit and labeled in TR. The four corners *together* spend less screen real estate than a typical sidebar.
- One-door / one-secondary: 9 — each corner owns one content domain, sole door. Only one panel can be expanded at a time (closing one before opening another is the rule). Library is a modal scrim at z3, mutually exclusive with corner panels. Slight knock because the BR widget has 3 tab chips (Files/Blueprint/DB) which could read as 3 doors but the README clarifies they all reach the same destination with focus-mirror as the difference.
- Direction adherence: 10 — every clause of meta.json materializes: four corners (yes, `.cw.tl/.tr/.bl/.br`), sharp non-overlapping roles (yes, each has a distinct content domain), live-status mirrors (yes — TL shows site name + state, TR shows WP/PHP versions, BL shows error count, BR shows current Files focus), expand inward from own corner (yes, anchor pins mark the origin), no top bar (yes — only the floating capsule), no rail (yes), no drawer (yes).
- Visual fidelity: 9 — Inter + JetBrains Mono via CDN, inline SVG icons (no CDN dependency for icons), real PHP log lines including strftime deprecation + include() warning + savepoints + request timing, real blueprint JSON with correct verbs (`runWpInstallationWizard`, `installPlugin`, `importWxr`), 20 library cards, 4 saved playgrounds, 8 desktop panels (A through H + I, K) + 2 mobile panels (J, L), per-corner accent gradient seams on expanded panels.
- Synthesis quality: 10 — CWS-applied-four-times is the synthesis itself. The four widgets share a common `.cw` base class with per-corner modifiers (`.tl/.tr/.bl/.br`), and the expanded panels share `.panel-card` with corner-specific positioning + accent gradients. Same primitive multiplied four times with disciplined non-overlap.

## What works

- The four-corner thesis is the most architecturally distinctive composition in the round. Other designs orbit a rail (010-015) or a seam (003-007); this one rejects both and uses the corners themselves as the IA primitive.
- Each corner has a unique color seam (violet/teal/amber/WP-blue) that runs along the top or bottom edge of its expanded panel. The accent color is the corner's identity, so the user knows where the panel came from even after expansion.
- The anchor pin (`.anchor-pin.tl/.tr/.bl/.br` with ☕/⚙/📋/📁 glyphs at -14px corner offset) is a clever spatial cue — it sits at the corner where the widget was, marking the panel's origin like a sticky note pin.
- The "TR has two affordances in one container" honest disclosure is sharper than most other designs' fudging. The README admits Fullscreen + Settings could share but counts them separately because they're distinct hover targets.
- BL Logs widget's count "breathes when it changes" — that's a real on-canvas animation cue that connects the idle status to the underlying state. Most designs use a static count.
- The address capsule is centered and floats, not anchored. README explicitly calls it "not a top bar" — DEV interpreted as the minimum browser address chrome, not as a status bar.
- 4 saved playgrounds with names that span states (Coffee Shop running, Vueo theme dev, Gaming News booting, Unsaved Playground) — the full unsaved → saved transition is demonstrated across panels A/C/D/F/H.
- The Apply+Reset modal (panel E) uses typed-phrase + visible diff + auto-snapshot with 7-day restorability — the most generous safety window in the round.
- Sub-tab chips inside BR widget (Files/Blueprint/DB) double as focus indicator + door — but all chips lead to the same expansion destination with the focus state determining which tab opens active. The README is candid that this is one door with three highlight states, not three doors.

## What's weak or missing

- The four-corner approach commits substantial corner real estate (4 % of viewport). At smaller desktop widths the corners could collide. The README mentions corner sizes (TL ≈ 280×60, BR ≈ 200×60) without disclosing the minimum viewport width where corners stop fitting.
- On mobile, the four corners collapse to a single top capsule + a 4-cell bottom dock. The README admits "some loss of 'corner-ness' by physical necessity at 390 px wide." The mobile compromise loses the design's distinguishing feature.
- The TR widget mixes Settings (a long-form configuration door) with Fullscreen (an instant verb). They have different lifespans — Settings opens a panel, Fullscreen toggles an entire mode. Pairing them in one container is a useful efficiency but blurs the "sharp non-overlapping role" claim.
- The BR widget's three tab chips (Files/Blueprint/DB) make the user pre-select which Site Manager tab they want before opening the widget — that's the *opposite* of v5 #036's CWS pattern (which expands to a default state with content swap inside). Whether this is good or bad depends on whether users typically know which tab they want; the design assumes they do.
- The save-pill lives in the floating address capsule (top center), and the TL Sites widget also shows site state ("Coffee Shop · Running"). Two surfaces showing site state — README justifies this as one source of truth (the save-pill is action, TL is identity), but a strict critic could call them duplicate status displays.
- The Logs tab is reachable from both BL widget AND BR Site Manager → Logs sub-list (referenced in panel C/D notes). README disclosure: "same destination" but the user has two visible doors.

## How to push harder

- Disclose the minimum desktop viewport width where the four corners stop fitting. If it's > 1280 px, document a "compressed mode" that collapses pairs of corners.
- Resolve the TR widget's mixed roles by separating Fullscreen into its own micro-affordance (a circular F button anchored elsewhere) and leaving Settings as the TR widget's sole content. That sharpens the "non-overlapping role" claim.
- Either drop the BR sub-tab chips and let the widget open to a default Site Manager state with internal tabs, OR rename them to "focus indicators" (currently they read as buttons because of the dot-on-Files highlight). The README is candid about the design intent but the visual reads as 3 doors.
- Demonstrate the panel-grows-inward animation by showing a mid-state in one panel: the widget at 80 % expansion, with the anchor pin still attached and the panel content just starting to render. Right now every panel is either idle or fully expanded.
- For mobile, consider keeping the four corners visible at 390 px wide as smaller chips (say, 80×40 each) rather than collapsing entirely. The "four corners" thesis is too distinctive to abandon at mobile.

## Lesson for the next v7 designer

If your composition is "CWS multiplied N times with sharp roles," use distinct accent colors AND distinct anchor positions in the DOM — the critic will look for both the visual signal (color seam) and the structural signal (per-corner positioning class) to verify the roles are non-overlapping.
