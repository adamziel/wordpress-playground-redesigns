# Review of v7 #035 — Bottom-rail everywhere · MOB + QR

**Direction:** Mobile-first QR — the rail anchors to the *bottom* at every width. The desktop literally borrows mobile's anchor: a 5-dock bottom tab bar on both 1440×900 and 390×844, with one secondary sheet rising from the rail at three snap detents (peek/mid/full).

**Composed primitives (claimed → verified):**
- **MOB** — claimed → **verified, and load-bearing**. The desktop's `.rail` and the mobile's `.m-rail` share the same 5-dock layout, the same Sites/Files/Blueprint/Database/Logs identities, the same active state, and the same bottom anchor. Frame R explicitly documents the shared CSS class system: `.sheet/.m-sheet`, `.det/.d`, `.rail/.m-rail`. This is genuine parity, not aspirational claim.
- **QR (quiet-rail-to-loud IA)** — claimed → **verified**. Five docks (Sites/Files/Blueprint/Database/Logs), silent at idle (icons + labels, badge only when Logs has errors). Tapping a dock opens the single `.sheet` (or `.m-sheet`) at one of three detents (peek 38% / mid 62% / full 92%). One secondary surface, swapped by dock selection. The Sites dock contains a 3-segment sub-control (Yours/New/Library) — that's the only sub-IA, and it lives *inside* the sheet, not as a second pane.
- **CWS** — claimed → **verified**. `.cws` widget at bottom-right of the iframe with `pip` + name + meta (`WP 6.7 · PHP 8.2 · 11.4 MB`) + up-chevron. Persistent across Frames A/M; fades in Frames B/M when the sheet covers it. Mobile equivalent `.m-cws` sits above the rail in Frames N/Q. State carried in pip color (green/blue/red).
- **DEV (slim top capsule)** — claimed → **verified**. Single 44-px row with Refresh + URL bar (lock + `https://` + domain + path) + title chip + save-pill (Saved/Saving/Booting/Unsaved variants) + ⋮ More + `⛶ Fullscreen F`. One row, no second row. Mobile compresses to 42 px in Frames N–Q.
- **VEN (≤ 5 per pane)** — claimed → **verified**. Rail = 5 docks. SM segment row = 5 tabs. Sites sub-segment = 3. Library categories = 10 (matches brief). Each individual surface is at or under cap.

**Affordance count at idle (Frame A):** README claims 8; legend strip in the HTML enumerates them: 1 Refresh · 2 URL · 3 Save · 4 ⋮ · 5 ⛶ · 6 Rail · 7 CWS · 8 Peek-strip = **8/8**. The peek-strip is the marginal counted item — README admits this honestly.

**Scores (out of 10):**
- Composition: 10 — MOB and QR genuinely fuse: the QR rail is repositioned by the MOB principle, and the result is a single rail anchor at every width. Frame R provides the proof of parity at the level of *shared CSS class names*, not just the visual layout. The thesis is "desktop = wider phone" taken literally and implemented at the class system level.
- Banned patterns avoided: 10 — single 44 px capsule; no depth-as-control (blur appears only on the iframe behind the sheet, decorative); no terminal echo (PHP logs are one channel — Frame G).
- Restraint: 9 — exactly 8 affordances at the ceiling. The peek-strip is the marginal item; honestly flagged. No hidden affordance creep.
- Iframe primacy: 10 — iframe is the only sharp z-0 wallpaper in every frame. Fullscreen has one canonical anchor (the `⛶ F` button in the top capsule) + `F` shortcut. Frame L renders fullscreen with all chrome retracted, only a floating Esc/F banner + ghosted CWS surviving.
- One-door / one-secondary: 9 — Sites = rail dock 1 + CWS widget (acknowledged duplicate, same destination); Settings = only Files dock's Settings sub-tab; Library = only Sites dock's Library sub-segment; Export PR / Download .zip = only ⋮ More menu (Frame M); Save = save-pill + acknowledged inline "Save locally" on the Unsaved row (one duplicate, named in README §3). The pane swap is the single secondary surface.
- Direction adherence: 10 — the literal direction is "bottom rail everywhere" and *every* desktop frame (A–M) shows the rail at the bottom; *every* mobile frame (N–Q) shows the same rail at the bottom. Frame R closes the loop with a parity table. This is the strongest direction-to-execution mapping in v7 to date.
- Visual fidelity: 9 — Nordly Roasters Coffee site mock with real-looking storefront, real PHP log lines (`include()` failure, `strftime()` deprecation, REST 504, SAVEPOINT, cron tick), real blueprint JSON verbs (`installPlugin`, `installTheme`, `runPHP`, `importWxr`, `setSiteOptions`, `login`), Apply+Reset modal with actual diff syntax (---/+++/-/+) and a named snapshot file (`coffee-shop-2026-05-21T18-42.snap`), 5 saved playgrounds with state variety (running/booting/stopped/unsaved). Inter + JetBrains Mono via CDN.
- Synthesis quality: 10 — Frame R's CSS-shared-classes diagram is the strongest synthesis proof in v7. The designer didn't just claim parity in prose; they wrote a single class system that both viewports inherit. The peek/mid/full vocabulary is mapped numerically across widths. The decision to *not* compose SR (smart-resize) is explicitly justified in README §4 as protecting the parity thesis from dilution — a confident negative composition decision.

## What works

- **Frame R is the most rigorous parity proof in v7 so far.** Most designs claim "same CSS class names on both viewports" in prose; this one shows a CSS snippet (`.sheet.peek, .m-sheet.peek { height: 38% / 36%; }`) and a detent-mapping table with pixel heights at each width. A critic can verify by grep — and the claim holds up.
- **The "QR doesn't rotate" thesis is genuinely original.** Sibling #015 (the other MOB+QR design) rotates the rail from left-edge to bottom on mobile. This design takes the opposite position: keep the bottom anchor at every width and make the *desktop* inherit mobile's form. README §8 explicitly contrasts the two designs in a table — the cohort needs both reads of the same direction, and this one is the strong literalist read.
- **The composition is articulated at the CSS class level, not just the visual level.** `.rail` and `.m-rail` share the layout; `.dock` is the same element; `.det` and `.d` are the same dot for the desktop seam and the mobile grab-handle.
- **Apply+Reset is rigorously escalated:** diff with explicit `--- current / +++ proposed` markers, `del`/`add` lines listing exact version transitions (PHP 8.2.14 → 8.3.6 with the `strftime()` deprecation hint), named auto-snapshot file (`coffee-shop-2026-05-21T18-42.snap · ~11.4 MB · OPFS`), and typed-phrase confirm (`Coffee Shop`). All three guards layered.
- **Hard-requirements checklist in README §6 is honestly auditable** — each item names the frame that demonstrates it. Cross-checking shows the Files (E), Blueprint (F), Database (H), Logs (G), Settings (I) tabs are all reachable in separate desktop frames. The 6 start-a-new sources are visible in Frame C. The 43-blueprint library with 10 categories is in Frame D (18 cards desktop) + mobile P (8 cards + all 10 category chips).
- **The peek-strip live status** above the rail is a clever pre-CWS pattern: the rail's accessory band shows current site + last log timestamp at idle, encouraging the user to swipe up. Honestly counted as the 8th affordance.
- **Sub-segments stay inside the sheet** — the Sites dock has 3 segments (Yours/New/Library) but they don't spawn new surfaces. Library at FULL detent is just the Sites sheet with the Library segment active. One secondary surface preserved.
- The decision to *decline* SR (smart-resize) despite the brief calling SR "gold" is well-reasoned: SR's momentum gauge encodes interaction history; QR's three-detent vocabulary encodes discrete positions; mixing them would dilute the parity thesis. This is a mature composition decision.

## What's weak or missing

- **Bottom rail on desktop is genuinely unusual.** Users coming from VS Code / Slack / Notion / Figma will expect a left rail. README §4 acknowledges this and mitigates with explicit labels under each icon and an active-state top-edge bar. Still, a critic reading the layout cold will register surprise — and surprise *is* a cost for new users.
- **Vertical real estate cost on desktop is real:** the rail + peek-strip + top capsule together claim ~ 150 px of the 900 px viewport height, leaving ~ 750 px for the iframe. That's still more than enough, but a left-edge rail would cost only 64 px of horizontal space on a 1440 px viewport (4.4 % vs 16.7 %). The trade-off is correctly named in §4.
- **Two doors to Sites** (rail dock 1 + CWS widget) is acknowledged, but the CWS widget is also positioned just above the rail dock — same vertical zone. Visually they're close enough that a user might read them as the same affordance, which is fine — but if they read them as duplicate, that's the design's only friction point.
- **The Live preview category is not in the rail.** The rail's 5 docks are Sites/Files/Blueprint/Database/Logs — none is "Live preview." The iframe is *always* the wallpaper, so Live preview is the *background* state of the design, not a dock. This is defensible but the v1 #14 Ventura convention had Live as the first row; this design subsumes Live into "no sheet open." A future reader may want a dock-numbered-0 for "go back to clean iframe."
- **Identity slot ("Playground · v7-build · workspace") + AZ avatar** at the rail's edges are decorative. README explicitly excludes them from the affordance count as "identity, not affordances" — defensible. But the strict-count critic could flag them.
- **The peek-strip is the 8th affordance and the marginal one.** If the strict critic counts the peek-strip as part of the rail (same backdrop, same anchor), the count drops to 7 with room to spare. If the strict critic counts it separately (it has its own `Open sheet ↑` shortcut hint), the count is 8 at the ceiling. README is honest about this; a future designer would be well advised to either fuse the peek-strip into the rail visually or to remove it.

## How to push harder

- Resolve the "two doors to Sites" by considering whether the CWS widget can become a *peek-strip extension* rather than a separate widget — e.g., the CWS lives *inside* the peek-strip's right end, making it visually part of the rail. That keeps the live-status feedback but collapses two affordances into one zone.
- Add a "Live preview" gesture: pressing Esc or tapping the iframe wallpaper anywhere outside the sheet should *dismiss* the sheet and return to the rail-only state. This makes Live preview a *gesture target*, not a dock — staying with the design's "rail is the IA" thesis but giving the user a one-handed escape.
- The desktop bottom rail's unfamiliarity could be addressed by a one-time tooltip on first launch ("Your docks live at the bottom — try ⌘1 to summon Sites"). Currently the design is silent about the first-touch onboarding moment for an unfamiliar IA.
- Consider whether the peek-strip needs to be a separate band. Fusing its content into the rail (live status as a text label that morphs into the rail's left-most cell) would close the 8-affordance count to 7 cleanly.
- Frame R could be tightened by adding a third column to the parity table: "*which CSS class lives in both viewports?*" — currently the answer is in a separate `<pre>` block. Merging them would make the proof a single visual.

## Lesson for the next v7 designer

If you claim viewport parity, prove it with shared CSS class names in a dedicated parity frame — not just by writing "the mobile and desktop use the same primitives" in the README. A critic who can grep your stylesheet and verify the shared classes will rate your composition higher than one who only reads your prose.
