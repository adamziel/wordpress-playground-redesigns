# Review of v7 #032 — Ventura Sidebar · Epic Hero in Live Preview

**Direction:** "Ventura sidebar with Live Preview as first row. Live Preview holds an Epic cinematic hero for the active blueprint when not yet booted; switches to iframe when running."

**Composed primitives (claimed → verified):**
- **VEN (Ventura sidebar)** — claimed → **verified**. 248-px left sidebar with grouped "Playground" + "Your Playgrounds" sections, hairline dividers, `.glyph` tinted gradient squares (`b-blue/b-purple/b-orange/b-graphite/b-red/b-green`), 10.5-px uppercase tracked-out section labels, 34-px rows, search field. The Settings.app idiom is rendered with fidelity.
- **EPC (Epic cinematic hero)** — claimed → **verified, but only in Panel A**. The `.hero` surface (lines 1565+) is a full-bleed dark warm-gradient poster with a CSS-built coffee mug, 44-px `hero-title` ("Coffee Shop. WooCommerce, hot."), `LAUNCH ⏎` button, 6-tile sources grid covering all six new-playground sources, and a `your-pgs` widget pinned to the bottom-right. The hero language matches the direction.
- **DEV (slim top bar)** — claimed → **verified**. Single 40-px row with traffic lights, refresh, URL pill, save-state pill (Unsaved + inline Save), Fullscreen `F` button, More kebab. One row.
- **TAB (tabs inside Site Manager)** — claimed → **NOT verified in the rendered HTML**. The README §6 lists Panels C–G for the five Site Manager tabs, but the file ends at Panel A (line 1670 contains the comment `<!-- We continue with additional panels in subsequent inserts -->` and `<div id="panels-tail"></div>` is empty). The `.tabstrip` CSS is defined but never instantiated in markup.
- **CWS (corner widget)** — claimed → **CSS defined, never rendered**. The `.cws` class is fully styled (lines 582-606) including the swatch + name + meta + fs-ico + green pip live indicator, but no `<div class="cws">` element exists in the markup — because the booted Panel B that would carry it was never rendered.
- **MOB** — claimed → **not rendered**. No 390×844 frame exists.

**Affordance count at idle (Panel A):** 8 by the legend dots (Refresh · URL pill · Save-state pill · Fullscreen · More · Sidebar-as-one-surface · LAUNCH · Browse all 43). README is candid that strict count is 12 (counting each sidebar row and each source tile).

**Scores (out of 10):**
- Composition: 4 — VEN + EPC + DEV compose cleanly *in Panel A*, but the README claims VEN + EPC + DEV + TAB + CWS + MOB and only the first three are demonstrated. The thesis (hero crossfading to iframe at same coordinates) is named in the README but the crossfade target state (Panel B with the live iframe + CWS) was never rendered.
- Banned patterns avoided: 9 — single 40-px row, no depth-as-control, no terminal echo. Clean on this axis.
- Restraint: 8 — 8 affordances at the ceiling by the dock convention; the legend honestly numbers them. The source tiles are children of the hero so don't multiply the count.
- Iframe primacy: 3 — the iframe is *promised* (Panel B's crossfade) but never rendered. At the hero state the iframe is conceptually absent (pre-boot), which is consistent with the direction, but with no booted-state panel the design can't demonstrate iframe primacy. Fullscreen button is present in the top bar.
- One-door / one-secondary: 7 — Library has two doors (the hero "Browse all 43 ⌘K" + the Blueprint sidebar category); README flags this honestly. The pane swap is the single secondary surface.
- Direction adherence: 5 — Live Preview is the first sidebar row ✓, hero in the pane ✓; but "switches to iframe when running" is *never shown*, which is half the direction's claim.
- Visual fidelity: 7 — pure-CSS coffee mug poster is charming, gradient sidebar glyphs are crisp, Inter substitute for SF Pro, real-looking save pill states declared in CSS. But only one panel exists, so the fidelity is concentrated in one frame.
- Synthesis quality: 4 — VEN and EPC genuinely cohabit one pane in Panel A. But the synthesis the README *claims* — same coordinates, two lives, crossfade between hero and iframe — is asserted, not demonstrated.

## What works

- The Epic hero is the most Apple-faithful EPC rendering in v7: the LAUNCH verb is one decisive primary, the kbd `⏎` hint inside the button is exactly the v1 #13 idiom, and the 6 sources tiles sit *underneath* the hero (not as competing primaries), making the "one verb, multiple doors" pattern legible.
- The sidebar's two-group structure ("Playground" categories + "Your Playgrounds" instance list) is a smart Ventura idiom — the v1 #14 sidebar separated nav from saved sessions; this design layers both into one sidebar with a divider, keeping the rail under 5 *categories* while exposing the 4 saved playgrounds inline.
- The `your-pgs` widget anchored bottom-right of the hero is a clever pre-CWS pattern: while no site is live, the corner widget shows the saved list (not status) — same affordance slot, different content state.
- Numbered legend dots (1–8) on the rendered panel are an honest critic aid: the design self-counts its own affordances on the surface of the artifact, not just in the README.

## What's weak or missing

- **The design is unfinished.** Line 1670 says "We continue with additional panels in subsequent inserts" and `<div id="panels-tail"></div>` is an empty placeholder. Panels B (booted), C–G (Site Manager tabs), H (library), I (boot), J (fullscreen), K (save transition), M (mobile) are all listed in the README but absent from the HTML.
- **CWS is defined in CSS but never instantiated.** The `.cws` rule with its swatch + name + meta + fs-ico + green pip is fully styled but no markup uses it. The README claims CWS as a composed primitive — it isn't, in the artifact.
- **No mobile frame.** README claims a 390×844 Panel M; no mobile container exists. The "MOB" primitive is asserted, not composed.
- **No Site Manager / Blueprint / Database / Logs panels.** The brief's hard requirement of all 5 Site Manager tabs reachable with visible content is not met.
- **No blueprint library, no save-state transition, no Apply+Reset triple-guard.** Hard requirements of the brief are missing.
- **The crossfade — the design's headline synthesis** — is described in the README ("the hero crossfades to the WordPress iframe at exactly the same coordinates") and the README admits in §4 that "the hero/iframe crossfade is the design's most distinctive moment, but it's shown only in stills" — yet even the stills (Panel A pre + Panel B post) are not both rendered. Only Panel A is.

## How to push harder

- Render Panel B with the iframe inside the same pane container as the hero, with the CWS pill in the top-right of the iframe surface and the active sidebar row still "Live Preview" — close the loop on the direction.
- Render at least one Site Manager tab to demonstrate the segmented `.tabstrip` (currently CSS-only). Settings tab with the Apply+Reset triple-guard would satisfy two hard requirements at once.
- Add the 390×844 mobile frame with the bottom-sheet category picker. The README claims same CSS class names on both viewports — prove it.
- Reduce the README's claim list to only what's actually rendered, OR finish rendering the panels the README claims. Right now the README writes checks the HTML doesn't cash.
- Consider whether the `your-pgs` widget (bottom-right of hero) and the CWS pill (top-right of iframe) are the *same affordance in two states* or two different widgets. If they're the same — show the transition; if different — justify the second door.

## Lesson for the next v7 designer

If your thesis is a *transition* between two states (hero ↔ iframe), you must render both endpoint panels; a single still of the "before" state cannot demonstrate composition because the composition only exists across the boundary.
