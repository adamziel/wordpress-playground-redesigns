# Review of v7 #033 — Corners on desktop · Status strip on mobile

**Direction:** Mobile-first MOB + CWS composition — four corner widgets (TL Sites · TR Settings+Fullscreen · BL Logs · BR Site Manager) on desktop; the *same four roles* collapse into a horizontal four-cell status strip across the top on mobile. Same IA on both viewports; only geometry changes.

**Composed primitives (claimed → verified):**
- **MOB** — claimed → **verified, and is the backbone**. The mobile frame (Panel B) renders the `.m-strip` containing four `.m-cell.{tl,tr,bl,br}` cells with the exact same accent colours and live-status fields as the desktop corner widgets. Panel D (TL → bottom sheet) renders the sheet with a "From TL · Sites" indicator chip with violet dot — the anchor-back metaphor survives the geometric collapse. Panel K1/K2 do the same for BL and BR. This is the rare v7 design where mobile is treated as the canonical IA, not a fallback.
- **CWS — applied four times** — claimed → **verified**. Panel A renders `.cw.tl` (Coffee Shop · running · OPFS), `.cw.tr` (WP 6.7 · PHP 8.2 + gear + Fullscreen), `.cw.bl` (Logs · count chip 3 · last 2m), `.cw.br` (Site Manager · Files · BP · DB tab chips). Each is a live-status mirror AND the sole door to its function. The four colour seams (`--tl/--tr/--bl/--br`) materialize as `panel.tl::before` etc. accent stripes when the panel expands — the "panel grows from this corner" cue is rendered.
- **DEV (slim browser-style top bar)** — claimed → **verified as a *floating address capsule***, not a full-width bar. `.addrcap` is a 38-px centered pill containing refresh + URL input + save-pill (Saved · Coffee Shop). On mobile `.m-addr` is a thin row below the strip. No two-row bar, no full-width chrome.
- **3LD (three-layer depth, visual only)** — claimed → **verified as quantified tokens**. `--z0 0px / --z1 8px / --z2 18px / --z3 36px` are used as `backdrop-filter: blur(var(--zN))` on widgets (z1), panels (z2), library scrim (z3). The iframe (z0) is the only sharp surface. Depth is read-only hierarchy, not a control surface.
- **GLS** — claimed → **verified**. Every widget and panel is `rgba(16,20,28,0.62)` + `backdrop-filter blur+saturate`. Each corner gets its accent-coloured seam (violet/teal/amber/blue) at the leading edge so the four glass surfaces don't read as identical.
- **VEN (≤ 5 per pane)** — claimed → **verified**. TL panel: 3 segments (Yours/New/Library) and 4 visible saved playgrounds. TR: ≤ 5 fields. BL: 5 level chips. BR Site Manager: exactly 5 tabs. Library: 10 categories (matches the brief).

**Affordance count at idle (Panel A):** README claims 8; counted: refresh + URL input + save-pill + Fullscreen + Settings gear + TL widget + BL widget + BR widget = **8**. (TR widget is the same role as Fullscreen+Settings gear, but README counts the gear and Fullscreen separately; the TR widget itself isn't counted as a separate affordance — defensible but mildly contestable.) Strict reading would say the WP/PHP version pills inside TR are extra visual chrome but README correctly classifies them as *status*, not affordances.

**Scores (out of 10):**
- Composition: 9 — MOB + CWS composed at a structural level: the mobile strip and the desktop corners share `.tl/.tr/.bl/.br` modifier classes and identical accent variables. This is the strongest "mobile = wider-mobile desktop" execution in the round so far. DEV/3LD/GLS/VEN are scaffolding rather than equals, but each has a named CSS role.
- Banned patterns avoided: 10 — no two-row top bar (no bar at all, just floating capsule); 3LD is purely visual depth; the BL Logs count chip is a *signal*, the panel is the *content* — single channel at two resolutions, not duplicate.
- Restraint: 8 — 8 idle affordances at the ceiling. Honestly counted with role tags excluded as status. Tab-chip mirrors inside the BR widget hint at internal state without adding interactive surfaces — a clever pattern.
- Iframe primacy: 9 — iframe is the only `--z0` sharp surface, fills the viewport behind every other layer. Fullscreen is reachable from the TR widget (button) and the mobile address row. Explicit `F` shortcut named. The four corner widgets are 44 px tall and pinned at the corners — they don't crowd the content frame.
- One-door / one-secondary: 9 — TL = sole Sites door, TR = sole Settings + Fullscreen, BL = sole Logs, BR = sole Site Manager. Save-pill = sole save surface. Address = sole URL surface. Library is reached only from TL → Library segment. The README is careful to enumerate that *one* panel can be expanded at a time.
- Direction adherence: 10 — the literal direction is "four corners on desktop, status strip on mobile" and the design renders exactly that. The accent colours, the live-status fields, and the anchor-back animation are isomorphic across both viewports. The transformation IS the design.
- Visual fidelity: 9 — wallpaper layer, accurate-looking WP storefront mock, real PHP log lines (strftime deprecation, include warning, REST endpoint timing, SAVEPOINT), real blueprint JSON verbs, 20-card library with the actual blueprint catalog names (Vueo, Spiel, Koinonia, Substrata, Readymade, Grammer, Piel, Kentwood), Apply+Reset modal with explicit version diff and 7-day snapshot retention. JetBrains Mono on the role tags is a nice typographic signal.
- Synthesis quality: 9 — the four-corner / four-cell duality is the headline synthesis and it works at the level of *shared class modifiers*. The accent seam on expanded panels visually points back to the originating corner, which is the v5 #036 spec correctly executed (and *correctly* extended to mobile via the "From TL · Sites" indicator chip).

## What works

- The mobile status strip is the strongest mobile-parity execution in the v7 round so far. Most "mobile" panels are either a hamburger collapse or a vertical stack of the desktop sidebar; this design genuinely transposes the *same role identities* into a horizontal row that fits next to a notch / dynamic island. The "From TL · Sites" anchor chip in the bottom sheet preserves the corner-to-panel mental model.
- Four colour seams (violet TL, teal TR, amber BL, blue BR) is a smart restraint: the panel expanded from a given corner inherits its colour, so the user can tell at a glance which door they came through. This is depth-as-identity, not depth-as-control — exactly what the brief asks for.
- The Apply+Reset escalation is the most thorough in v7: typed-phrase confirm with the literal site name + a 5-row diff table (WP/PHP/DB/Uploads/Plugins from→to) + a 7-day auto-snapshot promise from TL → Yours → "Coffee Shop · pre-reset 2026-05-21 16:42". The breadcrumb back to the snapshot location is named, not just promised.
- The save-pill is correctly consolidated: one capsule that morphs through Unsaved → Saving (with `41%`) → Saved (panels L1/L2/L3). The brief's "Unsaved → Saved transition visible" is met with three side-by-side stills, not just one state.
- Real blueprint catalog names (Vueo, Spiel, Koinonia, Substrata, Readymade, Grammer, Piel, Kentwood, Spiel, Piel) are accurate. This is rare; many designs invent plausible names.
- BR Site Manager `.cw.br` shows three inline tab chips (Files · BP · DB) — a clever way to surface internal state without spawning a separate affordance. Tapping the widget opens the panel pre-focused on the implied tab.
- The README's §8 "How this differs from neighbour designs" explicitly compares against #017, #022, #015, #027 — the designer read the v7 reviews and positioned the design against its closest neighbours.

## What's weak or missing

- The mobile strip "BL/BR" naming preserves identity but the *position* is inverted: the bottom-left and bottom-right corners on desktop become *top* cells on mobile (the strip lives next to the notch). README acknowledges this honestly as "bottom-orientation-blind" but a strict reading of "same IA" would want the BL/BR cells anchored at the *bottom* of the mobile screen. The strip's top placement is defensible (thumb reach + status-always-visible) but the role tag "BL" labelling a top cell is genuinely confusing.
- The TR widget combines *two* roles (Settings + Fullscreen) in one corner. README calls it "TR Settings + Fullscreen" — that's two affordances behind one cell, which doesn't quite fit "one door per role." A strict reading would split TR into a settings widget and a fullscreen button; the design fuses them to stay at 4 corners.
- Each corner's "role-tag" badge (TL · Sites etc.) is decorative chrome that some critics will count as additional surface area. README excludes them from the affordance count; defensible (they're labels, not buttons) but a tight critic might flag them.
- The desktop iframe is partially hidden by the four floating widgets at the corners (~ 220 × 44 px each). On a 1440 × 900 viewport this costs ~ 4 × 9700 = ~ 39 kpx of content area visible. Not large, but worth noting that "iframe primacy" is degraded by 4 occlusions rather than the typical 1 (CWS) or 0 (rail).
- The hard-requirements checklist in README §6 says "five Site Manager tabs reachable with visible content" but only Files (panel H), Blueprint + Database (panel I), Settings (panel E TR popover + panel K2 mobile), Logs (panels F + K1) are demonstrated. There's no explicit Files+Blueprint+Database+Settings+Logs *all together in one Site Manager view* — they're scattered across panels. The brief requirement is met in aggregate but not in a single screenshot.

## How to push harder

- Resolve the "BL/BR cells live at the top on mobile" tension by either (a) anchoring the BL/BR cells to the *bottom* of the mobile screen — splitting the strip into top-pair + bottom-pair, or (b) renaming the role labels on mobile to drop the directional prefix when the position no longer matches. Currently the design hedges.
- Split the TR widget into two distinct cells: one for Settings, one for Fullscreen. This drops you to 5 cells on desktop (still under the budget) and resolves the "two roles, one door" objection. Alternatively, demote Settings to be a sub-segment of the TL Sites widget (since "Settings" applies to the *currently active site*).
- Render a single Site Manager pane that shows all 5 tabs along the top with one tab selected, instead of distributing the 5 tabs across multiple corner-anchored panels. The Site Manager is the brief's heaviest IA surface and deserves a single concentrated frame.
- Consider rotating the colour-seam direction on expanded panels: TR/BR panels should have their seam on the right edge (currently `panel.tr::before` uses `linear-gradient(270deg, ...)` which is correct, but verify the seam visually points back at the corner). This is a small fidelity detail.
- The README is excellent at self-justification but borders on over-claiming in §8. A future designer reading this would benefit from a one-line "I would have lost more here" honest concession (e.g., the BL/BR position inversion).

## Lesson for the next v7 designer

If your synthesis depends on *role identity surviving geometric transformation* (corners → strip), the role labels themselves must keep meaning across the transformation — or the role names must change when the position changes; you cannot have a cell labelled "BL" sitting at the top of the screen.
