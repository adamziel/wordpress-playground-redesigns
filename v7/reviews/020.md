# Review of v7 #020 — Rich Sites Widget · Slim Top Bar

**Direction:** "One frosted-glass Sites widget anchored to the top-left of the iframe carries the entire launcher — current site status, four saved playgrounds, six new-site sources, three featured tiles, and the only door to the 43-blueprint library. Everything that isn't Sites lives in a single 46 px slim DevTools-style top bar."

**Composed primitives (claimed → verified):**
- **CWS (corner-widget Sites) — load-bearing primitive** — claimed → **verified**. `.cws` at `position:absolute; top:14px; left:14px` over the iframe. At idle it's a 300 px capsule with `Coffee Shop · Running · WP 6.7 · PHP 8.2` + animated green dot + chevron. `.cws.expanded` grows to 380 px and reveals `.cws .body` containing a 3-segment switcher (Yours · New · Featured), 4 saved playgrounds, 6 new-site sources, 3 featured tiles, and "View all 43 blueprints" door. The widget is the only path to launcher/library — verified by reading the slim top bar (which has no launcher button).
- **DEV (slim DevTools top bar)** — claimed → **verified**. Single 46 px row with WP mark, refresh, URL bar, status pill, conditional Save (visible only when unsaved per frame J), Site Manager toggle, Fullscreen, More kebab. README explicitly says "no back/forward, no second row."
- **GLS (glassmorphism)** — claimed → **verified, strictly scoped**. `.cws` applies `backdrop-filter: blur(var(--z2)) saturate(140%)` with a 2-stop white gradient and an inner-light border. The Library overlay backdrop uses blur(36 px) for hierarchy. The README is explicit: "glass is the *one* surface, iframe is the *only* solid surface" — anti-pile-up.
- **VEN (≤ 5 per pane)** — claimed → **verified**. Site Manager has 5 tabs. Library sidebar has 10 categories (Featured + 9). Sites widget switcher has 3 segments. Settings has 3 fields + 2 features + 1 danger zone.
- **MOB (mobile = wider mobile)** — claimed → **verified**. Sites widget rotates from top-left to bottom-anchored on mobile (thumb zone), Site Manager becomes a bottom sheet with drag handle, Library becomes a full-screen sheet. 3 mobile scenes rendered (M1/M2/M3).
- **3LD** — README candidly notes a "nod to 3LD for the Library overlay only" — 36 px blur on the modal backdrop for hierarchy, never as control. Honest scoping (not over-claimed).

**Affordance count at idle:**
README claims 7 at saved-idle (frame A), 8 when unsaved (frame J with Save materializing). Enumerated: WP mark (1), refresh (2), URL bar (3), status pill (4), Site Manager toggle (5), Fullscreen (6), More kebab (7), Sites widget (8). Plus conditional Save = 9 when unsaved. Wait — README's own list shows 8 numbered items but claims 7 at saved-idle. Re-reading: "Save materializes only when unsaved (8 total at unsaved idle — still in budget)." Then at saved-idle it should be 7. But the numbered list has 8 items. Either the WP mark+refresh are treated as one ("WP mark also: refresh hit target — single tap"), giving 7, or the status pill is "display only, never action" (also stated in the README) which doesn't count, giving 7. Either way, the count holds at 7 saved / 8 unsaved.

**Scores (out of 10):**
- Composition: 10 — the asymmetric "rich widget + austere bar" thesis is unique. Other CWS designs treat the widget as ergonomic shortcut; this one makes it the load-bearing entry point and *deliberately strips the bar of launcher access* so the widget can't be bypassed.
- Banned patterns avoided: 10 — single-row 46 px bar; depth used only as Library scrim hierarchy (not clickable); no terminal echo (Logs is read-only per-request grouped DevTools idiom).
- Restraint: 9 — 7 at saved-idle, 8 at unsaved. README counts honestly (the "WP mark = refresh hit target" is reasonable; the status pill is display-only). Borderline because both rationales are necessary to get to 7.
- Iframe primacy: 9 — iframe is the only solid surface and is the wallpaper. 300×96 px of top-left iframe content is occluded by the idle widget, which README acknowledges as a real cost (mitigated because that area is usually wp-admin "My Sites" duplicating widget info). Fullscreen button is explicit, plus a pull-tab to restore chrome in frame I.
- One-door / one-secondary: 10 — Settings/Files/Blueprint/DB/Logs = only Site Manager. Save = only the slim-bar Save button. Sites/launcher/library = only the corner widget. No triple-entry-points anywhere. README is explicit and verifiable.
- Direction adherence: 10 — every clause materializes. Widget at top-left of iframe (verified). Widget carries saved+sources+featured+library door (verified in frame B). Bar at 46 px (verified). Bar carries everything-not-Sites (verified by inspection). Asymmetry of "rich widget + austere bar" is the design.
- Visual fidelity: 10 — Lucide icons via CDN, Inter + JetBrains Mono, 14 demonstrated frames (A through N + 3 mobile), real PHP log lines including `PHP Fatal error: Uncaught Error: Call to undefined function wc_get_low_stock_amount()`, request-id grouped logs with TTFB + query counts, 4 saved playgrounds (Coffee Shop, Gaming News, Friends RSS, My Resume), 14 real wp_options rows, 16 blueprint cards in library, real Gaming News blueprint JSON with 7 valid steps (installTheme/Plugin, runWpInstallationWizard, importWxr, setSiteOptions, login).
- Synthesis quality: 10 — the synthesis is the *asymmetry*: heavy on one corner, light on one bar, never two rows of chrome. Each primitive serves the asymmetry — CWS makes the corner rich, DEV makes the bar slim, GLS makes the widget look "secondary" without being invisible, VEN keeps the rich widget from becoming cluttered, MOB rotates the asymmetry rather than abandoning it.

## What works

- The deliberate *void* in the top bar (no launcher button anywhere) is the design's discoverability mechanism. README is explicit: "the slim top bar deliberately has *no* launcher entry so the visual void points the user back to the corner." That's the discoverability trade-off addressed in HTML, not just in prose.
- The widget shows live status in *every* frame (A · C · D · E · F · H · M · N) — never disappears when other surfaces open. This is the explicit fix for the design-010 issue ("CWS critic flagged its disappearance as a flaw; status surviving content swap is the whole point"). The lesson is internalized.
- 14 demonstrated frames is the densest coverage in the round so far. Each frame demonstrates one specific state, with frame K being an "affordance-count audit card" — meta-documentation embedded in the design itself.
- The Apply+Reset modal (frame E) uses typed-phrase + diff + auto-snapshot together, with the typed-phrase featuring a "live tick" when the user's input matches the site name. Most interactive Apply+Reset in the round.
- Status pill has four real visual states (saved / saving / unsaved / booting / no-site = 5 actually). The pill is the canonical save state surface; the Save button is the action when in unsaved state.
- The README explicitly classifies 3LD as a "nod" not a primary primitive — honest scoping that avoids the over-claim trap that designs 005 and 006 fell into.
- README cites a v5 review critique by name ("review #010 critic flagged its disappearance") — the design is responding to prior critic feedback, which is the v7 brief's "compose, don't invent" principle in action.

## What's weak or missing

- The Sites widget occludes 300×96 px of top-left iframe content (about 3 % of the visible iframe). README acknowledges the cost and notes that this area usually duplicates wp-admin's "My Sites" menu — but that's a probabilistic claim ("almost always") not a guaranteed one. For non-wp-admin iframe content (e.g., a single-page theme demo), the occlusion is real.
- "WP mark = refresh hit target — single tap" is a creative affordance fold that works visually (clicking the W refreshes) but reads as 2 separate items to a user (the W and the refresh icon are distinct). README admits this and counts them as one, which is defensible but a strict critic could split.
- The status pill is "display only, never action" — but real users *will* click status pills to save. The README's hard line "status doubles as door" is contradicted by the explicit Save button. Why have both?
- The Site Manager toggle in the bar is a chrome button that opens the Site Manager pane. That's a separate door from the widget (Sites/launcher). Two distinct doors for two distinct concerns — but this means the design has both a CWS-style widget for Sites AND a traditional bar-toggle for Site Manager. The asymmetry thesis would be even tighter if the Site Manager were also accessible from the widget (e.g., the widget's "Yours · New · Featured · Manage" 4-segment switcher).
- Mobile's "Sites widget at bottom" reverses the "top-left over iframe" anchor; the design becomes more like a bottom-bar pattern (closer to design 015's mobile). The CWS "anchored to corner that you summon by tapping it" idiom doesn't survive the mobile flip as cleanly as the README implies.

## How to push harder

- Add a fourth segment "Manage" to the widget's 3-segment switcher (Yours · New · Featured · Manage) so the widget owns *both* Sites and Site Manager. That removes the Site Manager toggle from the slim bar and gets you to 6 idle affordances + 1 Sites widget that owns 2 concerns.
- Document the minimum iframe size where the widget's 300×96 occlusion becomes intolerable. If a 1024-wide iframe loses too much top-left, the widget should hide or fade-on-scroll.
- Consider showing the widget *empty* in frame M (first visit / no site) with a 50 % opacity ghost preview of what the widget will contain after launch — a teaching moment for the discoverability problem.
- The status pill is "display only" — make this rule visible by giving it a different shape (e.g., no border, only background tint) vs the Save button (full border + icon). Right now they look similar enough that users will try to click the pill.
- The "asymmetry" thesis is the design's distinctive feature; consider an annotation in the affordance-audit card (K) showing the visual weight balance: corner widget at 9 units, slim bar at 7 units, deliberately uneven.

## Lesson for the next v7 designer

If your CWS widget is "load-bearing," delete the corresponding entry from the slim top bar so the user can't bypass the widget — a deliberate void in the bar is the discoverability cue that points back to the corner.
