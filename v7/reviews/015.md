# Review of v7 #015 — Quiet Rail · Mobile Parity

**Direction:** "QR (quiet-rail-to-loud IA) + MOB (mobile-first wider-phone) composed so the mobile design is the canonical IA and the desktop is literally a 'wider phone'. A 5-dock quiet rail lives on the bottom on mobile (tab bar) and rotates 90° to the left edge on desktop. The same secondary surface is a sheet with three detents (peek 35 % / mid 65 % / full 85 %) — vertical on mobile, horizontal on desktop."

**Composed primitives (claimed → verified):**
- **QR (quiet-rail-to-loud IA)** — claimed → **verified**. 64 px left rail with 5 docks (Sites · Site Manager · Blueprint · Database · Logs). Silent at idle. One secondary pane swaps content per dock. Same v5 #018 IA.
- **MOB (mobile-first parity)** — claimed → **verified, with the most rigorous parity implementation in the round**. The detent classes `.pane.detent-peek/mid/full` (420/780/1180 px widths on desktop) are mirrored by `.m-sheet.peek/mid/full` (35%/62%/84% heights on mobile). The same three-name vocabulary (peek/mid/full) and the same percentages (35/65/85) survive the 90° rotation. Even the gauge — `.gauge` on desktop seam with 3 vertical dots, `.m-detents` on mobile under the grab handle with 3 horizontal dots — uses the same `.det` and `.det.on` class names. This is parity proven by CSS, not by rhetoric.
- **CWS (corner-widget Sites)** — claimed → **verified, with a smart twist**. The widget sits at bottom-*right* of the iframe (not bottom-left as the spec example suggests, but bottom-right works because it doesn't compete with the reading flow). Shows live status ("☕ Coffee Shop · WP 6.7 · PHP 8.2 · 11.4 MB"). Wired as a door to the Sites pane. Most importantly: panel B has an on-canvas annotation "CWS persists even when pane is open" — and the widget is visible at the peek detent. This addresses the design-010 issue where the CWS vanished as soon as the pane opened.
- **DEV (slim top capsule)** — claimed → **verified**. 38 px capsule with refresh, address, save-state pill (morphs through Saved / Saving / Booting / Unsaved across panels), ⋮ More menu, explicit ⛶ Fullscreen F button.
- **VEN (≤ 5 items per pane)** — claimed → **verified**. Rail has exactly 5 docks. Site Manager pane uses 5-tab segmented strip. Sites pane uses 3-segment strip.

**Affordance count at idle (Panel A):**
README claims 7: refresh, address bar, save-state pill, More menu, Fullscreen, Rail (as one surface), CWS widget. With the rail-as-one-surface convention (Apple Dock precedent), 7 is defensible. Strict count of 5 individual rail docks would bring it to 11 — README acknowledges this openly with "would be 11 if every dock counts separately." Documented honestly.

**Scores (out of 10):**
- Composition: 10 — QR + MOB is the cleanest mobile/desktop parity composition in the round. The detent vocabulary (peek/mid/full @ 35/65/85) is the synthesis point: it's a QR snap-position mechanism that survives intact across MOB's rotation. No other v7 design demonstrates this kind of width-agnostic IA preservation.
- Banned patterns avoided: 10 — single-row top capsule; depth used as decoration in boot/library overlays only (not as control); no terminal echo. Apply+Reset uses typed-phrase + diff + auto-snapshot trio.
- Restraint: 9 — 7 affordances under the rail-as-one-surface rule, transparent about the strict-11 reading. Inline Save button on Unsaved row is the only documented "second door" to Save (lands on same action).
- Iframe primacy: 10 — iframe is the sharp wallpaper at z0; explicit ⛶ Fullscreen F button in every panel's top capsule. The detent vocabulary preserves iframe primacy because even at "full" (85 %) the iframe is still 15 % visible.
- One-door / one-secondary: 9 — Settings → only Site Manager Settings tab. Save → only the save-pill (+ same-action inline duplicate on unsaved row). Library → modal overlay only. One documented duplicate: Sites rail dock + CWS widget reach the same Sites pane (intentional symmetry). Same accepted trade-off as designs 010 and 013.
- Direction adherence: 10 — every clause materializes. The rail rotates 90° (verified in `.rail` vs `.m-rail` class). The sheet detents share vocabulary (verified by class names). The detent gauge appears at both widths (verified in `.gauge` desktop + `.m-detents` mobile). The "parity" column between desktop and mobile frames in panels A/B/C/E makes the thesis visually literal.
- Visual fidelity: 9 — Inter + JetBrains Mono via CDN, light/warm palette (deliberately differentiated from the dark glass/Mica siblings 010-013), real PHP log lines including `strftime` deprecation, `include()` failed, SAVEPOINT, undefined function fatal, REST endpoint warning, real blueprint JSON, 5 saved playgrounds (including a booting one and an unsaved one), 8 desktop frames + 7 mobile frames (4 inline parity + 3 standalone).
- Synthesis quality: 10 — the synthesis is the shared detent class system. Same `.det / .det.on` classes power the desktop seam gauge and the mobile grab-handle dots. Same `.peek / .mid / .full` modifiers power both the horizontal pane and the vertical sheet. That's parity *implemented*, not described.

## What works

- The detent vocabulary (peek/mid/full @ 35/65/85) is one CSS-token system spanning both viewports. The desktop pane widths and mobile sheet heights are derived from the same percentages, so the IA literally translates rather than re-implements.
- "CWS persists even when pane is open" is annotated *on the canvas*, and the widget is rendered visible in panel B at the peek detent. This is the explicit fix for the design-010 issue where the CWS vanished — designed-in continuity rather than designed-around.
- The light/warm palette is deliberately chosen for differentiation. Section 8 of the README compares 015 against its 6 QR-cohort siblings explicitly. That's the v7 brief's "compose, don't invent" thesis taken seriously — and the README makes the differentiation case for a reviewer scanning thumbnails.
- The parity arrow column between desktop and mobile frames in panels A/B/C/E makes the thesis visual: same labels (PEEK 35% / MID 65% / FULL 85%) appear at both widths, with an explicit "same content, same detents" annotation.
- README explicitly refuses to add SR ("composing SR on top would dilute the parity thesis") — restraint as a design decision, documented. This is the v7 brief's anti-piling discipline.
- The README's section 8 (comparison table) is unusual and valuable: it positions 015 against its 6 siblings and names the differentiator (parity is the thesis, not finish).
- Five saved playgrounds in panel E, including the "Unsaved Playground" warning state and a "Gaming News booting now" state — the most diverse saved-list in the round.

## What's weak or missing

- "Mobile-first parity" claims that mobile is the canonical IA, but the README also says "the same v5 #018 IA verbatim" — v5 #018 is a desktop pattern. So mobile parity *follows* a desktop primitive's pattern rather than starting from mobile constraints. The parity is bidirectional, which is fine, but the "mobile is the source of truth" claim is more rhetorical than structural.
- The detent percentages 35/65/85 don't sum to anything meaningful; they're just three snap positions. Other designs (003/004/005/006/007) use SR ratios that sum to 100 (75/25 means 75 iframe + 25 pane). This design's percentages are isolated detent positions, which is internally consistent but reads differently from the SR cohort's conservation logic.
- The README says "Save → only the save-pill (inline Save on Unsaved row is the same action)" — but having two visible Save controls in the UI is still two doors by any user-facing reading. The "same action" caveat is logical but not visual.
- The desktop seam gauge sits at `left: 478px` in panel B (peek detent) — that's calculated as 64 (rail) + 414 (pane width) = 478. Position is correct. But the gauge has only 3 dots vs the README's "3-detent" claim — verified, no issue.
- Light/warm palette is differentiating but may read as less "designed" to a reviewer expecting v7's typical dark-glass polish. README acknowledges this trade-off ("may read as less premium").
- 7 mobile frames is generous, but the brief asked for both viewports rendered — many designs only show 1-2 mobile states. Worth noting this design over-delivers on mobile.

## How to push harder

- Sync the parity claim with the implementation by either renaming "v5 #018 IA verbatim" to acknowledge that you're *applying* it to mobile, or by showing one explicit mobile-first moment (a gesture that the desktop borrows rather than the other way around) — e.g., the swipe-down-to-dismiss sheet gesture working as a sweep-right-to-dismiss pane on desktop.
- Make the inline "Save" button on the unsaved row visually distinct from the save-pill so the "same action" caveat reads as "same action, different anchor" rather than "two Save controls." E.g., a small bookmark icon next to "Unsaved" on the row, not a full button.
- Consider a fourth detent state ("full" + "tear-off") where the pane becomes a separate window — that's the mobile→desktop affordance that exceeds parity.
- The README's section 8 (cohort comparison table) is excellent; consider exporting that as a designer-facing artifact for the next v7 round (not in the design itself).

## Lesson for the next v7 designer

If your composition is mobile/desktop parity, prove it by giving both viewports the *same CSS class names* on the corresponding elements (`.det / .det.on` for the gauge, `.peek/.mid/.full` for the detents) — the critic will diff your stylesheet and the shared classes are the proof.
