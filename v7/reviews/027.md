# Review of v7 #027 — Mica · Sites Widget

**Direction:** "W11 Mica + CWS Sites widget — Win11 Settings.app generous-whitespace category sidebar (5 categories only), one canonical Sites widget in a corner, big visible 'Open fullscreen' button on the Live Preview category. Fixes v1 #15's missing affordance."

**Composed primitives (claimed → verified):**
- **W11 (Mica)** — claimed → **verified, as a CSS token system**. `--mica-base / --mica-card / --mica-sidebar / --mica-stroke-out / --mica-stroke-in / --mica-shadow / --mica-shadow-lg`. Every chrome surface (sidebar, top capsule, CWS, panel, library overlay, mobile sheet, mobile bottom nav) uses these tokens. `backdrop-filter: blur(40px) saturate(180%)` is the universal Mica recipe. This is the second design (after 013) to apply Mica uniformly as a token system.
- **CWS (corner widget)** — claimed → **verified**. `.cws` is anchored at `top:14px;left:14px` *above* the sidebar (separate from it), with `width:264px;height:48px`. Shows swatch + name + status line. Sole door to Sites/saved/new/library per the README. Expands inward into a 460×740 Mica panel with a corner-anchor pin. The README explicitly says "a separate Mica capsule pinned to the top-left corner of the viewport (above the sidebar), persistent across every panel (A · B · D · E · F · G · H · M)."
- **VEN (≤ 5 items per pane)** — claimed → **verified**. Sidebar has exactly 5 rows (Live Preview · Site Manager · Blueprint Library · Your Playgrounds · Settings). Site Manager has 5 tabs. CWS panel has 3 segments. Library overlay has 10 category chips.
- **DEV (slim top bar)** — claimed → **verified**. 40 px row with refresh, address bar, save-state pill, Fullscreen button with F kbd, More kebab. Single row.
- **MOB** — claimed → **verified**. Sidebar rotates 90° to a 5-cell Mica bottom nav with the same icon-and-label cells. CWS becomes a top capsule. Sites expand as a top sheet, Site Manager as a bottom sheet.

**Big visible "Open fullscreen" button — the v1 #15 fix:**
README claims a hero CTA on the Live Preview category page. This is the design's distinguishing claim — addressing the v1 #15 explicit critique that "Win11 restraint must add a visible fullscreen iframe affordance." The button persists alongside the top-capsule Fullscreen button (two fullscreen anchors), which is the "two anchors, one verb" pattern justified as making the affordance impossible to miss.

**Affordance count at idle (Panel A):**
README claims 8 at idle. Sidebar's 5 rows count as 1 nav surface per Settings.app convention. Other affordances: top-capsule refresh + URL + save-pill + Fullscreen + More + CWS widget + big Fullscreen hero = 7 + sidebar = 8. At ceiling. Documented honestly.

**Scores (out of 10):**
- Composition: 9 — W11 + CWS is composed cleanly: Mica is the visual material applied to *all* chrome (including the CWS), CWS is the launcher door. They integrate because the CWS is itself a Mica surface inheriting the same token system.
- Banned patterns avoided: 10 — single 40 px capsule; Mica blur is uniform hierarchy not interactive control; no terminal echo (Logs is one-way reader); README explicitly declines 3LD, SR, GLS to avoid piling up.
- Restraint: 8 — 8 affordances at budget ceiling. The hero "Open fullscreen" + top-capsule Fullscreen = 2 fullscreen anchors. Library has 2 doors (sidebar row + CWS "View all 43") — README admits "strict critic could call this dual-channel" but defends as discoverability fix. Both are documented trade-offs.
- Iframe primacy: 10 — iframe is the Mica wallpaper sampled through every chrome surface. TWO visible fullscreen affordances explicitly address the v1 #15 critique. Panel I shows fullscreen state with chrome hidden + tiny restore pill.
- One-door / one-secondary: 8 — Settings has two depths (sidebar row → opens to Settings tab, also direct Site Manager → Settings tab). Library has two doors. Fullscreen has two anchors. README is candid: each is documented as a deliberate redundancy with a discoverability reason. Strict reading flags it; the design is honest about it.
- Direction adherence: 10 — every clause materializes: Win11 Settings.app generous-whitespace sidebar (yes), exactly 5 categories (yes — verified by counting), one canonical Sites widget in a corner (yes, top-left), big visible "Open fullscreen" button on Live Preview (yes, hero CTA in panel A), fixes v1 #15's missing affordance (yes, explicit hero button).
- Visual fidelity: 9 — Inter substitute for Segoe, real PHP log lines, real wp_posts table, three iframe wallpaper variants demonstrating Mica sampling (warm coffee / fresh non-profit / violet art gallery), 8+ desktop panels + 3 mobile screens, 5-row sidebar with descriptions per row, sidebar icons as gradient squares.
- Synthesis quality: 9 — W11 and CWS both share the Mica token system. The CWS is *itself* a Mica surface, so it doesn't visually compete with the sidebar — it just sits *above* it at the corner. The composition is "Mica everywhere, CWS as the launcher door inside Mica." Same family of materials, different roles.

## What works

- The hero "Open fullscreen" button on the Live Preview category is the literal v1 #15 fix. Other Mica designs (013) have the Fullscreen button in the top capsule only; this one *also* puts it as a category-page CTA, making it impossible to miss. The README explicitly cites the v1 #15 critique by number.
- CSS token system for Mica is uniformly applied — every chrome surface inherits `--mica-base / --mica-card / --mica-sidebar / --mica-stroke-*`. The critic can grep for these tokens and verify the material is consistent.
- Three wallpaper variants (warm/fresh/violet) demonstrate Mica's wallpaper-sampling property *in stills*. Design 013 had only one wallpaper, so the sampling claim was theoretical there; here it's demonstrable.
- The CWS at `top:14px;left:14px` (same x/y as the sidebar) sits *above* the sidebar rather than inside it. The two are visually adjacent but structurally separate — preserving "CWS is the corner widget, sidebar is the nav" distinction.
- README §2 explicitly declines 3LD, SR, GLS — "compose, don't pile up" lesson from review #013 internalized.
- Each sidebar row carries both label AND description (Settings.app convention) — self-documenting nav vs the icon-only QR designs.
- Apply+Reset uses typed-phrase + JSON diff + auto-snapshot trio.
- Panel I (fullscreen) is rendered with chrome hidden + tiny restore pill — closes the loop on the v1 #15 fix.

## What's weak or missing

- Two fullscreen affordances is the v1 #15 fix taken seriously, but it's also a one-door violation by the strict reading. The README defends it as "two anchors, one verb." Defensible but flagged.
- Sidebar consumes 264 px (18 % of 1440 viewport) — same trade-off as design 010 (which used 64-72 px rail). README acknowledges it: "more chrome than the QR-rail designs (10/013)." The defense is "labels + descriptions make the IA self-documenting." Trade-off documented.
- CWS at top-left occludes the same width (264 px) as the sidebar — but at 48 px tall. For wp-admin views this overlaps the admin bar (acceptable); for front-end views with a hero image it costs visible content. README acknowledges this.
- The "Library" door is duplicated (sidebar row + CWS "View all 43"). README defends as discoverability fix because "users miss that Saved Playgrounds is also where you browse 43 blueprints." Could be tightened by having the CWS only mention "View saved" and the sidebar own all-43-browsing.
- Inter substitute for Segoe — same fidelity gap as design 013. Acknowledged in the brief.
- Sidebar row "Settings" + Site Manager → Settings tab are two depths of the same door. README documents this but doesn't fix it. Could collapse by removing the standalone sidebar Settings row (it's a sub-tab of Site Manager anyway).

## How to push harder

- Pick one canonical anchor for Fullscreen: either the hero CTA on Live Preview (rich, prominent) OR the top-capsule button (always reachable). Currently you have both, which is the v1 #15 over-correction. The hero alone is sufficient — and the top-capsule slot could be reclaimed for something more universally useful.
- Remove the sidebar's "Settings" row and rely on Site Manager → Settings tab. The sidebar's job is to switch *categories*, and Settings is a sub-category of Site Manager. This gets you to 4 sidebar rows (still under VEN's ≤ 5).
- Make the Library door truly singular by removing the CWS "View all 43" link and letting the sidebar's "Blueprint Library" row own that destination. The CWS focuses on Saved + Start new (its persistent live-status concern).
- Demonstrate Mica's wallpaper sampling at a higher contrast — e.g., a dark wallpaper variant that proves the chrome adapts (sidebar text becomes light). Currently three light wallpapers are shown; a dark one would close the visual demonstration.
- Show one panel with the CWS expanded *and* the sidebar visible together — currently the CWS panel hides or overlays the sidebar context. A frame showing how the user navigates back from CWS to sidebar would close the workflow loop.

## Lesson for the next v7 designer

If you cite a specific prior critique by number (here v1 #15), make the fix *visually unmistakable in one panel* — a hero-sized button beats two smaller buttons; the v1 critique was about restraint hiding the affordance, so the answer is *one big* affordance, not two medium ones.
