# Review of v7 #011 — Quiet rail of corner cells

**Direction:** "QR + CWS composed: the 56→72 px quiet rail is a vertical stack of 4 corner-widget cells. Each cell carries live status (saved-count, PHP version, blueprint count, warning count) AND is the only door to its concern."

**Composed primitives (claimed → verified):**
- **QR (quiet-rail-to-loud)** — claimed → **verified**. A 72 px `<aside class="rail">` lives on the left in every desktop panel. Each `.cell` is monochrome at idle (no fill, just an icon outline) and gains a saturated state when `.active`. Cells stay quiet until clicked. The pane is a fixed-width 360 px secondary surface that opens between rail and iframe with content swapped via `.pane-seg` sub-segmented controls.
- **CWS (corner-widget Sites interaction)** — claimed → **verified, and this is the synthesis**. Each of the 4 cells carries: `.pip` (a colored dot indicating state), `.ico` (icon), `.lbl` (label), `.stat` (live status text — "3 saved" / "PHP 8.3" / "43" / "2 warn"). The state classes `.cell.warn / .info / .bad` change the pip color. In Panel C (booting), the Sites cell's stat text genuinely flips from "3 saved" to "Booting…" and the pip class changes from green to info-blue — so the cell *is* the indicator, exactly as CWS spec requires.
- **DEV (slim top bar)** — claimed → **verified, minimal**. A 36 px frosted `.urlpill` at the top centre carries refresh, address, save-state pill, fullscreen button. No persistent horizontal chrome bar — the pill *is* the chrome above the iframe.
- **3LD (visual only)** — claimed → **verified, used very sparingly**. The README is candid that depth is just decorative scrim/blur and never a control. Rail and pane are solid, not glass.

**Affordance count at idle (Panel A):**
The HTML has a `.count-strip` element labeled "Idle affordances 8/8" enumerating: Sites cell (1), Manage cell (2), Library cell (3), Logs cell (4), Refresh (5), URL/address (6), Save-state name pill (7), Fullscreen (8). Counted on the canvas. The avatar + help icon at the rail foot are explicitly excluded as "decorative identity, not affordances." Strict count holds at 8.

**Scores (out of 10):**
- Composition: 10 — QR + CWS is the explicit thesis, and the synthesis is real: each rail cell is *simultaneously* the QR quiet surface AND the CWS live-status door. The cells aren't QR sometimes and CWS other times — they're both at once.
- Banned patterns avoided: 10 — single 36 px URL pill (no two-row chrome); depth used only as scrim/blur decoration, never as control; no terminal echo (Logs cell is the only logs destination, boot panel shows steps not stdout).
- Restraint: 10 — exactly 8 affordances counted on-canvas with a labeled count-strip. The rail-foot avatar/help are explicitly disclaimed. The most disciplined count in the round so far.
- Iframe primacy: 10 — idle iframe is 1368 px of a 1440 px canvas (rail takes only 72 px). The pane is optional, opens between rail and iframe rather than over it. Explicit Fullscreen button + F key.
- One-door / one-secondary: 10 — Settings → Manage cell → Settings tab only. Save → name pill only. Export → kebab inside Sites > Your sites only. Each rail cell is the sole door to its concern by composition. One 360 px secondary pane, content swaps via sub-segmented controls.
- Direction adherence: 10 — every clause of meta.json materializes: 4 corner-widget cells (yes), live status per cell (yes — saved-count / PHP version / blueprint count / warning count visible in panel A), 360 px pane between rail and iframe (yes), mobile = rail rotated + bottom sheet (yes, panel I).
- Visual fidelity: 9 — Lucide icons referenced (need to verify CDN script is included), real PHP log lines in Logs panel (including "PHP Fatal error: Uncaught WC_Data_Exception"), real blueprint JSON with line numbers in panel E's Blueprint tab, 16 blueprint cards in panel B, 6 sources tiled in panel C, typed-phrase + diff + auto-snapshot modal in panel F, 3 saved playgrounds with running/saved/booting states.
- Synthesis quality: 10 — the cell is the single DOM element that carries both primitives. Click handler grows the pane (QR), pip color tracks state (CWS), stat text updates with reality (CWS). One element, two responsibilities, no duplication. Clean.

## What works

- The cell-as-icon-and-status double duty is the cleanest QR + CWS integration in the round so far. The cell isn't a QR icon next to a CWS widget — it's one square that does both jobs simultaneously, with the pip top-right encoding state and the stat text bottom carrying live data.
- Stat-text content is *load-bearing*, not lorem-ipsum. "3 saved" reflects the actual saved-playgrounds count visible in panel H. "PHP 8.3" reflects the Manage tab's selected version. "43" matches the brief's blueprint count. "2 warn" matches the Logs panel's warning count. The numbers cross-check.
- The README is unusually disciplined about what was *not* composed: it explicitly names SR, GLS, EPC, iPAD, VEN, TAB, W11, SPT, MOB as primitives deliberately excluded with reasoning. That's the kind of restraint the v7 brief asked for.
- The booting state (panel C) is the only one in the round that *changes the rail cell's text* to reflect runtime state ("3 saved" → "Booting…"). Other designs change the iframe content during boot but leave the rail static. This one updates the indicator.
- Panel F's destructive Apply+Reset modal has the typed-phrase + diff + auto-snapshot trio.
- The README documents lessons-applied from prior v5 reviews explicitly (no triple-drawer-stack, single Save embodiment, show overflow menu open). That's the v7 brief's "compose, don't invent" thesis taken seriously.
- The rail width (72 px) is acknowledged as +16 px over the brief's 56 px and disclosed with a one-line reason ("can't legibly carry per-cell stat text at 56 px"). Honest engineering trade-off.

## What's weak or missing

- 4 rail cells leaves only 4 doors for 5 Site-Manager tabs (Settings, Files, Blueprint, Database, Logs). README addresses this by folding all 5 into "Manage" with a sub-segmented control, but that means the Files / Blueprint / Database / Logs sections each need a sub-tab inside Manage — and a 5-way segmented control inside a 360 px pane is tight. Also: the Logs cell exists separately as cell #4, so there are now *two* doors to Logs (Manage → Logs sub-tab AND Logs cell). README claims Logs is its own rail cell because it needs live status (the "2 warn" pip), but this creates a one-door-per-feature soft violation.
- The rail at 72 px on mobile rotated to a bottom tab bar with all 4 cells — but a mobile bottom tab bar typically needs ~64 px height per tab for thumb-friendliness, so the mobile compromise may be tighter than the desktop. README acknowledges only "one-detent sheet" as the mobile compromise.
- Lucide icons referenced as `<svg data-lucide="layout-grid">` patterns, which depends on a Lucide JS init script. Need to verify it's included.
- The Library cell exists as a separate rail cell rather than being a sub-tab of Sites. README discloses this as a CWS-pref choice ("each widget wants its own live status"), but it does mean blueprint discovery has two doors: Library cell directly, and Sites → Library sub-tab. The README admits "a stricter QR-only reading would have made Library a third sub-tab of Sites."
- "2 warn" pip stays amber across all panels — even after panel F's reset, the Logs cell still says "2 warn." A small inconsistency: if reset destroys site state, the warning count should reset to 0 too.

## How to push harder

- Collapse Library into Sites → Library sub-tab (matching the strict QR thesis) and reclaim that rail slot for Files or Blueprint, so each rail cell maps to exactly one Site-Manager tab. That eliminates the Logs / Manage → Logs duplication and makes the cell ↔ tab mapping 1:1.
- Verify the Lucide script tag is included; if not, replace `<svg data-lucide="*">` with inline SVG paths so icons render in stills.
- Add per-state stat updates beyond the boot moment: after Apply+Reset (panel F), show the warning count reset to 0 in the same frame. The synthesis is strongest when the stat numbers track all state changes, not just boot.
- The rail foot has Help + avatar; consider showing one user action (notifications? account?) so the foot is functional and the design has a documented "decorative" boundary the critic can verify.

## Lesson for the next v7 designer

When composing two primitives into one element, make every state change ripple through both primitives at once — the rail-cell's pip color, label, stat text, and door target should all derive from the same state object, so the critic can grep the HTML for the state-binding and see the synthesis.
