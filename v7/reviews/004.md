# Review of v7 #004 — Smart resize · Sites corner

**Direction:** "Smart-resize between iframe and one secondary pane; a frosted Sites widget in the top-left corner shows live status (current site + state) AND is the launcher door. Click the widget → the secondary pane swaps into Sites mode and the gauge snaps to 40/60 in the pane's favor."

**Composed primitives (claimed → verified):**
- **SR (smart resize)** — claimed → **verified**. The `.body` element uses `.ratio-50-50`, `.ratio-40-60`, `.ratio-75-25`, `.ratio-35-65`, `.ratio-60-40`, `.ratio-100-0` classes, and each panel anchors a `.seam` + `.gauge` at `left: 50% / 40% / 75% / 35% / 60% / 100%`. The gauge contains a three-bar height-coded visualization showing which side is heavier, the ratio numerals, and a directional arrow (e.g., "→ pane"). Ratios shift across the 10 demonstrated panels.
- **CWS (corner widget Sites)** — claimed → **verified, and well-composed**. The `.sites-widget` lives `position: absolute` over the top-left of the iframe in every desktop panel. It has a thumb + name + state line (e.g., "running · 7m · WP 6.7 · PHP 8.2") + a `⌘K` corner key. In state B, the same widget gets `.open .active` and grows into an inline rail at the same corner with 4 saved sites, 6 new-source buttons, and a "Browse all 43" footer. The widget transforms in place rather than spawning a separate sheet — this is the cleanest CWS implementation possible given the brief.
- **DEV (slim top bar)** — claimed → **verified**. Single 44 px row with wp-mark, refresh button, urlbar, name-pill (status), optional Save button (visible only in unsaved state I), Fullscreen button, More kebab.
- **3LD (three-layer depth)** — claimed as "visual only, never interactive control" → **verified**. The receding `.pane-iframe.dim` class applies a small blur + saturation drop in panels B/C/D/E/F. Quantified scale referenced in README. Never used as a click target.
- **TAB (tabs-only)** — claimed → **verified, but 6 tabs**. The pane-head has six tabs (Sites · Settings · Files · Blueprint · Database · Logs). Same edge-of-VEN issue as design 003 — six is one over the brief's ≤ 5 guideline. README does not claim VEN, so technically fine, but worth flagging.

**Affordance count at idle (panel A):**
By strict count: wp-mark (1) + refresh (2) + urlbar (3) + name-pill (4) + Fullscreen button (5) + More kebab (6) + Sites widget (7) + seam gauge (8). README claims 8 exactly. The Save button is conditional. The pane tabs sit inside the secondary surface and the README argues they're pane-internal not idle chrome — a defensible reading. Count holds at 8.

**Scores (out of 10):**
- Composition: 10 — SR + CWS is the headline composition the meta line asked for, and they are *interlocked*: clicking the widget drives the SR ratio change. The synthesis point is the click event, not a coincidence of layout.
- Banned patterns avoided: 10 — single-row top bar; depth used as visual cue only (the dim pane is dimmed, but the click target is the pane content); no terminal echo (Logs is one-way).
- Restraint: 8 — 8 affordances exactly at idle, drops to 7 when saved. The widget at 248 px is bigger than v5 #036's purist 64×64, README discloses honestly. The 6-tab pane is borderline.
- Iframe primacy: 9 — iframe is always visible (never < 35 %); Fullscreen button + F hotkey + dedicated state J on mobile. Slight knock because the Sites widget sits *on* the iframe at idle and at ratio 35/65 occupies a non-trivial percentage of the iframe surface area.
- One-door / one-secondary: 8 — Settings = one tab; Launcher = one widget; Save = name-pill (status) + conditional dark button (action). README admits Export PR / Download .zip appear both in Blueprint toolbar AND in More kebab — that's documented duplication.
- Direction adherence: 10 — every literal phrase in meta.json is realized in state B: the widget click pushes the secondary pane to Sites mode, the gauge snaps to 40/60 in the pane's favor, the seam carries the gauge. Verbatim.
- Visual fidelity: 10 — Lucide icons via CDN, Inter + JetBrains Mono, four saved playgrounds with running/paused state, 18 blueprint cards with descriptions, 10 category chips matching the brief counts, real blueprint JSON, realistic PHP log lines (notice/warn/error/fatal), wp_posts table preview, theme functions.php in the Files tab with `register_nav_menus` / `wp_enqueue_style` / WooCommerce class_exists guard.
- Synthesis quality: 10 — the headline trick (widget click triggers SR snap to 40/60 AND widget self-expansion to a rail at the same corner AND pane swaps to Sites mode) is a three-effect chord from a single user action. That's the v7 thesis made concrete.

## What works

- State B is the most coherent demonstration of a synchronized multi-primitive interaction in the round so far. One click, three coordinated effects. The widget *expanding in place* (rather than opening a separate sheet) preserves spatial memory.
- The gauge's three-bar height-coded visualization is a smart variant of the v5 #023 momentum gauge — at 50/50 the middle bar is tallest; at 40/60 the right bar is tallest. The ratio is legible without reading numerals.
- The unsaved widget (state I) borrows the amber accent color and shows "in memory · refresh = lose" — fear messaging at the right level.
- State I's save-state timeline (Unsaved → Saving "First demo" with pulsing amber dot → Saved "First demo" 142 MB) is the cleanest unsaved→saved demo in the round. Includes an explicit toast confirmation.
- The blueprint browser in state B shows 18 cards visible above the fold with 10 category chips numerically matching the brief (Featured 5, Website 10, Personal 7, Content 6, Themes 6, Gutenberg 4, Experiments 3, WooCommerce 4, News 2 — sums to 49 with All-43 overlap).
- Mobile state J is a real fullscreen demonstration with chrome auto-hidden and a single Esc/F exit pill.

## What's weak or missing

- The pane has 6 tabs. Brief's VEN says ≤ 5 per pane. The design doesn't claim VEN so it isn't bound, but combining "Sites" (a launcher at session level) with Settings/Files/Blueprint/DB/Logs (operations on a single site) puts two different scopes in one segmented control. Folding Sites entirely into the widget (it's already the launcher door) would buy back one slot and cleaner semantics.
- The seam gauge is counted as one idle affordance, but it's "informational, not a drag handle" per README. If it's not interactive, it shouldn't count — which gets you to 7. Conversely if it counts because it's visible, the pane tabs (which are also visible) should count too. The boundary is rhetorical; pick one ruling consistently.
- The widget on state H (booting) sits at 60/40 with the iframe showing a boot ring. The widget still says "running" with a green pip — should read "booting · amber pip" to match state I's pattern.
- Export to GitHub PR + Download .zip appear in BOTH the Blueprint toolbar AND the More kebab. README admits this and justifies it as contextual, but a strict reader will call it a duplicate door.

## How to push harder

- Drop the "Sites" tab from the pane and let the widget be the only launcher door. The pane becomes 5 tabs (Settings · Files · Blueprint · Database · Logs) which respects VEN. The widget already covers Sites — having both is dual-channel.
- Resolve the seam-gauge-as-affordance ambiguity. Either make the gauge clickable (e.g., click to cycle through snaps as a discoverability aid) and count it, or treat it as pure status and don't count it (idle = 7).
- Vary the widget thumb icon when state changes — running (coffee filled), booting (spinner), paused (greyed), unsaved (∅). State I does this; the other states (A, C) could too.
- Tie the More kebab's Export PR / Download .zip directly to the Blueprint tab via a hover preview — same destination, but the visual cue makes the duplication a single door rendered twice rather than two doors.

## Lesson for the next v7 designer

A primitive composition is strongest when one user gesture triggers all of them at once — design the synchronization, not the surface; the critic will look for the click handler, not the screenshots.
