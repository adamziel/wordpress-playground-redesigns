# Review of v7 #034 — Smart-resize · Axis rotates with viewport

**Direction:** MOB + SR composed so smart-resize is the IA at both widths, with the same primitive rotated 90° between desktop (vertical seam: iframe left ↔ pane right) and mobile (horizontal seam: iframe top ↔ sheet bottom). Same detent vocabulary (peek 35% / mid 65% / full 85%), same gauge etched on the seam, same focus-grows gesture.

**Note:** This design has **no README.md** and the HTML file is **a fragment that does not close** (no `</main></body></html>`). The artifact is incomplete.

**Composed primitives (claimed → verified):**
- **SR (Smart Resize)** — claimed → **partially verified, only at rest detent**. Panel A renders the desktop seam at 50/50 with a `.gauge` pill showing `iframe 50 / 50 pane`, a 4-dot `.snaps` row with one active, and `⌘[ ⌘]` keyboard hints. Mobile renders a horizontal `.seam-h` and a `.gauge-h` reading `iframe 65 / 35 sheet` with the peek detent dot lit. The composition is *demonstrated* but only at one detent — no Panel B/C/D showing the seam at mid or full, no actual demonstration that the gauge "moves" with focus.
- **MOB** — claimed → **verified at the rest state only**. Desktop and mobile are placed side-by-side in Panel A, both running Coffee Shop, both with the same Sites/Settings/Files/Blueprint/Database/Logs seg-control inside the pane. The mobile rotates the pane to a bottom sheet; the seg-control collapses labels to icons-only on mobile while keeping the same 6 cells. Shared `.gauge` class is claimed in the panel-foot tag strip but the desktop uses `.gauge` and mobile uses `.gauge-h` — different classes for the two orientations.
- **CWS** — claimed → **verified at idle**. `.cws` pill at bottom-left of desktop iframe showing `CS swatch + Coffee Shop + running · WP 6.7 · PHP 8.2 · 11.4 MB` + chev. Mobile equivalent `.cws` at bottom-right of iframe. Live status + door — present at idle.
- **DEV (slim 44 px top capsule)** — claimed → **verified**. Single top row with refresh + back + forward + URL bar + pgtitle + save-pill + Fullscreen `F` + More. Mobile compresses to refresh + URL + save (`SAV` short) + Fullscreen + More. One row.
- **VEN (≤ 5 per pane)** — claimed → **verified inside the pane, but the seg-control has 6 tabs** (Sites · Settings · Files · Blueprint · Database · Logs), which is *over* the cap of 5. The claim is "≤ 5 tabs per pane" but the actual count is 6.

**Affordance count at idle (Panel A desktop):** README is absent, but the in-frame overlay claims `7 affordances at idle · under budget of 8`. Counting on the HTML: refresh + back + forward + URL + pgtitle (passive) + save-pill + Fullscreen + More + seg-control (6 cells) + CWS = ~ 9 strict. Even excluding back/forward and pgtitle as identity, the count is 7-8.

**Scores (out of 10):**
- Composition: 4 — SR + MOB are present in concept (vertical seam on desktop, horizontal seam on mobile with shared gauge vocabulary) but only at one rest detent. The composition's headline claim — *"same detents, same gauge, same focus-grows gesture"* — cannot be verified because no second-detent panel exists. The "axis rotates" thesis needs at least two panels (rest + grown) to demonstrate.
- Banned patterns avoided: 8 — single 44 px top capsule on both widths; no depth-as-control; no terminal echo. Clean on this axis.
- Restraint: 5 — overlay claims 7 at idle, but the seg-control inside the pane has 6 cells (over the ≤ 5 cap), and back/forward/refresh + URL + pgtitle + save + Fullscreen + More gives 8 just in the top capsule before counting the seg-control or CWS. The honest count is 9+.
- Iframe primacy: 6 — iframe is the wallpaper, but at rest 50/50 the iframe loses 50% of its area to the pane — that's a *lot* of chrome. Fullscreen has one anchor (top-capsule button + F key) — clean. No fullscreen panel rendered to demonstrate the state.
- One-door / one-secondary: 6 — pane swap is the single secondary surface ✓. But the seg-control has *6* doors inside the pane (Sites + 5 SM tabs), and the brief is ambiguous about whether Sites and Site Manager should share the same surface — most v7 designs separate them.
- Direction adherence: 5 — the direction is "axis rotates with viewport" and Panel A does demonstrate one axis rotation. But the direction also says *same detents at both widths* and only the rest detent is shown. The rotation is rendered, the detent vocabulary is asserted only in the panel-foot tag strip.
- Visual fidelity: 7 — Brooklyn coffee roastery mock with "Maya Okafor" byline, three product thumbnails (El Tambor / House Blend / Cold Brew Bag) with prices, 4 saved playgrounds with state variety (Coffee Shop saved, Unsaved Playground, Art Gallery saved, Gaming News booting), real-looking version meta strings.
- Synthesis quality: 4 — the headline synthesis (axis-rotates SR) is *named* but only *shown* at rest. With no second-detent or focus-grows panel, the composition is asserted rather than demonstrated. The HTML being unclosed reinforces the unfinished feel.

## What works

- Panel A side-by-side comparison of desktop and mobile in the *same* `<div class="panel-body">` is a clean way to argue parity at-a-glance: the reader sees both renderings at once and can verify the same primitives in both. This layout choice is strong for an SR + MOB direction.
- The desktop `.gauge` pill (`iframe 50 / 50 pane`) and the mobile `.gauge-h` (`iframe 65 / 35 sheet`) share the same readout format — `iframe N / N target` — which is the right visual language for SR's "whichever side you interact with grows" rule.
- The CWS pill at desktop bottom-left + mobile bottom-right is functionally identical (live name + version meta + state dot), demonstrating CWS-as-shared-primitive across viewports.
- The Brooklyn coffee shop mock is genuinely rich: byline, date, read-time, three product cards with light/medium/dark labels and prices. This is a realistic-looking content frame, not a placeholder.
- The 4 saved-playgrounds list has real state variety: Saved (Coffee Shop / Art Gallery), Unsaved (with amber "not saved · lost on refresh" warning), Booting (Gaming News with "62% loaded" progress). The "Unsaved Playground" row is the only design in v7 so far to explicitly render the unsaved state inside the saved list.

## What's weak or missing

- **The HTML file does not close.** It ends at `</section>` with no `</main></body></html>`. A self-contained `index.html` is a hard requirement of the brief; this file is a fragment.
- **No README.** Brief requires README with thesis + primitives + remix + trade-offs + self-critique. None exists. The meta.json `summary` paragraph is the only prose argument for the composition.
- **Only Panel A is rendered.** The brief requires three demonstrated states minimum (working / booting / launcher) and all 5 Site Manager tabs reachable. This design renders only the rest state. There is no:
  - Library overlay (43 blueprints + 10 categories + 16 cards)
  - Booting state
  - Apply+Reset escalation (typed-phrase + diff + auto-snapshot)
  - Save transition (Unsaved → Saving → Saved)
  - Fullscreen state
  - More menu (Export PR + Download .zip)
  - Logs panel
  - Files / Blueprint / Database / Settings tabs with content
  - At least one demonstration of the SR seam at a non-rest detent
- **The seg-control inside the pane has 6 cells (Sites + Settings + Files + Blueprint + Database + Logs), exceeding the ≤ 5 VEN cap.** README claim of "≤ 5 tabs per pane" is wrong; the actual count is 6. Either Sites or one of the SM tabs needs to move elsewhere.
- **The composition's load-bearing claim — same SR primitive rotated 90° — is asserted in the panel-foot tag strip (`SR=true · data-ratio="rest" · data-detent="peek"`) but never demonstrated with a second detent or a focus-grows transition.** A still of the rest state can't prove the gauge moves with interaction.
- **The shared-class claim is partially false:** the desktop uses `.seam` (vertical) and `.gauge`; the mobile uses `.seam-h` (horizontal) and `.gauge-h`. These are *different* classes for the two orientations. The shared-vocabulary claim holds for `.snaps` / `.det-dots` and `.row` / `.sites-list` but the headline "seam + gauge are the same primitive rotated" is *not* shared at the class-name level.
- **The Brief requirement "Both 1440 × 900 desktop and 390 × 844 mobile rendered" is met for one state (rest) but not for any other state.** Mobile booting, mobile library, mobile fullscreen — none exist.
- **Affordance count claim of 7 is wrong by my count.** Refresh + back + forward + URL bar + pgtitle (passive label) + save-pill + Fullscreen + More = 7 in the top capsule alone, before counting the pane's seg-control or the CWS widget. Strict count is 9–10.

## How to push harder

- **Finish the file.** Close `</main></body></html>`. Add at minimum Panels B (booting), C (library), D (Apply+Reset), E (Logs), F (More menu), G (fullscreen), H (save transition). Mobile equivalents in a second row.
- **Write the README.** State the thesis, name the primitives, account for the 6-cell seg-control exceeding VEN's cap (either justify or fix).
- **Demonstrate the seam at a non-rest detent.** A second desktop panel showing the iframe focused (e.g., ratio 75/25 with the gauge reading `iframe 75 / 25 pane` and a different snap dot lit) would prove the SR primitive works. The mobile equivalent would be sheet at mid (65%) or full (85%).
- **Either move Sites *out* of the seg-control** (e.g., the CWS widget IS the Sites door, and the pane only has 5 SM tabs) **or move Settings out** (e.g., into the More menu or the Site Manager's per-site config). Either way drops to 5.
- **Decide whether desktop is left-iframe / right-pane or vice versa.** Currently the desktop puts the pane on the right (default state), but if smart-resize lets the user grow either side, which side does the pane live on? The "axis rotates" thesis is *one* orientation per width — but if either side can grow, the IA is symmetric. Worth being explicit.
- **Either share the seam class name across orientations** (`.seam` with a `.seam.vertical` / `.seam.horizontal` modifier) **or admit that the rotation is implemented as two separate primitives** (vertical seam + horizontal seam) sharing only the visual vocabulary.

## Lesson for the next v7 designer

A primitive that *moves* (like Smart Resize's growing pane) requires at least two stills to argue: one at rest, one at the grown state — a single still of the rest position can only assert that the primitive *could* move, not that the design actually composes with its motion.
