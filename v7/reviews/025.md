# Review of v7 #025 — Cartograph · Atlas beside the iframe

**Direction:** "SPT + SR composed: saved playgrounds occupy stable 2D positions on a pannable atlas; that atlas IS the persistent secondary pane of a smart-resize layout. Whichever pane you touch grows; the other recedes. The atlas never disappears."

**Composed primitives (claimed → verified):**
- **SPT (spatial canvas)** — claimed → **verified**. `.canvas` is a positioned surface; tiles use real x/y coordinates (Coffee Shop at x:-410/y:+128 per README). Pan/zoom controls live in the atlas head with `.zoom-btn` + readout. The atlas head crumb says "3 saved · 43 blueprints · 6 sources" — status displayed as cartographic legend. Regions are dashed-boundary clusters ("Your sites", "Featured", "Experiments", "Themes & content").
- **SR (smart resize)** — claimed → **verified**. The body is a `display: grid` with `grid-template-columns: var(--iframe) var(--atlas)`. Body classes `.body.r-65-35 / .r-50-50 / .r-35-65 / .r-25-75 / .r-100-0` set the two custom properties to matching values (e.g., `.r-25-75 { --iframe: 25%; --atlas: 75% }`). Five demonstrated ratios + fullscreen. `.iframe-focus` and `.atlas-focus` modifiers apply a scrim veil to the receding pane.
- **DEV (slim top bar)** — claimed → **verified**. Single 40 px row with refresh, URL pill, name+state pill, conditional Save button, Fullscreen, kebab. Save button is conditional (visible only when unsaved).
- **CWS** — README claims "deliberately excluded." Verified: there is no `.cws` or `.sites-widget` class in the file. The atlas is the live-status surface, and the minimap appears *only* in Panel J (fullscreen 100/0 with atlas collapsed) as a replacement door. This is honest restraint, not over-composition.

**The synthesis point:** README §2 explicitly states "SPT and SR share the same DOM element": the `.pane-atlas` div is *both* the spatial canvas (with positioned `.tile` children) AND the right pane of the SR grid (width = `var(--atlas)`). Move the seam, the canvas resizes. Click into the atlas, the seam snaps in the atlas's favor. **Two primitives, one DOM element.**

**Affordance count at idle (Panel A, 65/35):**
README claims 8: refresh (1), back/forward (2 as one cluster), URL bar (3), name+state pill (4), Fullscreen (5), kebab (6), atlas pane (7 as one cluster), seam gauge (8). Save button hidden when saved. Compass, legend, position readout are atlas-internal indicators. Documented on-canvas in a count-strip below Panel A. **8 exactly — at the budget ceiling.**

**Scores (out of 10):**
- Composition: 10 — SPT + SR is the headline composition, and the synthesis is at the DOM-element level (one element does both jobs). Same load-bearing pattern as designs 011 (CWS-rail-cells) and 023 (3LD-rendering-SPT) — one element, two primitives.
- Banned patterns avoided: 10 — single 40 px row; depth as decorative scrim only (never clickable); no dual-channel echo (Logs is read-only, boot panel shows structured steps with timings not stdout); README explicitly cites which banned patterns were avoided.
- Restraint: 9 — 8 affordances at ceiling. The compass/legend/position-readout grouping as "atlas-internal cluster" is defensible by the established "rail is one surface" convention. Save button hidden when saved drops it to 7.
- Iframe primacy: 9 — iframe is the larger pane in every non-launcher panel (65/35 idle, 50/50 settings/boot, 35/65 SM modes, 100/0 fullscreen). Panel B (atlas focus 25/75) is the only state where iframe shrinks below 50 %, and that's the explicit launcher state. Fullscreen via F key + Esc exit + visible button.
- One-door / one-secondary: 10 — Settings = Settings tab only. Files = Files tab. Sites = atlas. Export = kebab only. Save = conditional top-bar button. WP Admin/Home = SM-head chips. Minimap appears only when atlas is collapsed (replacement, not coexisting). README is explicit and verifiable.
- Direction adherence: 10 — meta line materializes verbatim. Atlas at stable 2D positions (yes, x/y coordinates). Atlas IS the SR right pane (yes, `.pane-atlas` width = `var(--atlas)`). Whichever pane you touch grows (yes, `.iframe-focus` / `.atlas-focus` body classes). Snap ratios 25/75 · 35/65 · 50/50 · 65/35 · 100/0 (yes, exact five `r-*` modifiers). Momentum gauge etched into the seam (yes, with 5 snap dots).
- Visual fidelity: 9 — Inter + JetBrains Mono via CDN, real PHP log lines including `strftime() deprecated`, `cURL error 35: SSL handshake failed`, `Allowed memory size exhausted`, real wp_posts table preview, real `style.css` editor in Files tab, 9-step blueprint JSON with steps + timings in boot panel, 11 demonstrated panels (A through K), 4 saved playgrounds, 18 blueprint tiles + 10 category chips with real counts, regions grouped on canvas.
- Synthesis quality: 10 — DOM-level synthesis. `.pane-atlas` is simultaneously SPT's canvas container AND SR's right pane. The CSS custom properties `--iframe` and `--atlas` drive both the SR grid AND the canvas pan/zoom origin. One element, two responsibilities, bound by the same variables.

## What works

- The DOM-element-level synthesis is the cleanest "compose, don't pile up" example in the round. SPT + SR aren't side by side; they share the same div, the same CSS variables, the same click handler. Move the seam, the canvas resizes; click into the atlas, the seam snaps.
- The atlas head crumb ("3 saved · 43 blueprints · 6 sources") tells the user what the surface contains *as a single phrase*. Most CWS implementations show site status alone; this one shows surface scope.
- README §9 ("Composition discipline — what I declined") is the cleanest "did NOT compose" section in the round. CWS, 3LD-as-interaction, EPC, QR, TAB-on-the-seam are each listed with reasons. This is the v7 brief's "compose, don't pile up" thesis taken seriously.
- The mini-atlas inside the SM head (Panels C/D/E) preserves the same x/y coordinates as the full atlas — spatial memory survives mode switches. This is the strongest spatial-continuity signal in any v7 design.
- The minimap appears only in Panel J (fullscreen, atlas collapsed) as a *replacement door*, not a coexisting widget. README is explicit: "it's a replacement door, not a coexisting one." That's "one door" discipline at the surface level, not just at the chrome level.
- Save button is conditional — present only when the name pill reads "Unsaved." Lesson from earlier v7 reviews applied.
- The 9-step blueprint JSON in Panel D includes `runWpInstallationWizard`, `installTheme`, `installPlugin`, `setSiteOptions`, `importWxr`, `runPHP`, `updateUserMeta` — comprehensive verb coverage.
- Real PHP log lines (12) span deprecation, cURL timeout, cron, SQLite cache, WooCommerce boot — most diverse log set in the round.
- The "Apply & Reset" Panel G uses typed-phrase `coffee-shop` + diff `WP 6.7 → 6.8 RC1` + auto-snapshot `coffee-shop / pre-reset-2026-05-21-1422` — three escalations together.

## What's weak or missing

- The atlas-as-launcher discoverability is the known risk (same as design 023). README mitigates with the head crumb, the click-to-grow at 75 % showing the launcher state, and ⌘K for search — but the first-time visitor still has to learn that the small atlas at 35 % contains the launcher. The mitigations are documented but the risk is real.
- "Save button hidden when site is saved" — defensible per the v7 lesson. But Panel A's idle count includes the Save button slot (counted via "(7) atlas pane (8) seam gauge" — Save is excluded from the 8). Then unsaved is 9? README claims "8 exactly" then says "Save button hidden when saved" → so saved state is 7, unsaved is 8. The 8-exact claim only holds in the unsaved state.
- The atlas mode and SM mode share one surface, swapping content on double-click. README acknowledges "returning to atlas requires explicit clicks (the Atlas back button)." This is "one secondary surface" honest, but the back-navigation cost is a real workflow tax.
- Mobile drops to 2 detents (vs desktop's 5 ratios). Same compromise as design 015 — README admits it. The "spatial primitive survives" claim is mostly carried by the mini-canvas inside the expanded sheet.
- The category chip counts in §7 say "All 46" but elsewhere the brief specifies 43 blueprints. Off by 3 — possibly includes the Featured-overlap, but the inconsistency is visible.
- The atlas pane "regions" (dashed-boundary clusters) are a clever idea, but in stills they look like grouping headers rather than functional containers. A reader scanning Panel A might not know whether the regions are clickable or just decorative.

## How to push harder

- Add an explicit "Click into the atlas to expand to launcher mode" tooltip on the atlas's first paint, with a one-time dismiss. The README's mitigations are good but a teach-on-first-visit closes the loop.
- Reconcile the "8 exactly" affordance count: explicitly state "8 at saved-idle, 9 at unsaved" or restructure so Save doesn't push count over 8 (e.g., conditional Save *replaces* one of the kebab items rather than appearing alongside).
- Sync category counts: All should equal 43, not 46. Featured-overlap should be footnoted (Featured 5 includes 5 of the 43, so All ≠ Featured + Website + ...).
- Make the atlas regions clickable: clicking a region name (e.g., "Themes & content") should pan/zoom the canvas to that cluster. Currently regions look like static labels; promoting them to navigation targets would amplify the spatial thesis.
- Show one panel demonstrating the seam-snap animation (e.g., a frame between r-65-35 and r-50-50 with the gauge thumb mid-travel). The README claims "five demonstrated ratios" but they're all snap states — showing the in-between would prove the SR "momentum gauge" claim.

## Lesson for the next v7 designer

If your synthesis claim is "primitive A and primitive B share the same DOM element," show the binding in CSS — same custom properties, same class system, same click handler — so the critic can grep one selector and see both responsibilities served at once.
